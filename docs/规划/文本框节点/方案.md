# 文本框节点 - MVP 方案 v0.1

> 立项日期：2026-05-18
> 触发：业务部门需求"导出图片后图片上能显示流程说明文字"
> 归档方式：嵌套式（5/13 起新功能约定，与 `docs/规划/游客草稿能力/` 同风格）

## 1. 需求背景

业务部门反馈：当前 PNG 导出（v1.19.0 已支持）只截画布内容，**离开应用上下文后看图人不知道"这画的是什么"/"哪个版本"**。给业务部门看图、放进 PPT、贴进 Word 都会遇到。

业务原话：
> "新增一个功能，需要增加一个'文本框'节点，用于在画布合适的位置添加一个说明性的描述，例如：XXXX流程，方便在导出成图片后，图片上可以显示相关描述。"

## 2. 范围与非范围

### 2.1 范围（MVP v0.1）

- ✅ 新增 `'text'` 节点类型（与 `terminator/process/decision/data/subprocess/group` 并列第 7 种）
- ✅ 纯文字渲染，**无 Handle**（不参与连线，纯展示元素）
- ✅ 三个可调项：**字体 / 字号 / 颜色**
- ✅ 默认样式：**完全透明背景 + 黑色文字 + 24px + 黑体**（适配所有画布主题，方案 §6 决策 A）
- ✅ 编辑入口：复用 `NodeDetailPanel`，文本框 type 时只显示精简面板（文字内容 + 字体 + 字号 + 颜色 4 项）
- ✅ 添加菜单：与 `process/decision/terminator/data` 同一组（方案 §6 决策 3）
- ✅ PNG 导出验证文字可见、字体/字号/颜色正确渲染
- ✅ 权限：复用 `process` 节点权限（可编辑、可删除）

### 2.2 非范围（先不做，等业务反馈再加）

- ❌ 背景色 / 边框 / 阴影 / 描边 / 富文本（粗体/斜体/下划线）
- ❌ 对齐方式（左/中/右）— 默认左对齐
- ❌ 多行换行 — 由 textarea 原生支持，不做特殊处理
- ❌ 预设模板（标题/副标题/脚注/水印）
- ❌ 画布上双击行内编辑（所见即所得）— 走详情面板编辑
- ❌ 工具栏快捷按钮（字号 +/-）
- ❌ 自动适应内容宽度 — 用户手动拖拽 resize
- ❌ codex 取舍审 / 末尾审 — MVP 走自审 4 条门禁（详见 §7）

## 3. 字体候选清单（5 种系统字体）

| key | 显示名 | font-family 字符串 |
|---|---|---|
| `heiti` | 黑体（默认）| `'SimHei', 'Microsoft YaHei', sans-serif` |
| `yahei` | 微软雅黑 | `'Microsoft YaHei', sans-serif` |
| `kaiti` | 楷体 | `'KaiTi', '楷体', serif` |
| `songti` | 宋体 | `'SimSun', '宋体', serif` |
| `fangsong` | 仿宋 | `'FangSong', '仿宋', serif` |

理由：5 种覆盖 90% 中文场景（黑体/雅黑 = 标题屏显 / 楷体 = 标注 / 宋体/仿宋 = 正式文档）。

## 4. 字号档位（5 档）

`14 / 18 / 24 / 32 / 48` （px）

默认：**24px**（适合大多数说明文字，介于"段落"和"标题"之间）。

## 5. 颜色候选

**复用现有 `NodeStyleEditor` 的颜色选择器**——具体色板以现有实现为准（避免引入新色板增加维护面）。默认 `#000000`（纯黑）。

## 6. 关键决策（用户拍板）

| # | 决策点 | 决策 | 理由 |
|---|---|---|---|
| 1 | 档位 | MVP | 业务诉求是"导出图片能显示说明文字"，三可调已 100% 满足；字体/字号/颜色由用户组合接近图里效果 |
| 2 | 默认样式 | A：透明 + 黑字 + 24px | 最普适，适配所有画布主题；图里"深紫底+白字+蓝描边"是装饰层，下一次迭代再说 |
| 3 | 菜单入口 | 与 process/decision 同一组 | 避免界面层级膨胀 |
| 4 | 字体清单 | 5 种系统字体 | 黑体/雅黑/楷体/宋体/仿宋覆盖中文场景；不引入外部字体（避免打包体积 + 字体许可） |
| 5 | 立即做 | v1.19.0 后立即 → v1.20.0 | 不等游客草稿真实反馈观察期 |

## 7. 自审 4 条门禁（替代 codex）

按 CLAUDE.md 工程红线"何时跳过 codex 跑自审"：

| 门禁 | 本方案是否满足 |
|---|---|
| 1. 纯展示组件 | ✅ TextNode 无业务逻辑、无 state、无 mutation |
| 2. 无 mutation 副作用 | ✅ 只读 props 渲染 |
| 3. 依赖底层已审 | ✅ CustomNode / FlowCanvas / useFlowOperations / NodeDetailPanel 都是阶段 3-5 审过的 |
| 4. 边界 ≤ 10 可穷举 | ✅ 节点类型新增 / schema enum / nodeTypes 注册 / onAddNode 分支 / 详情面板分支 / 权限判定 / PNG 导出 / 拷贝粘贴兼容 / 撤销重做兼容 / 自动布局兼容 共 10 项 |

**4 条全满足 → 跳过 codex，走自审**。

## 9. 风险与挑战

| # | 风险 | 严重度 | 应对 |
|---|---|---|---|
| R1 | html-to-image 字体不嵌入 PNG（5/14 D-4 已踩） | 高 | T-10 必须验证 5 种字体 PNG 导出渲染；如有问题改用 SVG `<text>` 直出 |
| R2 | TextNode 无 Handle 时拷贝粘贴 / 撤销重做是否正常 | 中 | T-8 单测覆盖；如有问题在 `useFlowClipboard` 加 type='text' 分支 |
| R3 | NodeResizer 与无 Handle 的兼容（其他节点用 Handle 锚定 resize handle） | 中 | TextNode 直接用 React Flow `<NodeResizer>` 组件官方文档示例 |
| R4 | schema enum 加项后旧画布兼容 | 低 | enum 加项不删项，向后兼容；不需要 migration |
| R5 | 详情面板 type 分支增多影响可维护性 | 低 | 用 if/else 显式分支即可，未来超过 3 种特殊节点再考虑抽象 |
