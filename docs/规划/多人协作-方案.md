## 1. 需求确认（产品规则书）

### 1.1 协作模式

**异步协作**。不做实时光标/拖拽广播。

具体表现：
- A 编辑画布 → 主动点保存 / 关闭页面前提示保存 → 写入服务端
- B 打开画布看到 A 上次保存的版本
- A B 同时编辑时通过冲突合并 + 弹窗处理（见 1.5）

**为什么不做实时**：
- 实时协作需要 CRDT/OT、WebSocket 长连接、状态合并、断线重连，工程量是异步的 5-10 倍
- 业务流程梳理场景天然异步，不是 Figma/Google Docs 那种逐字共编
- 用户描述里"保存提示""手动保存按钮"本身就是异步模型的特征

### 1.2 用户与角色

两类身份、两个角色：

| 身份 | 注册方式 | 默认权限 |
|---|---|---|
| **管理员**（admin） | 启动时通过 `.env` 注入第一个；后续在管理后台创建 | 全部权限 + 管理共有画布元信息 + 物理删除节点 + 关闭任意批注 |
| **普通用户**（user） | 仅管理员能在后台创建账号 | 编辑画布、创建个人画布、批注、协作 |
| **游客** | 免登录，点"以游客身份继续"按钮进入 | 只读 + 仅可见 `is_public_to_guest=true` 的共有画布 |

用户名规则：3-20 字符，字母数字下划线汉字均可。
密码规则：≥8 字符，bcrypt 加密（cost factor 10）。
认证方式：JWT，access token 有效期 7 天 + 自动续期。
用户能在设置页修改自己的密码。

**为什么这么轻**：
- 用户规模 20-50 人，管理员手动建账号一周不到 10 次，比上 SMTP + 邮箱验证 + 找回流程划算
- 没有部门/团队/SSO 等企业级机制（明确放弃）

### 1.3 画布

两种 visibility：

| Visibility | 谁能看 | 谁能编辑 | 谁能创建 | 谁能改元信息 |
|---|---|---|---|---|
| **public**（共有） | 所有登录用户 + （勾了开关时）游客只读 | 所有登录用户 | 仅管理员 | 仅管理员 |
| **private**（个人） | 仅 owner 本人 | 仅 owner 本人 | 任意登录用户 | owner 本人 |

每张共有画布有 `is_public_to_guest`（默认 `false`）开关，仅管理员能勾。

每张画布有"导出 JSON"按钮（任何能看到的人都能导出）和"从 JSON 导入"入口（任何登录用户能导入到自己的个人画布）。

### 1.4 节点级权限规则

#### 节点（node）

| 操作 | 谁能做 |
|---|---|
| 添加节点 | 任何登录用户 |
| 修改自己的节点（位置/文字/颜色/详情）| 节点 creator 本人 |
| 修改别人的节点 | **禁止** —— 想动 → 加批注请原作者改 |
| 删除节点（public 画布）| **禁止**（含自己创建的）—— 仅可"标记为废弃"，物理删除联系管理员 |
| 删除节点（private 画布）| **画布 owner / admin**（私有空间，owner 自由清理）|
| 标记任意节点为废弃 | 任何登录用户（视觉变灰、保留可见、仍可被引用） |
| 物理删除节点（admin） | 任何画布的任何节点（admin 全能）|

> v5 修订（2026-04-30，P3D-2 前置）：原 v3/v4 写"物理删除任意节点仅管理员"过严 —— private
> 画布只 owner 自己能看见能改，让 owner 清理自己节点合理。public 仍保持严格。详见 §5.7。

节点元信息（永久保存）：
- `creator_id`：第一次保存时的当前登录用户，**永久不变**
- `created_at`：创建时间
- `updated_by` / `updated_at`：每次该节点被修改时更新

UI（v3 修订：默认不常驻，避免大画布视觉污染）：节点上**默认不显示创建者**；只在以下情况显示：
- 鼠标 hover 节点 → 节点底部浮出小字 "创建者：张三 · YYYY-MM-DD"
- 节点被选中 → 右侧详情面板顶部显示完整元信息（创建者、创建时间、最后修改人、最后修改时间）
- 用户在工具栏勾选"显示所有节点创建者" → 全局常驻显示（默认关闭，记到 localStorage 偏好）

**保存时提示**（可勾"不再提示"，存 localStorage；按画布 visibility 区分文案）：
- public 画布：
  > ⚠ 保存到公共画布后，节点将无法物理删除（含你自己创建的），仅可标记为废弃。
  > 物理删除请联系管理员。  [取消] [确认保存]
- private 画布：
  > ⚠ 保存到私有画布。你（owner）和管理员可以稍后物理删除节点；其他人看不到此画布。
  > [取消] [确认保存]

#### 连接线（edge）

任何登录用户可以**加 / 改 / 物理删除**任意连接线。不区分作者。

**为什么线没有作者保护**：
- 线没有"创作权"语义，只承载"A→B 的关系"
- 业务流程梳理最常见的协作动作就是"换个连法"，让原作者审批会卡死协作
- 破坏力小：误删一根线一眼能看出来、一键加回来

#### 批注（annotation）

**业务规则**：

- 任何登录用户可以在任意节点上加批注气泡
- 批注内容：作者、时间、纯文本、状态（unresolved / resolved）；**不支持 @mention / markdown**（2026-05-05 P3E 取舍 3 拍板，永久不做）
- 节点右上角显示气泡数字徽章；点击徽章 = 选中节点 + 自动打开详情面板"批注"tab（不弹浮层）
- 关闭批注（标记 resolved）：批注作者 + 节点 creator + 管理员
- 批注本身不能修改（创建即不可改）—— 写错了重新加一条
- **游客不可见批注**（2026-05-05 P3E 取舍 4 拍板）：即使是公共画布游客可见，批注是团队内协作信息，对外不暴露
- **标废弃节点仍可批注**（2026-05-05 P3E 一审产品规则）：节点被标废弃后仍允许新增和关闭批注，因为讨论"该不该废弃"或"何时实际清理"本身可能就需要批注

**展示规则**（2026-05-05 P3E 一审产品规则）：

- 列表按 `created_at` **正序**（早→晚），形成对话上下文
- 节点徽章只统计 `unresolved` 数量（resolved 不计入）
- tab 默认只显示 unresolved 列表，提供"显示已解决"开关展开 resolved
- 长批注在面板内保留换行，单条最大高度限制（超出滚动或折叠）

**容量限制**（2026-05-05 P3E 一审产品规则，前后端双层校验）：

- `content` trim 后长度：1-2000 字符；超出返回 400 + 中文提示
- 单节点 `unresolved` 批注上限 100 条；超限返回 409 + 中文提示
- 单节点全部批注（含 resolved）上限 500 条；超限返回 409 + 中文提示
- 这些限制是防误操作（如粘贴大文本）+ 数据库膨胀，不是业务硬约束

### 1.5 冲突处理

详见第 5 章《合并算法》。简述：

1. **每张画布有 version 字段**（自增整数）
2. 客户端保存时携带 `baseVersion`
3. 服务端检测：当前 version == baseVersion → 直接保存为 baseVersion+1
4. 不一致 → 计算 ΔA（baseVersion → current）和 ΔB（客户端想做的改动）
5. 按业务规则判定真冲突。常见可合并情形（不同节点、不同字段、新增不同对象）由服务端自动合并；少数真冲突场景（同节点同字段、标废弃 vs 改字段、admin 删 vs 别人改、同边同字段、sheet 元信息冲突）需人工裁决。完整分类见 §5.4 / §5.5
6. 无真冲突 → **服务端自动合并**，保存为 current+1
7. 真冲突 → 返回 409 + 冲突详情，前端弹窗让用户选"看对方版本"或"覆盖"

辅助机制：
- **心跳**：客户端每 30 秒打卡 → 服务端记录"还有谁在编辑此画布" → 列表/编辑器顶部显示
- **auto-save 草稿**：客户端每 30 秒自动暂存到 `drafts` 表（不增加版本号），刷新/崩溃后能恢复
- **冲突日志**：每次返回 409 写一行日志，事后看频率决定是否升级算法

### 1.6 数据迁移

新版上线后，登录页旁边检测浏览器 localStorage 里是否存在 `business-flow-multi-canvas` 键。如有 → 用户登录后弹窗：
> 检测到本地有 N 张画布数据，是否导入到云端为个人画布？
> [全部导入] [选择性导入] [稍后处理] [不再提示]

导入后这些画布作为 `private` 画布归属当前用户。原 localStorage 数据**保留不动**（用户可能还想用旧版继续看）。

### 2.3 技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 前端框架 | React 19 + React Flow（不变） | 沿用现有 |
| 后端框架 | **Express 5** | 老牌稳、生态最广、维护门槛低 |
| 数据库 | **SQLite**（WAL 模式） | 单文件零运维；20-50 用户、5+ 同时在线绰绰有余；备份用 sqlite3 `.backup` 命令（详见 §3.3，**WAL 模式下 cp 不安全**） |
| 数据访问 | **better-sqlite3**（裸 SQL） | 同步 API、性能好；表 8 张，手写 SQL 完全可控；不引入 ORM 抽象 |
| 认证 | **JWT (jsonwebtoken)** + **bcrypt** | 标准方案 |
| 进程管理 | PM2（不变） | 沿用现有 |
| 反向代理 | **不需要** | 单进程同源，无跨域 |
| 邮件服务 | **不需要** | 不做注册、不做密码找回 |

**明确放弃**：PostgreSQL / MySQL / Prisma / Drizzle / Hono / Fastify / WebSocket / Redis / nginx / Docker / 实时协作 / CRDT / 自助注册 / SSO / 部门-角色权限矩阵。每条都是为了"快速做完 + 5 人能用"的目标砍掉的。

## 3. 数据模型

> v2 修订：增加 `canvas_versions`（合并算法必需）、`nodes_meta`（防客户端篡改创作权）；`annotations` 增加 `sheet_id`（避免跨 sheet 同 node id 串批注）。

### 3.1 表结构

#### `users`

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'user' CHECK(role IN ('user', 'admin')),
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  deleted_at INTEGER                              -- v3 新增：软删时间戳，NULL = 在岗
);

CREATE INDEX idx_users_username ON users(username);
CREATE INDEX idx_users_active ON users(id) WHERE deleted_at IS NULL;
```

软删行为：
- `deleted_at != NULL` 的用户视为已离职
- 登录中间件检测到 `deleted_at IS NOT NULL` → 拒绝登录（HTTP 403, "account_deactivated"）
- 用户名释放：保留原 username 但允许新用户注册同名（管理员判断）。如果坚持唯一性，复用前要先把老记录的 username 改成 `<原名>_deleted_<id>`
- 引用完整性：`nodes_meta.creator_id` 等外键继续指向已软删用户；前端展示"该节点的创建者已离职（用户名）"
- admin 接管：管理员可在 §9.5 流程下把 private 画布的 owner_id 改为别人

#### `canvases`

```sql
CREATE TABLE canvases (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  description TEXT,
  visibility TEXT NOT NULL CHECK(visibility IN ('public', 'private')),
  is_public_to_guest INTEGER NOT NULL DEFAULT 0,  -- 0/1
  owner_id INTEGER,                                -- private 必填，public 可空
  data TEXT NOT NULL,                              -- JSON: MultiCanvasProject 序列化
  version INTEGER NOT NULL DEFAULT 1,              -- 乐观锁
  created_by INTEGER NOT NULL,
  created_at INTEGER NOT NULL,
  updated_by INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  archived INTEGER NOT NULL DEFAULT 0,             -- 0/1
  FOREIGN KEY (owner_id) REFERENCES users(id),
  FOREIGN KEY (created_by) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id)
);

CREATE INDEX idx_canvases_visibility ON canvases(visibility);
CREATE INDEX idx_canvases_owner ON canvases(owner_id);
CREATE INDEX idx_canvases_guest ON canvases(is_public_to_guest) WHERE is_public_to_guest = 1;
```

`data` 字段存的就是当前 [src/types/flow.ts:201](src/types/flow.ts#L201) 定义的 `MultiCanvasProject` JSON 序列化。**整体覆盖式存储**——每次保存写一份完整 JSON。但**节点的创作元信息不在 data 里**——见 `nodes_meta` 表。

#### `canvas_versions`（v2 新增 —— 合并算法必需）

```sql
CREATE TABLE canvas_versions (
  canvas_id INTEGER NOT NULL,
  version INTEGER NOT NULL,
  data TEXT NOT NULL,                              -- 该版本的完整 MultiCanvasProject 快照
  saved_by INTEGER NOT NULL,
  saved_at INTEGER NOT NULL,
  PRIMARY KEY (canvas_id, version),
  FOREIGN KEY (canvas_id) REFERENCES canvases(id) ON DELETE CASCADE,
  FOREIGN KEY (saved_by) REFERENCES users(id)
);

CREATE INDEX idx_canvas_versions_saved_at ON canvas_versions(canvas_id, saved_at DESC);
```

**初始快照（关键）**：POST `/api/canvases` 创建画布时，服务端必须**在同一事务内**完成以下三步（v4 修订：补 nodes_meta 初始化）：
1. INSERT `canvases` → 拿到 `canvas_id`、设置 `version = 1`
2. INSERT `canvas_versions(canvas_id, version=1, data, saved_by, saved_at)` 写初始快照
3. **遍历初始 data 中所有 sheet 的所有节点**，每个节点 INSERT 一条 `nodes_meta(canvas_id, sheet_id, node_id, creator_id=当前用户, created_at=now, updated_by=当前用户, updated_at=now)`

否则后续任何对这些节点的修改都会被 §5.7 拒绝（"已存在节点的 nodes_meta 无记录"判定为越权）。

**初始快照同样适用于** import 接口（`POST /api/canvases/:id/import`）：导入完成后画布必须同时有 v1 初始快照 **+** 所有节点的 `nodes_meta` 记录（creator_id = 导入操作者）。导入语义上"接管"画布所有权——原作者信息丢失，全部归属导入者。

每次成功保存都向 `canvas_versions` 表追加一条记录。合并算法用它取 `baseVersion` 对应的 `baseData`：客户端 PUT 携带 `baseVersion=3`、当前 `currentVersion=5` → 服务端从 `canvas_versions WHERE canvas_id=? AND version=3` 取 baseData，从 `canvas_versions WHERE canvas_id=? AND version=5` 取 currentData，对 ΔA / ΔB 做合并。

**保留策略**：每画布最近 50 个版本 + 最近 30 天的所有版本（取并集）。超出部分定期清理。理由：合并算法只需要"最近活跃的几个版本"；30 天窗口给数据恢复留底。

**为什么必须有这张表**：v1 方案的合并算法假设服务端能拿到 baseData，但 `canvases.data` 只存最新版本——拿不到。codex 指出这是 v1 的最大硬伤。要么客户端 PUT 时把 baseData 一起发上来（数据量翻倍且不可信），要么服务端存快照——选后者。

#### `annotations`

```sql
CREATE TABLE annotations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  canvas_id INTEGER NOT NULL,
  sheet_id TEXT NOT NULL,                          -- v2 修订：定位到具体 sheet
  node_id TEXT NOT NULL,                           -- 节点的客户端字符串 ID
  author_id INTEGER NOT NULL,
  content TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'unresolved' CHECK(status IN ('unresolved', 'resolved')),
  resolved_by INTEGER,
  resolved_at INTEGER,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (canvas_id) REFERENCES canvases(id) ON DELETE CASCADE,
  FOREIGN KEY (author_id) REFERENCES users(id),
  FOREIGN KEY (resolved_by) REFERENCES users(id)
);

CREATE INDEX idx_annotations_canvas_sheet_node ON annotations(canvas_id, sheet_id, node_id);
CREATE INDEX idx_annotations_status ON annotations(status);
```

批注独立成表（不嵌进 `canvases.data`），原因：
- 批注变更很频繁，独立表避免每次批注都触发画布版本号 +1
- 批注的 CRUD 不参与画布的合并算法，边界清晰

v2 修订：node id 在客户端按"项目内每个 sheet 各自唯一"生成（理论可能跨 sheet 重复），所以批注必须 `(sheet_id, node_id)` 复合定位。

#### `nodes_meta`（v2 新增 —— 节点创作元信息防篡改）

```sql
CREATE TABLE nodes_meta (
  canvas_id INTEGER NOT NULL,
  sheet_id TEXT NOT NULL,
  node_id TEXT NOT NULL,
  creator_id INTEGER NOT NULL,                     -- 第一次保存时的用户，永久不变
  created_at INTEGER NOT NULL,
  updated_by INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  is_deprecated INTEGER NOT NULL DEFAULT 0,        -- 0/1，软删标记
  deprecated_by INTEGER,
  deprecated_at INTEGER,
  PRIMARY KEY (canvas_id, sheet_id, node_id),
  FOREIGN KEY (canvas_id) REFERENCES canvases(id) ON DELETE CASCADE,
  FOREIGN KEY (creator_id) REFERENCES users(id),
  FOREIGN KEY (updated_by) REFERENCES users(id),
  FOREIGN KEY (deprecated_by) REFERENCES users(id)
);

CREATE INDEX idx_nodes_meta_creator ON nodes_meta(creator_id);
```

**为什么独立成表**：v1 方案把 `creator_id` 放在 `canvases.data` JSON 里，客户端可以伪造（提交时改成别人的 id）。codex 指出这是权限完整性风险——服务端必须有**唯一可信的元信息源**。

**服务端在 PUT `/api/canvases/:id` 时必须**：
1. 对比 `canvases.data` 的节点列表与 `nodes_meta` 表
2. 对每个**新增节点**：写入 `nodes_meta` 时强制 `creator_id = 当前用户`、`created_at = now`，**忽略客户端传的任何 creator 字段**
3. 对每个**已存在节点**：检查 `creator_id == 当前用户`（否则禁止修改，权限校验在服务端）；写入 `updated_by/updated_at`
4. 对每个**标废弃**操作：写入 `is_deprecated=1, deprecated_by, deprecated_at`
5. 对每个**已不在 data 中**的节点：仅当当前用户是 admin 时允许从 `nodes_meta` 删除（普通用户不能物理删节点）

`canvases.data` JSON 里的节点对象**仍可携带 creator/created_at 等字段**作为前端展示用（避免每次都 JOIN），但服务端在保存前**用 `nodes_meta` 重写这些字段**，永远以 `nodes_meta` 为准。

#### 备份策略（v2 修订：不能简单 cp）

WAL 模式下 `app.db` 不包含全部已提交数据——可能还在 `app-wal` 里。**简单 cp 会丢数据**。

正确方案（每日 cron）：

```powershell
# E:\business-flow\scripts\backup-db.ps1
$src  = "E:\business-flow-data\app.db"
$dst  = "E:\business-flow-data\backups\app-$(Get-Date -Format yyyy-MM-dd).db"

# 用 sqlite3.exe 的 .backup 命令，会原子地拿到一致快照（不阻塞写）
& "C:\Program Files\SQLite\sqlite3.exe" $src ".backup '$dst'"

if ($LASTEXITCODE -ne 0) {
    # 输出到 PM2 错误日志或邮件
    Write-Error "Backup failed at $(Get-Date)"
    exit 1
}

# 保留 30 天
Get-ChildItem "E:\business-flow-data\backups\app-*.db" |
    Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-30) } |
    Remove-Item
```

替代方案：若不想装 sqlite3 CLI，可在 Node.js 服务里加一个备份接口，调用 `db.backup('path/to/dst')`（better-sqlite3 内置 backup API），通过定时器或外部 cron 触发。

**禁止**：`cp app.db backup/`、`Copy-Item app.db backup/`——这些都不安全。

#### 数据丢失边界（非正式说明 —— v3 新增）

不向用户/上级承诺正式 SLA，但运维上心里要有数：

| 失败场景 | 最坏数据丢失窗口 | 恢复操作 |
|---|---|---|
| Vite/Express 进程崩溃 | 0（数据已落 SQLite，未保存草稿在 `drafts` 表） | PM2 自动重启；用户重开页面恢复草稿 |
| 数据库文件损坏 | **最多 24 小时**（最近一次 `.backup` 之后的所有改动） | 从 `E:\business-flow-data\backups\app-YYYY-MM-DD.db` 拷回 |
| 服务器整机故障 | 同上（备份只在本机） | 同上；如要更短的 RPO，需引入异地备份（暂不做） |
| 用户误删画布 | 0（DELETE 只归档，30 天内可恢复） | admin 在管理后台恢复 |
| 用户误改画布 | 0（`canvas_versions` 保留最近 50 个版本 + 30 天） | admin 选指定版本回滚 |

简言之：**最坏丢一天，恢复人工操作**。这不是 SLA 承诺，只是给运维者一个心理预期。

## 4. API 设计

> v2 修订：补全 schema 校验、限流、事务边界、body size limit。

### 4.0 全局约定

- 所有 `/api/*` 走 JSON
- 时间戳用 Unix milliseconds (number)
- 错误响应：`{ error: "code", message: "human readable" }`，HTTP status 对应语义
- 鉴权：登录后获得 JWT，所有需鉴权接口要求 `Authorization: Bearer <token>`
- **Body size limit**：默认 2MB（覆盖大画布场景）；导入接口单独放宽到 5MB
- **所有写接口（POST/PUT/PATCH/DELETE）必须用 Zod schema 校验入参**——见 4.7
- **画布写接口（PUT/POST canvases，PUT draft）必须包在 SQLite 事务里**：`db.transaction(() => { ... })`，失败回滚

### 4.4 批注

| 方法 | 路径 | 鉴权 | 描述 |
|---|---|---|---|
| GET | `/api/canvases/:cid/annotations` | **仅 JWT**（2026-05-05 P3E 取舍 4 修订，原"可游客"作废） | 列出该画布所有批注（整画布一次返）。游客访问返回 401；前端游客模式不发此请求 |
| POST | `/api/canvases/:cid/annotations` | JWT + 画布访问权 | `{sheetId, nodeId, content}` → 创建批注。服务端必须校验：(1) 当前用户对该画布有读权限；(2) `sheetId/nodeId` 对应节点存在于当前 `canvases.data` 最新版本中（含已标废弃节点）；(3) `content` trim 后 1-2000 字符；(4) 单节点 unresolved ≤ 100 / 全部 ≤ 500 |
| PATCH | `/api/annotations/:id/resolve` | JWT + 关闭权限 | 标记 resolved。服务端通过 `annotations.canvas_id/sheet_id/node_id` 复核：(1) 当前用户对该画布有读权限；(2) 当前用户是 `annotations.author_id` 或 `nodes_meta.creator_id` 或 admin 之一 |
| PATCH | `/api/annotations/:id/reopen` | JWT + 关闭权限 | 重开。校验同 resolve |

**权限校验设计要点**（2026-05-05 P3E 一审产品规则）：

- **绝不能只看 `annotation.id` 操作**：必须先按 id 查出 `canvas_id`，再复核当前用户对该画布的访问权——防止"知道 id 跨画布越权"
- **node creator 来源**：通过 `annotations.sheet_id/node_id` 联表查 `nodes_meta.creator_id`（P3A 已建表 + 索引 `idx_nodes_meta_canvas_sheet_node`）
- **物理删除节点的批注理论不存在**：§5.7 同事务清理已落地（`canvases.ts:853-855`），所以 resolve/reopen 时若 nodes_meta 查不到记录，应返回 404 而非继续走权限判定
- **标废弃节点仍可批注**：POST/resolve/reopen 都不检查节点的 `is_deprecated` 状态——标废弃只是显示语义，不影响批注流

### 4.7 输入校验（v2 新增）

所有写接口入参经 Zod schema 校验，校验失败返回 HTTP 400 + 详细错误。**最关键的是 PUT canvases 和 import 接口**——它们接受用户构造的画布 JSON。

#### 画布 JSON schema（伪代码）

```typescript
// 节点 ID / 边 ID / sheet ID 共用规则
const ShortIdSchema = z.string().min(1).max(64).regex(/^[A-Za-z0-9_-]+$/);

const NodeSchema = z.object({
  id: ShortIdSchema,
  name: z.string().max(200),
  type: z.enum(['terminator', 'process', 'decision', 'data', 'subprocess', 'group']),
  position: z.object({ x: z.number().finite(), y: z.number().finite() }),
  size: z.object({
    width: z.number().positive().max(10000),
    height: z.number().positive().max(10000),
  }),
  // 元信息字段：客户端可传（用于回写显示），服务端在 §5.7 流程中
  // 强制以 nodes_meta 表为准重写，因此这里允许但不信任。
  creator_id: z.number().int().optional(),
  created_at: z.number().int().optional(),
  updated_by: z.number().int().optional(),
  updated_at: z.number().int().optional(),
  is_deprecated: z.boolean().optional(),
  detailConfig: z.object({
    databaseTables: z.array(z.unknown()).max(50).optional(),
    screenshots: z.array(z.unknown()).max(100).optional(),
    description: z.string().max(10000).optional(),
  }).optional(),
  // ...
}).strict();  // 拒绝未知字段

const EdgeSchema = z.object({
  id: ShortIdSchema,
  sourceID: ShortIdSchema,
  targetID: ShortIdSchema,
  sourceHandle: z.string().max(64).optional(),
  targetHandle: z.string().max(64).optional(),
  label: z.string().max(200).optional(),
  edgeType: z.enum(['straight', 'smoothstep', 'step']).optional(),
  notImplemented: z.boolean().optional(),
  waypoints: z.array(z.object({
    x: z.number().finite(),
    y: z.number().finite(),
  })).max(20).optional(),
  style: z.object({
    stroke: z.string().max(32).optional(),
    strokeWidth: z.number().positive().max(20).optional(),
    strokeDasharray: z.string().max(32).optional(),
  }).optional(),
  // markerEnd / markerStart / labelStyle / labelBgStyle / data 等可选字段同上
}).strict();

const SheetSchema = z.object({
  id: ShortIdSchema,
  name: z.string().max(100),
  nodes: z.array(NodeSchema).max(500),
  connectors: z.array(EdgeSchema).max(2000),
}).strict();

const MultiCanvasProjectSchema = z.object({
  version: z.literal(2),
  id: ShortIdSchema,
  name: z.string().max(200),
  description: z.string().max(2000).optional(),
  activeSheetId: ShortIdSchema,
  sheets: z.array(SheetSchema).min(1).max(20),
  createdAt: z.number().optional(),
  updatedAt: z.number().optional(),
}).strict()
.superRefine((project, ctx) => {
  // 跨字段一致性校验（v3 新增）
  const sheetIds = new Set<string>();
  for (const sheet of project.sheets) {
    // 1. sheet ID 全局唯一
    if (sheetIds.has(sheet.id)) {
      ctx.addIssue({ code: 'custom', message: `Duplicate sheet id: ${sheet.id}`, path: ['sheets'] });
    }
    sheetIds.add(sheet.id);

    // 2. node ID 在 sheet 内唯一
    const nodeIds = new Set<string>();
    for (const node of sheet.nodes) {
      if (nodeIds.has(node.id)) {
        ctx.addIssue({ code: 'custom', message: `Duplicate node id ${node.id} in sheet ${sheet.id}`, path: ['sheets', sheet.id, 'nodes'] });
      }
      nodeIds.add(node.id);
    }

    // 3. edge ID 在 sheet 内唯一 + sourceID/targetID 必须指向本 sheet 已存在节点
    const edgeIds = new Set<string>();
    for (const edge of sheet.connectors) {
      if (edgeIds.has(edge.id)) {
        ctx.addIssue({ code: 'custom', message: `Duplicate edge id ${edge.id} in sheet ${sheet.id}`, path: ['sheets', sheet.id, 'connectors'] });
      }
      edgeIds.add(edge.id);
      if (!nodeIds.has(edge.sourceID)) {
        ctx.addIssue({ code: 'custom', message: `Edge ${edge.id} has invalid sourceID ${edge.sourceID}`, path: ['sheets', sheet.id, 'connectors'] });
      }
      if (!nodeIds.has(edge.targetID)) {
        ctx.addIssue({ code: 'custom', message: `Edge ${edge.id} has invalid targetID ${edge.targetID}`, path: ['sheets', sheet.id, 'connectors'] });
      }
    }
  }

  // 4. activeSheetId 必须在 sheets 中存在
  if (!sheetIds.has(project.activeSheetId)) {
    ctx.addIssue({ code: 'custom', message: `activeSheetId ${project.activeSheetId} not in sheets`, path: ['activeSheetId'] });
  }
});
```

**关键限制**：
- 单画布 ≥ 1 且 ≤ 20 sheets
- 单 sheet ≤ 500 nodes / 2000 edges
- 单节点 description ≤ 10000 字符
- 单边 ≤ 20 路径点
- ID 字符集仅允许字母/数字/下划线/短横线（拒绝特殊字符避免 SQL/路径注入面）
- 字段 `.strict()` 拒绝未知字段（防止客户端塞恶意字段绕过 schema）
- 嵌套深度通过 schema 间接限制（无 z.lazy 递归）

**v4 修订**：节点元信息字段（creator_id / created_at / updated_by / updated_at / is_deprecated）在 schema 中显式声明为 optional，避免与 `.strict()` 冲突。服务端在保存流程里**仍以 `nodes_meta` 为唯一可信源**，schema 阶段只做"形状"校验，不做"权威"校验——客户端传任何值都会被 §5.7 流程覆盖。

**v3 新增的跨字段校验**（superRefine 阶段）：
- sheet ID 全局唯一（防止 activeSheetId 引用歧义）
- node ID 在 sheet 内唯一（防止节点 id 撞车）
- edge ID 在 sheet 内唯一
- 边的 sourceID/targetID 必须指向**同 sheet 内已存在**的节点（导入/保存时拒绝悬空边；这是与 §5.6 后置端点清理的分工：schema 阶段拒绝外部输入的悬空边，§5.6 处理合并过程中产生的悬空边）
- activeSheetId 必须存在于 sheets 中

超限响应 HTTP 400，前端在保存按钮上做软限制提醒（"画布节点数已达上限"）。

## 5. 合并算法（核心 —— v2 重写）

> v1 此章存在两个硬伤（codex 评审指出）：
> 1. 假设服务端能拿到 baseData，但实际拿不到（v2 已加 `canvas_versions` 表解决）
> 2. "权限规则砍掉节点级冲突"论证不严密（v2 已重写论证 + 补全冲突分类）

### 5.1 输入与读取

PUT `/api/canvases/:id` 请求体：
- `baseVersion`：客户端 GET 时拿到的版本号
- `data`：客户端想保存的完整 `MultiCanvasProject` JSON
- 操作者 `userId`（来自 JWT）

服务端**在一个 `BEGIN IMMEDIATE` 事务内**读取：
- `currentVersion` ← `canvases.version`
- `currentData` ← `canvases.data`
- `baseData` ← `canvas_versions WHERE canvas_id=? AND version=baseVersion`
  - 如果找不到（baseVersion 太旧已被清理）→ 直接返回 409 `{error: "base_version_expired"}`，让客户端重载

事务边界保证整个 read-merge-write 原子，避免两个并发 PUT 同时通过版本检查写入。

### 5.2 三种情况

**情况 1：无并发**（`baseVersion == currentVersion`）
- 直接写：`canvases.data = clientData`、`version = currentVersion + 1`
- `canvas_versions` 追加新快照
- 同步更新 `nodes_meta`（见 5.6）
- 返回 `{ok: true, version: new, merged: false}`

**情况 2：可合并的并发**（`baseVersion < currentVersion` 且无真冲突）
- 计算 ΔA、ΔB → 检测 → 合并 → 写
- 返回 `{ok: true, version: new, merged: true, mergedFrom: currentVersion, mergeReport: [...]}`

**情况 3：真冲突**（`baseVersion < currentVersion` 且检测到真冲突）
- 不写 → 返回 HTTP 409 + 冲突详情 → 前端弹窗

### 5.3 `computeDelta` 实现要点

把 sheet 内的 `nodes` / `edges` 视为按 id 索引的字典。Δ 的字段：

```typescript
interface SheetDelta {
  nodes: {
    added:    Map<nodeId, Node>;
    removed:  Map<nodeId, Node>;     // 物理删除（v5: admin 任何画布 / private 画布 owner）
    modified: Map<nodeId, FieldChanges>;
  };
  edges: {
    added:    Map<edgeId, Edge>;
    removed:  Map<edgeId, Edge>;
    modified: Map<edgeId, FieldChanges>;
  };
}

interface FieldChanges {
  before: object;   // 原值
  after: object;    // 新值
  changedFields: Set<string>;  // 哪些字段变了
}

interface Delta {
  sheetsAdded:    Map<sheetId, Sheet>;
  sheetsRemoved:  Set<sheetId>;
  sheetsModified: Map<sheetId, SheetDelta>;
}
```

实现注意点：

1. **field 级 diff**，不是整对象 diff：因为合并节点时要判断"A 改了 position、B 改了 label" → 字段不重叠 → 可合并字段级
2. **忽略元数据时间戳的假 diff**：`updated_at` 之类的字段被服务端重写，不应进入 changedFields。计算 diff 时白名单字段
3. **deepEqual 用稳定序列化**：对象字段排序后比较，避免 JSON key 顺序导致假 diff

### 5.4 节点冲突分类（重写论证）

v1 错误地说"权限规则保证节点不会冲突"。v2 实际分类：

| 场景 | A 操作 | B 操作 | 结果 |
|---|---|---|---|
| **N1** | A 加节点 X | B 加节点 Y | **可合并**（id 不同） |
| **N2** | A 改自己的节点 X 字段 a | B 改自己的节点 Y 字段 a | **可合并**（id 不同） |
| **N3** | A 改节点 X 的 position | A 改节点 X 的 label | **同人，正常不会发生**（仅多窗口场景） |
| **N4** | A 改节点 X 的 position | B 改节点 X 的 label | **不会发生**：B 不能改 X（只有 creator 能改） |
| **N5** | A 给节点 X 标废弃 | B 给节点 X 也标废弃 | **可合并**（同操作幂等：取后者时间戳） |
| **N6** | A 给节点 X 标废弃 | B 改节点 X 的字段 | **真冲突**：废弃语义和编辑语义矛盾 |
| **N7** | admin 物理删节点 X | B 改节点 X | **真冲突**：节点已不存在 |
| **N8** | A 加节点 X 并连 X→Y | B 改节点 Y 的位置 | **可合并**：节点 + 边都可加，Y 修改不冲突 |
| **N9** | A 加节点 X 并连 X→Y | B 物理删节点 Y（admin） | **真冲突**：合并后边 X→Y 端点失效 |
| **N10** | A 改 sheet 名 | B 改 sheet 名 | **真冲突**（sheet 元信息字段级冲突） |

**核心修订**：节点级真冲突客观存在（N6、N7、N9、N10）。算法必须显式处理，不能简单宣称"不会发生"。

### 5.5 边冲突分类

边不区分作者：

| 场景 | A 操作 | B 操作 | 结果 |
|---|---|---|---|
| **E1** | A 加边 L1 | B 加边 L2 | **可合并** |
| **E2** | A 加边 L1 | B 加边 L1（同语义：source/target/handle 都同） | **去重**：保留一条（取 A 的，因为 A 先到） |
| **E3** | A 改边 L 字段 a | B 改边 L 字段 b | **可合并**（字段级不重叠）|
| **E4** | A 改边 L 字段 a | B 改边 L 字段 a | **真冲突**（字段级重叠） |
| **E5** | A 删边 L | B 改边 L | **真冲突**：A 想删，B 想改 |
| **E6** | A 删边 L | B 删边 L | **可合并**（同操作幂等） |
| **E7** | A 加边 L: X→Y | 合并时发现节点 X 已被 admin 删 | **降级丢弃**：端点无效，边自动丢弃 + 记入 mergeReport.warnings（**不算真冲突，不弹窗** —— v3 修订与 N9 区分） |

### 5.6 合并伪代码

```typescript
function tryMerge(
  baseData: MultiCanvasProject,
  currentData: MultiCanvasProject,
  clientData: MultiCanvasProject,
  userId: number
): MergeResult {
  const deltaA = computeDelta(baseData, currentData);  // 别人做的
  const deltaB = computeDelta(baseData, clientData);   // 我想做的

  // 1. sheet 级冲突
  for (const sheetId of deltaB.sheetsRemoved) {
    if (deltaA.sheetsModified.has(sheetId)) {
      return { ok: false, conflicts: [{ type: 'sheet_removed_modified', sheetId }] };
    }
  }
  for (const [sheetId, sheetDeltaB] of deltaB.sheetsModified) {
    const sheetDeltaA = deltaA.sheetsModified.get(sheetId);
    if (!sheetDeltaA) continue;
    // sheet 元信息字段级冲突
    const metaConflicts = mergeFields(sheetDeltaA.metaFields, sheetDeltaB.metaFields);
    if (metaConflicts.length) return { ok: false, conflicts: metaConflicts };
  }

  // 2. 节点级
  const nodeConflicts: Conflict[] = [];
  for (const [sheetId, sheetDeltaB] of deltaB.sheetsModified) {
    const sheetDeltaA = deltaA.sheetsModified.get(sheetId) ?? emptyDelta();
    nodeConflicts.push(...detectNodeConflicts(sheetDeltaA.nodes, sheetDeltaB.nodes));
  }
  if (nodeConflicts.length) return { ok: false, conflicts: nodeConflicts };

  // 3. 边级
  const edgeConflicts: Conflict[] = [];
  for (const [sheetId, sheetDeltaB] of deltaB.sheetsModified) {
    const sheetDeltaA = deltaA.sheetsModified.get(sheetId) ?? emptyDelta();
    edgeConflicts.push(...detectEdgeConflicts(sheetDeltaA.edges, sheetDeltaB.edges));
  }
  if (edgeConflicts.length) return { ok: false, conflicts: edgeConflicts };

  // 4. N9 显式检测（v4 修订）：B 主动加边的端点是否已被 A 删除
  //    与 E7（合并过程产生的悬空边）区分：N9 是用户主动行为应弹窗、E7 是被动后果应自动丢弃
  const n9Conflicts: Conflict[] = [];
  for (const [sheetId, sheetDeltaB] of deltaB.sheetsModified) {
    const sheetDeltaA = deltaA.sheetsModified.get(sheetId) ?? emptyDelta();
    const removedNodeIds = new Set([...sheetDeltaA.nodes.removed.keys()]);
    if (!removedNodeIds.size) continue;
    for (const [edgeId, edge] of sheetDeltaB.edges.added) {
      if (removedNodeIds.has(edge.sourceID) || removedNodeIds.has(edge.targetID)) {
        n9Conflicts.push({
          type: 'edge_endpoint_removed_by_other',  // N9
          sheetId,
          edgeId,
          missingEndpoint: removedNodeIds.has(edge.sourceID) ? edge.sourceID : edge.targetID,
        });
      }
    }
  }
  if (n9Conflicts.length) return { ok: false, conflicts: n9Conflicts };

  // 5. 应用：以 currentData 为底，按规定顺序应用 ΔB 中不冲突的部分
  //    顺序：先 sheet 增 → 节点增/改 → 边增/改 → 节点废弃/删 → 边删
  //    （保证应用边时端点节点存在）
  const merged = deepClone(currentData);
  applyDelta(merged, deltaB);

  // 6. 端点有效性后置校验（E7）：N9 已在第 4 步拦截
  //    走到这里仍失效的边只能是 ΔA 中其他用户加的边引用了被本次 ΔB 删除的节点（admin 删）
  //    → 自动丢弃 + warning，不触发 409
  const warnings = validateEdgeEndpoints(merged);  // E7 路径：仅自动丢弃

  return {
    ok: true,
    mergedData: merged,
    mergeReport: collectReport(deltaB, warnings)
  };
}
```

### 5.7 `nodes_meta` 的同步规则（v3 重写）

> v2 的版本写成"对最终 data 中每个节点都校验 creator_id == 当前用户"，会导致 B 保存时只要画布里有 A 的节点就 403。这是错的——画布里 99% 的节点都是别人的，B 的保存请求只是恰好携带了它们的副本而已。**真正要校验的是 deltaB 中实际发生改动的节点**。

每次成功保存（情况 1 或 2）后，服务端在**同一事务**内根据 deltaB 的内容更新 `nodes_meta`。注意 deltaB 是相对 baseData 的差集，不是相对 currentData——这意味着别人改过的节点**不在** deltaB 里，自然不会触发当前用户的越权检查。

**对 deltaB.nodes.added 中的每个节点 id**：
- `nodes_meta` 必须无记录（有则说明 ID 撞车，拒绝整个保存）
- INSERT：`creator_id = 当前用户`、`created_at = now`、`updated_by = 当前用户`、`updated_at = now`
- 服务端**忽略客户端传的 creator/created_at 字段**，强制以服务端值为准

**对 deltaB.nodes.modified 中的每个节点 id**：
- `nodes_meta` 必须有记录（无则说明 ID 是凭空冒出来的，拒绝整个保存）
- 校验 `nodes_meta.creator_id == 当前用户`：
  - 是 → UPDATE `updated_by = 当前用户`、`updated_at = now`
  - 否 → **越权信号**：拒绝整个保存（HTTP 403），写日志
- 例外：如果 deltaB 改动的字段**只**是 `is_deprecated`（从 false→true），不是普通修改，走"标废弃"路径——见下条
- 例外：当前用户是 admin → 允许改任意节点，记入 admin 操作审计日志

**对 deltaB 的"标废弃"操作**（节点 `is_deprecated` 字段 false→true）：
- 任何登录用户都可以标任意节点废弃（见 §1.4 规则）
- UPDATE `nodes_meta`：`is_deprecated=1`、`deprecated_by = 当前用户`、`deprecated_at = now`
- 不更新 `updated_by/updated_at`（这是状态变更不是内容编辑，避免污染历史）
- 同节点重复标废弃幂等（ΔA、ΔB 同标 → 取后者时间戳，详见 §5.4 N5）

**对 deltaB.nodes.removed 中的每个节点 id（物理删除）**（v5 修订：按 visibility 区分）：
- **public 画布**：仅 admin 允许（普通用户对任何节点都禁删，含自己创建的，应使用"标废弃"代替）
- **private 画布**：admin **+** owner 都允许（私有空间，owner 想清理自己节点合理）
- 其他情况（public 普通用户 / 非 owner 的 private 用户）：HTTP 403 `forbidden_remove_node`
- 在同事务内显式按顺序执行（v4 修订：annotations 表无 nodes_meta 外键，不靠级联）：
  1. `DELETE FROM annotations WHERE canvas_id=? AND sheet_id=? AND node_id=?`
  2. `DELETE FROM nodes_meta WHERE canvas_id=? AND sheet_id=? AND node_id=?`
- 同步检查：合并后的 data 中没有任何边以此节点为 source/target（端点孤儿由 §5.6 applyDelta 阶段处理）

**v5 修订背景**（2026-04-30，P3D-2 前置）：
v3/v4 写的是"removed 仅 admin"，导致 private 画布的 owner 想清理自己节点也要走 admin 介入。
P3D-2 实施前发现这条规则对 private 偏严 —— private 画布只 owner 自己能看见，没有"破坏别人工作"
风险，让 owner 自由清理是合理的产品需求。public 仍保持严格（普通用户不能删任何节点）。

**未在 deltaB 中出现的节点（即没改的节点）**：
- 完全不读、不写 `nodes_meta`，不做任何权限校验
- `updated_by/updated_at` 保持原值不变（v2 旧版会把它们污染成当前用户，已修正）

**`canvases.data` JSON 中所有节点的展示字段**（creator_id / created_at / updated_by / updated_at / is_deprecated）：
- 写库前服务端从 `nodes_meta` 重写一次，永远以表为准
- 这一步对**所有节点**做（不只是 deltaB），保证 JSON 与表一致
- 客户端读到的就是可信值，可以直接渲染

**关于"批量标废弃"的注意**：如果 admin 接管离职用户后批量标该用户的节点为废弃，会产生大量 deltaB.modified——但因为只改 `is_deprecated` 字段，走"标废弃"分支不触发 creator 校验，行为正确。

这是 nodes_meta 防篡改的唯一切入点：所有节点级权限校验都集中在这里。

### 5.8 真冲突的响应

返回 HTTP 409：

```json
{
  "error": "conflict",
  "currentVersion": 5,
  "conflicts": [
    {
      "type": "edge_modified_both",
      "sheetId": "sheet_1234",
      "edgeId": "abc123def",
      "field": "label",
      "byUser": "张三",
      "byUserAt": 1730000000000,
      "yourValue": "中标",
      "theirValue": "已中标"
    }
  ]
}
```

前端弹窗（v2 简化版）：

> ⚠ 画布已被【张三】于 5 分钟前修改，与你的改动有 1 处冲突
>
> 详情：边 abc123 的 label —— 你写了"中标"、张三写了"已中标"
>
> [查看张三的版本（你的改动会被保存到草稿）] [覆盖张三的改动] [取消]

"查看张三的版本"行为：
1. 客户端先把当前未保存的画布数据 PUT 到 `drafts` 表（带特殊标记 `from_conflict=true`）
2. 然后 GET 最新画布重载
3. 编辑器顶部条提示："你之前的改动已存到草稿，可在【画布菜单 → 草稿】查看"

这样**用户的改动永远不丢**。

### 5.10 实施分级（v3 修订：场景编号加节点/边前缀）

> codex 评审意见：v1 推荐"上线后再做合并算法"是错的——409-only 在你描述的高频长窗口场景下两周内就会让用户放弃工具。

> v3 修订：v2 §5.10 误把节点场景 6（标废弃 + 改字段）列到"无重叠自动合并"，但它在 §5.4 是真冲突。v3 用 N1-N10 / E1-E7 前缀消除歧义。

引用 §5.4 节点场景 N1-N10 + §5.5 边场景 E1-E7：

**MVP 必做（上线给真人用之前）**：

可自动合并的场景：
- N1 A 加不同节点 + B 加不同节点
- N2 A B 各改自己的不同节点
- N5 A B 都对同节点标废弃（幂等取后者）
- N8 A 加节点 X + 边 X→Y、B 改节点 Y 字段
- E1 A B 加不同边
- E2 A B 加同语义边（去重）
- E3 A B 改同边的不同字段
- E6 A B 都删同边（幂等）

真冲突需 409 + 弹窗的场景：
- N6 A 标节点 X 废弃 + B 改 X 字段（语义矛盾）
- N7 admin 物理删节点 X + B 改 X
- N9 A 加边 X→Y + admin 删节点 Y（端点失效）
- N10 A B 同时改 sheet 元信息字段
- E4 A B 改同边同字段
- E5 A 删边 + B 改同一边

需自动降级丢弃的场景（不算真冲突）：
- E7 加边时端点节点已被删 → 边丢弃，记入 mergeReport.warnings

**v2.1 后续优化**（看实际使用反馈再做）：
- 字段级合并更精细化处理：N3 类（同人在不同窗口改同节点不同字段，仅 admin 接管或异常重连场景才会出现）
- 冲突弹窗的 diff 可视化展示
- 自动合并报告的 UI 美化

**永久放弃**（确认不做）：
- CRDT
- 实时光标广播
- 操作级 undo/redo（保留画布级版本恢复即可，可从 `canvas_versions` 回滚）

## 6. 关键交互流程

### 6.1 登录流程

## 9. 待决策点（v2 修订：阶段 0 必须拍板）

> codex 评审意见：v1 把这些列为"实施时再定"会让阶段 2 的 schema 反复改。v2 修订为**阶段 0 必拍板**。

### 9.1 节点元信息存储

✅ **已拍板（v2）**：独立 `nodes_meta` 表。
- 理由：服务端唯一可信源，防客户端篡改；codex 反对放 JSON 内。
- 影响：阶段 2 schema 含此表；阶段 3 服务端权限校验依赖此表。

### 9.2 auto-save 草稿维度

✅ **已拍板（v2）**：项目级。
- 理由：与现有 `MultiCanvasProject` 存储模型一致，最简单。
- 实现：`drafts.data` 存整个 project 的 JSON。

### 9.3 用户删除策略

✅ **已拍板（v2）**：软删（`users.deleted_at`）。
- 理由：节点的 `creator_id` 永久外键引用，不能因删用户而打破完整性。
- 删除后行为：登录被拒；UI 显示"该节点的创建者已离职（用户名）"。
- admin 可"接管"该用户的节点（见 9.5）。

### 9.4 画布归档策略

✅ **已拍板（v2）**：归档可恢复 + 30 天后允许物理删（仅 admin）。
- DELETE canvases：默认归档（`archived=1`）。
- admin 在管理后台能看到归档列表 + "恢复"按钮 + "彻底删除"按钮。
- 归档画布对游客和普通用户**完全不可见**（含列表）。

### 9.5 admin 接管 owner

✅ **已拍板（v2）**：admin PATCH `/api/canvases/:id` 时可改 `owner_id`。
- 含确认弹窗（前端）+ 服务端校验（不能把 public 画布的 owner_id 设非空）。
- 用于"用户离职、把他的私有画布转给接班人"场景。

### 9.6 版本快照保留策略

✅ **已拍板（v2，新增）**：每画布最近 50 个版本 + 最近 30 天的所有版本。
- 取并集（不是交集）。
- 定期清理任务：每天跑一次，清理超出条件的版本。
- 影响：极少画布会膨胀，但若发生（如内测期间一天保存 200 次）会触发清理逻辑。

## 10. 明确放弃的备选方案（避免回看时反复讨论）

| 方案 | 为什么放弃 |
|---|---|
| 实时协作 (WebSocket + CRDT) | 工程量 5-10 倍；用户场景不需要 |
| 自助注册 + 邮箱验证 | 不想配 SMTP；20-50 人手动建够用 |
| PostgreSQL | SQLite 在此规模性能/运维都更优 |
| Prisma / Drizzle | 8 张表手写 SQL 完全可控；少一层抽象 |
| 部门-角色权限矩阵 | 内部小工具，过度设计 |
| nginx 反向代理 | 单进程同源，不需要 |
| Docker | 当前生产环境是 Windows + PM2，引入 Docker 是另起一摊 |
| 节点级"谁能改谁的"白名单 | 当前规则"创作者 + 管理员"已经足够，加白名单只会让 UI 更复杂 |
| 实时操作日志 / 审计追踪 | 不是合规场景；conflict_logs 已能覆盖核心可观测性 |
| 文件级"本地路径"存储（File System Access API） | 浏览器兼容性差（仅 Chromium），授权弹窗烦；导出/导入 JSON 已经满足"想本地化"的人 |
| **httpOnly refresh cookie + 短 access token**（v2 评审中考虑过） | 内部工具 XSS 风险低；JWT in localStorage 简单够用；如未来要做对外应用再升级 |
| **跨节点同字段的字段级智能合并**（v3 澄清边界） | **MVP 实际做了**：边的字段级合并（E3 场景：A 改边 L 字段 a、B 改边 L 字段 b → 合并；§5.3 已用 field-level diff）。**MVP 放弃**：N3 类场景（同人在不同窗口改同一节点的不同字段后两次保存）—— 因为权限规则下只有 creator 一个人能改自己的节点，理论上该用户能感知到自己开了两个窗口，且 409 冲突弹窗强制刷新即可避免；不值得为此做"同一节点内字段级合并"。也放弃同节点的 markerEnd / waypoints 等复杂结构的深度合并（含数组顺序保留逻辑）。v2.1 视真实使用反馈再决定 |
| **国际化 / 移动端 / 离线模式 / SLA 承诺** | 不是当前内部小工具的目标；明确不做 |

## 11. 风险与未解问题（v2 重写）

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|

| **Vite middleware mode 在 Express 5 下不稳定** | 低 | 中 | 阶段 0 调通后立即测；fallback：双进程（Vite 3001 + Express 3002 + 客户端 fetch /api proxy） |
| **PM2 restart 在 Windows 偶发死锁** | 中 | 高 | 已知坑：post-receive 钩子曾出现 PM2 错位（见 [.claude/skills/deploy/SKILL.md](.claude/skills/deploy/SKILL.md) Gotcha #2）；阶段 6 部署验证时按 SKILL 流程跑 |

## 阶段 2 P2H 残留风险（2026-04-30 codex 八审定稿）

P2H 系列经历 **8 轮 codex 审查 + 7 个 commit + 2 个 CSP 修复**，整体 production-ready 评级 **90%**（详见 [docs/规划/codex审查记录/阶段2/P2H/](codex审查记录/阶段2/P2H/)）。
codex 终审标记为"能进 P2I"，但留下 3 条**不阻塞但需要后续注意**的风险：

### R1. 部署 hook ssh 静默失败（升级到 P1，影响阶段 3 实施）

post-receive.ps1 通过 ssh 调 server 跑 `npm install / build / pm2 restart`，hook 输出全是绿字"PM2 restarted ✓"，但实际**进程可能 5 小时没真正重启**。
P2H 期间踩了一次：commit `2e0c7fa` 的 helmet CSP 修改 push 后 hook 报告成功，但 `pm2 list` 看 uptime=5h，期间整整 7 个 P2H commit 都没在生产生效。手动 `ssh + pm2 restart` 才激活。

**当前缓解**：每次 push 后**强制**用 `ssh administrator@172.16.0.138 'pm2 list'` 看 uptime 是否 `0s`，否则手动 restart。详见 [`.claude/skills/deploy/SKILL.md`](../../.claude/skills/deploy/SKILL.md) Gotcha #12。

**根治方向**：阶段 3 实施前**必须**先解决底层 hook，三选一：
- 文件触发器（hook 写文件，server 端 watcher 执行）
- server 端守护脚本轮询 git HEAD
- 把 hook 从 server 端 post-receive 改到客户端 deploy script

### R2. CSP 配置假定纯 HTTP 部署

当前 `server/middleware/security.ts` 显式 `upgradeInsecureRequests: null`（helmet 默认会强加）。
**未来上 HTTPS 时必须重新评估**：
- `upgradeInsecureRequests` 该启用回来
- `connect-src` / `script-src` / 各 directives 是否需要把 `wss:` 等加进白名单
- HSTS / Strict-Transport-Security 是否要开

P2H 期间踩过一次：仅设置 `upgradeInsecureRequests: []` 在 HTTP 部署下让浏览器把 `/assets/*.js` 强制走 https，导致整站白屏（commit `2e0c7fa` 修复）。

### R3. readOnly 面板的本地 no-op 交互

`NodeDetailPanel` 子组件（NodePropertiesPanel / EdgePropertiesPanel / GroupPropertiesPanel）的 input 控件在游客只读模式下**仍能输入**，但 `safeOnNodeChange` 等回调在 hook 层 no-op 吞掉。
**用户感知**：以为改成功了，但关闭面板/刷新就丢。属于体验风险，**不是数据风险**（hook + 后端 canWrite 双拦保证 server 不收）。

**根治方向**：阶段 3 节点级权限系统重写时一并做（通过 nodes_meta + canEditNode 给每个节点单独决策，子组件 input 直接 disabled）。

### 部署诀窍 / 经验沉淀

详见 [docs/规划/codex审查记录/阶段2/P2H/README.md](codex审查记录/阶段2/P2H/README.md) 的"关键经验沉淀"章节，记录：
- async 竞态需要 ref-first 模式（save/createOnServer 进函数捕获 capturedId）
- finally 身份校验有边界（global UI state 无条件释放，per-canvas state 才校验）
- readOnly 必须穿透到所有 mutation 入口（8+ 个）
- 运行时 UI 标记不能进 storage（`autoSaveFilter` 必须排除 readOnly）
- discriminated union 强制处理所有分支（`switch + default { const _: never = result }`）
