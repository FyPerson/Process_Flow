## 1. 需求背景与产品定位

### 1.1 业务场景

**触发场景**：新需求方在评估上线前没有正式账号 / 不值得开账号，但临时需要画流程图表达需求给开发/产品看。

**当前痛点**：需求方往往不是已有系统内成员（系统内最多 5 个登录用户 / < 100 用户总量），他们没账号又要画流程图就只能口头描述 / Visio / 截图发过来，沟通损耗大。

**项目当前定位呼应**：业务调研窗口期内必须有可用工具——拉产品 + 业务部门关键角色一起画。游客草稿能力是这个定位的自然延伸。

### 1.2 目标用户

**唯一目标用户**：**公司内部、没有账号、临时需要画流程图表达需求的人**。

**非目标用户**：
- 已有 5 人系统内成员（他们用正式画布）
- 外部合作伙伴（不在零信任网关白名单内）
- 客户演示场景（不在本方案范围）

### 1.3 产品定位（一句话）

**单向流转工具**：需求方进入 → 画图 → 导出 PNG/SVG → 发给开发/产品 → 结束。

**不是什么**：
- 不是协作工具——一个画布只服务一个画图人，不支持多人编辑同一草稿
- 不是云端存储——草稿不进服务端数据库，不跨设备同步
- 不是数据交换格式——刻意不开"导入"入口，避免成为数据搬运绕过权限的漏洞
- 不是正式画布——能力是简化版，不要把它当成"正式画布的简化版本"做能力堆砌

## 2. 能力边界

### 2.1 支持的能力

| 能力 | 说明 |
|---|---|
| **添加节点** | 4 种类型：开始 / 普通步骤 / 决策 / 结束（见 §2.3） |
| **连线** | 节点间用箭头连接（React Flow 默认 edge） |
| **拖动 / 删除** | 拖动节点重新布局；选中后 Backspace 删除 |
| **撤销 / 重做** | Ctrl+Z / Ctrl+Y（基础历史栈） |
| **导出 PNG / JPG / SVG**（仅图片格式） | 顶栏「📥 导出图片」下拉，调用 html-to-image 库；**不支持 JSON**（见 §2.2） |
| **本地持久化** | localStorage 自动保存（debounce 500ms），刷新页面恢复 |

### 2.2 不支持的能力（产品红线）

| 能力 | 不做原因 |
|---|---|
| **导入** | 防止成为数据搬运绕过权限的漏洞；PNG/SVG 本身也无法 import 回 React Flow，约束自动满足 |
| **JSON 导出** ⭐ | **2026-05-13 用户拍板**：游客**仅支持图片格式 / 不支持 JSON**。理由：JSON 是天然的数据搬运格式，外发后可能被 import 回正式画布污染权限边界，跟"单向流转工具"产品定位冲突；图片自带"不可 import"约束。**登录用户**则支持 JSON + 图片（见技术债 [#38](../技术债务登记.md)） |
| **分组 / 子流程** | 正式画布范畴，需求方不需要 |
| **详情面板 / 批注** | 同上 |
| **多 sheet** | 单画布够用，不要复杂化 |
| **多人协作 / 心跳** | 游客草稿是单人独占场景 |
| **服务端持久化** | 临时数据不进数据库 |
| **跨设备同步** | localStorage 仅本设备本浏览器 |
| **快捷键 Ctrl+G / Ctrl+Shift+G**（分组） | 简化能力，不开 |
| **复制粘贴跨画布** | 单画布无对端可粘贴 |

### 2.3 节点类型最小集

4 种类型，覆盖"画需求流程图"的实际场景：

| 类型 | 视觉 | 用途 |
|---|---|---|
| **开始** | 圆角矩形 / 绿底 | 需求触发点 |
| **普通步骤** | 矩形 / 白底 | 中间步骤 |
| **决策** | 菱形 / 黄底 | 判断分支 |
| **结束** | 圆角矩形 / 红底 | 需求终点 |

**为什么是 4 个**：覆盖业务流程图核心语义（开始-中间-判断-结束），更多类型反而让需求方纠结"应该选哪个"，降低易用性。

## 3. 架构设计

### 3.1 路由与 Auth 隔离

**新增独立路由 `/draft`**，完全绕开 `/` 主应用的 AuthProvider 守门。

```
未登录访客
  → 看到登录页（src/auth/LoginPage.tsx）
  → 点底部「📝 我来画需求（无需登录）」按钮
  → 跳 /draft 路由
  → 完全独立的 DraftSandbox 页面 + 裸 React Flow
  → 顶栏「📥 导出 PNG/SVG」+ html-to-image 触发下载
  → 「🔙 返回登录」回主应用
```

**为什么是独立路由**：

- 现有 `status='guest'` 语义是"已登录用户的只读模式"（仅在登录页选了"以游客继续"才进入），跟新需求"未登录访客 + 私人草稿沙箱"是两件事，不能混用
- 主应用 BFV 走 `AuthProvider` 守门，强制有 token；草稿路由完全不进 AuthProvider，独立挂载
- 隔离干净，未来若想给草稿加新功能也不影响主应用

### 3.2 组件选型：裸 React Flow vs 裁剪 FlowCanvas

**决策：用裸 `@xyflow/react`，不复用 FlowCanvas**。

**裁剪 FlowCanvas 的反向理由（已放弃）**：
- FlowCanvas 已经被多人协作 / sheet / 分组 / 快捷键 / 权限 / autosave 耦合得很深
- 加 `mode="draft"` 会变成"每一处分支 if (mode === 'draft')"，污染主组件
- 后续 FlowCanvas 演进需要同时考虑草稿场景，背包越背越重

**裸 React Flow 的理由（采纳）**：
- 需求方画图就是基础 React Flow 能力——加节点、连线、拖动、删除——`@xyflow/react` 裸用 50-100 行代码就能跑起来
- 隔离干净，DraftSandbox 是单文件页面，未来改造不影响主应用
- 节点类型最小集（4 个）是新写的简单 React 组件，不复用现有 FlowNodeData 复杂类型

### 3.3 数据持久化

**localStorage 单 key**：`business-flow:draft-sandbox:v1`

**存储结构**：

```typescript
interface DraftSandboxState {
  version: 1;
  nodes: DraftNode[];
  edges: DraftEdge[];
  viewport: { x: number; y: number; zoom: number };
  updatedAt: number;  // Date.now()
}

interface DraftNode {
  id: string;           // nanoid()
  type: 'start' | 'step' | 'decision' | 'end';
  position: { x: number; y: number };
  data: { label: string };
}

interface DraftEdge {
  id: string;           // nanoid()
  source: string;
  target: string;
  label?: string;
}
```

**写入策略**：debounce 500ms，避免拖动节点时频繁写 localStorage。

**容量保护**：节点数超过 100 在顶栏黄字提示「⚠️ 节点较多，建议导出后清空」（不强制阻断）。

### 3.4 导出实现

**新增依赖**：`html-to-image`（约 30KB，是 React Flow 官方推荐的截图方案）。

**实现**：

```typescript
import { toPng, toSvg } from 'html-to-image';

async function exportPng() {
  const wrapper = document.querySelector('.react-flow') as HTMLElement;
  const dataUrl = await toPng(wrapper, { backgroundColor: '#ffffff' });
  triggerDownload(dataUrl, `需求草稿-${formatTimestamp()}.png`);
}

async function exportSvg() {
  const wrapper = document.querySelector('.react-flow') as HTMLElement;
  const dataUrl = await toSvg(wrapper, { backgroundColor: '#ffffff' });
  triggerDownload(dataUrl, `需求草稿-${formatTimestamp()}.svg`);
}
```

**文件名格式**：`需求草稿-${YYYYMMDD-HHmm}.png`（中文 + 时间戳），需求方收到截图能直接转发不用改名。

**注意**：导出前先 `reactFlowInstance.fitView()` 让所有节点进入视窗，否则截图只截当前 viewport 可见部分。

## 4. UI 设计

### 4.1 顶栏布局

```
┌──────────────────────────────────────────────────────────────┐
│ 📝 需求草稿（⚠️ 仅本地保存，请及时导出）                                │
│                              📥导出PNG  📥导出SVG  🔙登录          │
└──────────────────────────────────────────────────────────────┘
```

**关键约束**：
- 红字警告**始终可见**，不要折叠 / 收起——这是产品红线（防止用户以为已云端保存）
- 「🔙 登录」按钮跳回 `/` 主应用登录页（清空草稿前**先弹确认窗**：「未导出的草稿将丢失，确认离开？」）
- 顶栏 height = 48px，跟主应用一致

### 4.2 节点工具栏（左侧）

```
┌────┐
│ 🟢 │ 开始
├────┤
│ ⬜ │ 步骤
├────┤
│ 🟡 │ 决策
├────┤
│ 🔴 │ 结束
└────┘
```

**交互**：点击工具栏按钮 → 在画布中央生成对应类型节点（同主应用「添加节点」侧栏按钮的交互，不做拖动）。

### 4.3 登录页入口

登录页底部追加按钮（[src/auth/LoginPage.tsx](../src/auth/LoginPage.tsx)）：

```
┌─────────────────────────┐
│  [用户名]                │
│  [密码]                  │
│  [登录]                  │
│                          │
│  ───── 或者 ─────        │
│                          │
│  📝 我来画需求（无需登录） │
│  无账号也能画流程图，仅本地保存 │
└─────────────────────────┘
```

**为什么不在主应用顶栏放入口**：主应用是给登录用户用的，不应该让正式用户看到"游客入口"产生迷惑（正式用户应该用正式画布）。

## 6. 关键决策记录

### 6.1 决策表

| 编号 | 决策点 | 选择 | 理由 |
|---|---|---|---|
| D-1 | 路由架构 | 独立 `/draft` 路由不进 AuthProvider | 现有 `status='guest'` 语义冲突，独立更干净 |
| D-2 | 组件选型 | 裸 React Flow 不复用 FlowCanvas | 避免污染主组件，DraftSandbox 隔离 |
| D-3 | 持久化位置 | localStorage 单 key | 临时数据不入库，无配额管理负担 |
| D-4 | 节点类型 | 4 种（开始/步骤/决策/结束） | 覆盖业务流程图核心语义，避免选择困难 |
| D-5 | 导出格式 | PNG / JPG / SVG 三选（**游客仅图片 / 不支持 JSON**） | **2026-05-13 用户拍板**：游客图片 only / 登录用户图片+JSON 差异化策略。理由：JSON 是天然数据搬运格式，外发后可能被 import 回正式画布污染权限边界；图片自带"不可 import"约束。登录用户的图片导出归属技术债 [#38](../技术债务登记.md)（共用 html-to-image 依赖） |
| D-6 | 导入入口 | **刻意不做** | 产品红线，防绕过权限漏洞 |
| D-7 | 入口位置 | 登录页底部 | 不在主应用顶栏，避免登录用户混淆 |
| D-8 | 节点类型扩展 | 仅 4 种，不开放添加 | 简化能力，避免堆砌正式画布功能 |

### 6.2 明确放弃的备选方案

| 备选方案 | 放弃理由 |
|---|---|
| 裁剪 FlowCanvas 加 `mode="draft"` | 污染主组件，每一处分支判断都是技术债 |
| 草稿存服务端匿名 user | 引入配额管理 / 清理任务 / 安全考量，跟"临时不入库"定位冲突 |
| 支持 JSON 导出 | 跟"不支持导入"产品红线冲突——JSON 本来就能 import 回来 |
| 复用 `status='guest'` 现有状态 | 语义不同（现有 guest = 已选登录但走只读 / 新需求 = 完全未登录），混用会产生维护混乱 |
| 在主应用顶栏放游客入口 | 登录用户看到会困惑"为什么我有这个选项" |
| 草稿分享链接机制 | 增加服务端持久化需求，跟产品定位冲突 |
| 节点类型可扩展（让用户自定义） | 偏离"简化能力"定位，降低易用性 |

## 7. 风险与开放问题

### 7.1 已识别风险

| 风险 | 严重度 | 缓解措施 |
|---|---|---|
| **localStorage 容量爆**（节点过多） | 中 | 顶栏 >100 节点黄字提示导出，不强制阻断 |
| **刷新页面 / 误关浏览器丢草稿** | 中 | localStorage 自动恢复 + 顶栏红字警告 |
| **隐身模式 / 清缓存丢草稿** | 低 | 红字警告已提示「请及时导出」，属于用户预期内 |
| **导出 PNG 节点超出 viewport 被裁掉** | 中 | 导出前 `reactFlowInstance.fitView()` 强制框入全部节点 |
| **零信任网关未来变化** | 低 | 已确认内网员工不需要白名单（用户 2026-05-13 拍板） |
| **真实使用频率未验证** | 中 | MVP 上线后观察 1-2 周，如冷清则下线；如热门考虑加节点类型 |

### 7.2 待启动时再决策的开放问题

- **撤销/重做的栈深度**：默认 20 步够用？还是 50？（待 D-5 实施时拍板）
- **导出 SVG 的字体嵌入**：html-to-image 默认不嵌入字体，对方电脑没装中文字体会乱码 —— 是否需要 svg embedFont 选项？
- **欢迎引导**：第一次进入 /draft 是否弹简短引导（左侧工具栏指引 + 「画完别忘了导出」）？还是直接进入空画布？
- **viewport 持久化**：是否要存 `{x, y, zoom}` 让刷新后保持视角？还是刷新就 fitView 重置？
