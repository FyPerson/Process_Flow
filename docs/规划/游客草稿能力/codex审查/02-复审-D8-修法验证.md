## 二、codex 原话（按 severity）

### Summary

> H2 的 CustomEvent 竞态基本闭环，改名路径已经集中到 DraftSandbox，DraftNode 内也没有残留 setNodes 调用。H1 仍有一个实质风险：删除节点时 edges 快照可能包含已删除节点的连线；另外把 history.pushSnapshot 放进 setState updater 属于 React 副作用风险，建议再收敛。

### 🔴 high（1 条）

#### R-H1 H1 修法不完整 — dangling edge

- **category**: edge-case
- **location**: `src/pages/DraftSandbox/index.tsx:253-276`
- **problem**: H1 修法仍可能在删除节点时拍出不一致快照。`wrappedOnNodesChange` 只用 `applyNodeChanges` 计算 next nodes，然后用 `edgesRef.current` 原样推历史；但删除节点通常会连带删除关联 edge。React Flow 可能随后再触发 `onEdgesChange(remove)`，导致历史栈中出现一个"节点已删但边还指向旧节点"的中间快照，undo/redo 可能短暂恢复出 dangling edge。
- **suggestion**: 节点 remove 时同步过滤受影响边：从 changes 提取被 remove 的 nodeId，计算 `nextEdges = edgesRef.current.filter(e => !removedIds.has(e.source) && !removedIds.has(e.target))`，并用 `history.pushSnapshot(nextNodes, nextEdges)`。同时确保实际 `setEdges(nextEdges)` 与节点删除同批发生，或改用 React Flow 的删除回调/统一 delete handler 在一个入口里同时处理 nodes 和 edges。

### 🟡 medium（3 条）

#### R-M1 setState updater 内写 history 副作用风险

- **category**: edge-case
- **location**: `index.tsx:150-158, 253-276, 283-292`
- **problem**: 当前修法把 `history.pushSnapshot` 放在 `setNodes/setEdges` 函数式 updater 内。updater 理论上应保持纯函数；React StrictMode/并发渲染下 updater 可能被重复调用或被丢弃，虽然 `snapshotsEqual` 可缓解重复入栈，但仍存在历史栈这种外部 mutable ref 与 React commit 不完全一致的风险。
- **suggestion**: 更稳的做法是引入一个本地 reducer，把 nodes/edges/history 作为同一状态机更新；或使用"计算 next → 提交 state → 在受控的同步动作入口提交 history"的封装，避免在 React updater 内做副作用。若短期不重构，至少保留针对 StrictMode 的 undo/redo 回归测试。

#### R-M2 结构性判断漏 replace 类型

- **category**: edge-case
- **location**: `index.tsx:263-266, 275-278`
- **problem**: 结构性 change 只识别 `add/remove`，可能漏掉 React Flow 的 `replace` 类型。若后续 React Flow 或某些操作用 replace 更新节点/边，当前路径会走默认 `onNodesChange/onEdgesChange` 且不推历史，导致不可撤销。
- **suggestion**: 把结构性判断扩展为 `remove/add/replace`，或显式列出当前依赖的 @xyflow/react NodeChange/EdgeChange 类型并在测试里固定。

#### R-M3 H3 不应仅挂账

- **category**: edge-case
- **location**: `useDraftHistory.ts:66-97`
- **problem**: H3 仍未修，仅挂账。考虑到当前 D-5 已交付撤销/重做，`setTimeout(100)` 会吞掉用户 undo 后 100ms 内的合法操作；这是可实际触发的行为问题，不只是架构债。
- **suggestion**: 建议不要只挂账。最小修法是删除 `isUndoRedoRef`，因为当前 history 主要由显式动作入口调用，`applySnapshot(setNodes/setEdges)` 本身不会调用 `pushSnapshot`。若保留标记，也应改为一次性同步标记，在 applySnapshot 完成后立即清除，而不是固定 100ms。

### ⚪ low（3 条）

#### R-L1 DraftNode 内 setNodes 担心已消除

- **category**: readability

- **problem**: H2 后 DraftNode 内没有残留 `setNodes` 调用，改名已经通过 Context 回调交给 DraftSandbox 处理。任务 brief 中"应该删 DraftNode 内 setNodes 调用"的担心在附件代码里已经不存在。
- **suggestion**: 无需再删 setNodes。可以把相关说明更新为"DraftNode 仅采集 prompt 输入并调用 onRename"。

#### R-L2 返回登录前未 flush debounce

- **category**: edge-case
- **location**: `index.tsx:203-222`
- **problem**: storageError 的 debounce 写入路径整体可接受，但若用户连续操作后立刻离开页面，最后 500ms 内的改动可能尚未写入，且不会触发失败提示。
- **suggestion**: 在组件卸载或返回登录前可主动 flush 一次 `saveDraft(nodesRef.current, edgesRef.current)`，并根据结果决定是否显示提示或阻止导航。

#### R-L3 反向断言可扩展 BroadcastChannel 等

- **suggestion**: 按产品红线强度决定是否扩展。建议至少新增 BroadcastChannel、SharedWorker、navigator.sendBeacon 的负向断言；postMessage 容易误伤嵌入场景，可先不禁。

## 三、Recommendations（codex 总结）

1. 优先补 H1 的节点删除联动边删除快照，否则撤销栈仍可能出现不一致状态。
2. H3 建议本轮顺手修掉，不建议仅挂账；它的最小修法成本低，且直接影响撤销/重做可信度。
3. H2/M5 修法方向正确：Context 避免全局事件串扰，改名快照不再依赖 queueMicrotask/ref 时序。
4. M1/M2/M3/M4/L1/L4 的文档与 guard 修法基本到位，后续重点应放在撤销栈状态机一致性测试。

## 四、Risks

- 删除节点后的 dangling edge 快照可能导致 undo/redo 结果异常。
- setState updater 内写 history 属于副作用，未来在 StrictMode 或 React 并发行为下可能出现难排查的不一致。
- H3 不修会让快速 undo 后立即新增/改名/连线的操作不进历史栈。
- 返回登录前未 flush debounce 保存时，最后 500ms 的改动存在丢失窗口。
