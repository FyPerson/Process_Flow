## 二、codex 原话（按 severity 分组）

### Summary

> 整体取舍符合"内网、单人草稿、图片单向流转"的产品边界，但当前落地里有几处会直接影响撤销/重做正确性和"返回登录需确认"的方案一致性。最高优先级应先修撤销快照时机，其次补齐导出格式/离开确认/反向断言与持久化异常提示。

### 🔴 high（3 条）

#### H1 删除节点/边后的快照拍到旧状态

- **category**: edge-case
- **location**: `src/pages/DraftSandbox/index.tsx:254-276`
- **problem**: 删除节点或边后的快照很可能拍到旧状态。`wrappedOnNodesChange/wrappedOnEdgesChange` 调用 React Flow 的 `onNodesChange/onEdgesChange` 后用 `queueMicrotask` 读取 `nodesRef/edgesRef`，但这些 ref 依赖 `useEffect` 更新；React 状态提交和 effect 执行不保证在该 microtask 前完成。结果可能是删除动作没有进入历史栈，或 undo/redo 行为错乱。
- **suggestion**: 不要在删除后异步读 ref 推快照。改为自己用 `applyNodeChanges/applyEdgeChanges` 在 `setNodes/setEdges` 函数式 updater 中计算 next，并在同一个 updater 内 `history.pushSnapshot(next, edgesRef.current)` 或 `history.pushSnapshot(nodesRef.current, next)`。这样删除快照与实际状态原子一致。

#### H2 节点改名后的快照拍到旧 label

- **category**: edge-case
- **location**: `src/pages/DraftSandbox/index.tsx:140-146`, `src/pages/DraftSandbox/DraftNode.tsx:57-64`
- **problem**: 节点改名后的快照也可能拍到旧 label。DraftNode 内 `setNodes` 后立即 dispatch CustomEvent，父组件监听后 `queueMicrotask` 再读 `nodesRef`；`nodesRef` 同样依赖 `useEffect` 更新，不保证已经包含新 label。表现为双击改名后 Ctrl+Z 可能无法撤回改名，或历史栈重复旧快照。
- **suggestion**: 把改名逻辑收回 DraftSandbox，通过 nodeTypes 的闭包/自定义 data callback 传入 onRename，或在 DraftNode 中 setNodes updater 内构造 updatedNodes 后直接调用一个注入的 `onChanged(updatedNodes)`。如果继续用事件，事件 payload 至少应包含 `nodeId` 和 `nextLabel`，由父级用当前 ref 同步合成 next 再推快照。

#### H3 isUndoRedoRef setTimeout(100) 时间窗假设

- **category**: edge-case
- **location**: `src/pages/DraftSandbox/useDraftHistory.ts:66-97`
- **problem**: `isUndoRedoRef` 用 `setTimeout(100)` 复位是时间窗假设。用户执行 undo 后 100ms 内如果立刻添加节点、连线或改名，该合法操作会被 `pushSnapshot` 静默跳过；低端机器或 React 并发渲染下也可能出现相反竞态。
- **suggestion**: 取消固定 100ms 窗口。让 undo/redo 只返回 snapshot，不在 hook 内设置长期 flag；由 applySnapshot 使用一个"下一次由 undo/redo 触发的状态变更不入栈"的同步标记，并在 setNodes/setEdges 完成路径中立即清除。当前实现里大多数 push 是显式调用，实际上可删除 `isUndoRedoRef`，只确保 applySnapshot 本身不调用 pushSnapshot。

#### M2 方案写"返回登录弹确认"但实现直接 navigate

- **category**: contradiction
- **location**: `docs/规划/游客草稿能力/方案.md §4.1` 与 `src/pages/DraftSandbox/index.tsx:367-383`
- **problem**: 方案要求点击"返回登录"前弹确认窗"未导出的草稿将丢失，确认离开？"，但实现直接 navigate('/')。虽然 localStorage 不会因返回登录而丢失，但这与"红字提醒/离开确认"的用户预期不一致，也可能让用户误以为已提交或云端保存。
- **suggestion**: 若草稿不会清空，则文案应改为"返回登录不会清空本机草稿"；若产品仍要强调未导出风险，返回按钮应在 nodes/edges 非空时 confirm，确认文案改成"草稿仅保存在本机浏览器，确认返回登录页？"避免误称会丢失。

#### M3 localStorage 读写失败 UI 无降级提示

- **category**: missing-constraint
- **location**: `src/pages/DraftSandbox/persistence.ts:38-68`, `src/pages/DraftSandbox/index.tsx:102-130`
- **problem**: localStorage 读写失败只 console.warn，UI 没有任何降级提示。隐身模式、浏览器禁用存储或 QuotaExceededError 时，用户仍会看到"仅本地保存"，但实际刷新可能丢失，和产品红线提示的可信度冲突。
- **suggestion**: 让 loadDraft/saveDraft 返回状态或错误类型，在 DraftSandbox 顶栏显示黄/红字"本地保存失败，请立即导出"。不需要复杂内存模式，至少要把"当前无法持久化"暴露给用户。

#### M4 D-6 反向断言未禁服务端/实时通信

- **category**: security
- **location**: D-6 反向断言范围（任务简述）
- **problem**: 反向断言目前覆盖了导入、JSON、AuthProvider、主画布复用等核心红线，但没有明确禁止 DraftSandbox 访问服务端或实时通道。对"游客草稿不入库、不协作、不绕权限"的边界来说，`fetch('/api/')`、axios、WebSocket/EventSource、useAuth 等都应作为负向保护。
- **suggestion**: 在 D-6 增加针对 DraftSandbox 目录的反向断言：禁止 fetch/axios 指向 `/api`，禁止 WebSocket/EventSource，禁止 import AuthContext/useAuth（已禁），禁止导入主应用 autosave/collaboration/socket 相关模块。self-test 继续保留反例字符串，避免 grep 规则失效。

#### M5 CustomEvent 全局事件耦合数据流

- **category**: readability
- **location**: `src/pages/DraftSandbox/DraftNode.tsx:21-24`, `src/pages/DraftSandbox/index.tsx:135-148`
- **problem**: CustomEvent 通信能工作，但把节点改名和历史栈耦合到 window 全局事件，隐藏了数据流。单实例页面下风险可控，但未来如果 /draft 被嵌入 iframe 或同页挂多个 ReactFlowProvider，事件会串扰，且测试更难精确断言。
- **suggestion**: 优先改成显式回调：创建 nodeTypes 的工厂或把 onRename 放进 node.data，由 DraftNode 调用 `data.onRename(id, label)`。如果担心函数进入持久化，需要保存前剥离 transient 字段，或使用 React Flow context/store 包一层。当前内网场景可暂缓，但应作为近期技术债记录。

## 三、Recommendations（codex 总结）

1. 先修撤销栈的状态来源问题：删除、改名、拖动结束都应基于同步计算出的 next state 推快照，避免依赖 queueMicrotask + useEffect ref。
2. 把方案和实现的导出格式、返回登录确认、Delete 删除键三处口径统一；末尾审最怕"实现没错但验收文本不同"。
3. D-6 反向断言建议从"禁导入/禁 JSON"扩展到"禁服务端通信/禁实时通信/禁 Auth 依赖"，更贴合游客草稿的安全边界。
4. localStorage 失败不需要复杂恢复，但必须在 UI 上告知用户当前没有自动保存能力，否则"仅本地保存"的红字会误导。
5. CustomEvent 可暂时保留，但建议至少把事件 payload 化；更好的最终形态是显式 callback，避免全局事件串扰。

## 四、Risks（codex 提示）

- 当前撤销/重做在删除和改名场景下存在实际行为不稳定风险，可能被用户很快踩到。
- 实现只导出 PNG，但方案仍写 PNG/JPG/SVG，后续排期或验收可能误判 D-4 未完成。
- 存储失败静默降级会导致用户以为刷新可恢复，实际丢草稿。
- 反向断言如果不禁止 /api/WebSocket，未来重构可能无意引入服务端持久化或协作通道，破坏"单向本地草稿"边界。
- 全局 CustomEvent 在当前单页面风险低，但会增加未来嵌入式复用或多实例测试的维护成本。
