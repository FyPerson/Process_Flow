## 第二层：Codex 原话技术细节

### Summary（codex 原话）

> 总体判断：Claude 草案方向基本适合 5 人内网语境，但有两处需要在开工前拍板：草稿恢复后的版本校验语义，以及卸载前兜底是否值得进入阶段 4。其余点可按简化方案落地，但需要把边界条件写进实现约束，避免后续行为含混。

### High（1 条）

#### Issue #2（high / missing-constraint）— 判断点 2：草稿恢复时 baseVersion 落后于 currentVersion

**Codex 倾向**：阶段 4 可以允许恢复到编辑器，但不应允许无校验地 PUT 主版本覆盖 currentVersion。

**Problem**：
> Claude 草案里的"toast + 允许覆盖"如果等同于恢复后直接按旧 baseVersion 保存为 v7，本质是 last write wins，会把 B 在 v6 的修改整体丢掉；这在小团队里也容易造成不可解释的数据损失。风险/盲点：草稿恢复和主动保存是两个动作，草案没有明确恢复后保存时携带哪个 baseVersion，也没有明确服务端是否拒绝过期 baseVersion。

**Suggestion**：
> 恢复弹窗允许"恢复到编辑器"，但把状态标记为 basedOnOutdatedDraft=true；主动保存 PUT canvases 时必须携带 baseVersion，并在服务端校验等于 currentVersion，冲突时拒绝保存并提示"主版本已更新，请先另存/复制内容/放弃草稿"。如果阶段 4 不做合并，至少不能静默覆盖 v6。

### Medium（4 条）

#### Issue #1（medium / edge-case）— 判断点 1：diff 检测算法选型

**Codex 倾向**：A（整体序列化），但建议改成"canonical JSON 字符串 + 可选 hash 缓存"。

**Problem**：
> 单画布小于 2MB、30 秒一次，整体序列化成本可接受；浅比较很容易漏掉 detailConfig、嵌套 data、connector 属性等深层变更，漏保存比多发一次 PUT 更严重。风险/盲点：如果 nodes/edges 或 convertToFlowData 输出顺序不稳定，JSON.stringify 会产生假阳性，导致无意义草稿 PUT；如果对象属性插入顺序受不同构造路径影响，也可能出现同义 JSON 不同字符串。

**Suggestion**：
> 在 compare 前对 nodes、connectors 按稳定 key 排序，例如 id 或 source-target-id；必要时对对象做稳定 stringify，至少保证 convertToFlowData 的输出顺序稳定。2MB 字符串直接比较问题不大，hash 只能减少比较占用，不能替代 canonicalization；若加 hash，仍应以 canonical 字符串作为生成源。

#### Issue #3（medium / edge-case）— 判断点 3：localStorage 是否完全删除

**Codex 倾向**：主路径完全删除 localStorage，但不建议在阶段 4 做复杂的 localStorage 断网回退。

**Problem**：
> 服务端 drafts 表作为唯一真相源能显著降低跨标签页、跨用户、切画布恢复的歧义；5 人内网下，断网收益不足以抵消双写、重放、过期清理和冲突处理复杂度。风险/盲点：完全删除后，PUT draft 连续失败期间的最近 30 秒到数分钟编辑确实可能丢失；如果 UI 仍显示"自动保存中/已保存"但实际失败，会误导用户。

**Suggestion**：
> 删除旧 localStorage 恢复链路，但必须保留清晰失败状态：草稿 PUT 失败时保持 dirty=true，显示"草稿保存失败"状态，并在下一轮自动重试；切画布、刷新、关闭前沿用 dirty 二次确认。可在上线迁移窗口只读一次旧 localStorage 并提示是否恢复，之后清理，避免老用户遗留草稿消失。

#### Issue #4（medium / readability）— 判断点 4：dirty 标记策略

**Codex 倾向**：Claude 的"不清 dirty"是对的，但 UI 语义需要拆清楚。

**Problem**：
> dirty 应表示"主版本未保存"，草稿保存成功只表示"有可恢复副本"，二者不是同一状态。风险/盲点：如果只有 dirty 红点/标题星号，用户可能无法区分"完全未保护的改动"和"已自动保存到草稿但未提交主版本"；这会影响他们对切画布弹窗的理解。

**Suggestion**：
> 保留 dirty=true 直到主动保存成功；新增轻量状态文案或图标，例如"草稿已保存 14:23""草稿保存失败""正在保存草稿"。CanvasSwitcher 仍应弹二次确认，因为离开画布并不等于提交主版本；弹窗文案应说明草稿会保留，而不是笼统说"未保存可能丢失"。

#### Issue #5（medium / edge-case）— 判断点 5：onbeforeunload 兜底

**Codex 倾向**：阶段 4 不做 sendBeacon 专用 endpoint，优先做 visibilitychange/pagehide 触发一次普通草稿保存尝试，并保留 beforeunload 提示。

**Problem**：
> sendBeacon 不保证送达，不能用 PUT，通常要新增 POST endpoint 和测试分支；为了最多 30 秒改动引入专门协议，阶段 4 性价比偏低。风险/盲点：浏览器进程被杀、断电、系统关机时 sendBeacon 和 fetch 都可能丢；同步 XMLHttpRequest 已废弃，会损害关闭体验，不建议作为新实现。

**Suggestion**：
> 阶段 4 采用更轻方案：页面 visibilitychange 到 hidden、pagehide 时如果 dirty 且 draft snapshot 有变化，就触发一次 keepalive fetch 或普通异步保存尝试；失败不做强保证。若用户直接关闭且仍 dirty，使用 beforeunload 原生确认提示。sendBeacon 专用 endpoint 推迟到阶段 5 或在真实反馈显示 30 秒丢失不可接受时再做。

### Low（1 条）

#### Issue #6（low / edge-case）— 判断点 6：心跳上报频率/超时阈值

**Codex 倾向**：30 秒上报、90 秒过期合理，适合小规模内网，不需要复杂化。

**Problem**：
> 草案没有明确过期判断放在服务端还是客户端，也没有处理页面隐藏、跨标签页重复心跳、用户空闲但页面仍打开的显示语义。

**Suggestion**：
> 服务端查询"活跃编辑者"时按 now - last_seen_at <= 90s 过滤，客户端只展示服务端结果，避免各端时钟偏差和判断不一致。客户端用 setInterval(30s)，页面 hidden 时可暂停或降频，并在 visible 时立即补发一次。同一用户同画布多 tab 因主键会去重，浪费很小；若要优化，可后续用 BroadcastChannel 选一个 tab 上报，但阶段 4 不必做。

### Recommendations

> - 可按 Claude 草案直接落地的点：判断点 1 选择整体序列化比对，判断点 3 删除 localStorage 主链路，判断点 4 草稿保存不清 dirty，判断点 6 采用 30s/90s。
> - 需要用户进一步拍板的点：判断点 2 是否接受"恢复到编辑器但过期 baseVersion 禁止提交主版本"；判断点 5 是否把卸载前兜底限制为轻量 pagehide/visibilitychange，而不新增 sendBeacon endpoint。
> - 建议补一个第 7 个判断点：草稿配额和失败语义。既然 drafts 有"单画布 ≤2MB / 用户 ≤50"，需要明确超过 2MB、超过 50 个草稿、JSON parse 失败、DELETE draft 失败时的服务端错误码和前端提示，否则阶段 4 测试会漏掉最容易出现场景歧义的分支。

### Risks

> - 最大数据风险是过期草稿恢复后无版本校验地保存，导致他人 v6 修改被静默覆盖。
> - 最大体验风险是 dirty、草稿已保存、保存失败三种状态混成一个 UI 信号，用户无法判断自己的修改是否安全。
> - 最大实现风险是为 sendBeacon 新增专门 endpoint 后，形成两套草稿写入路径，增加测试和权限校验遗漏概率。

### 顺带定下两条隐含约束

- **约束 A — 单画布 ≤ 2MB 触达处理**：服务端返 413 + error="draft_too_large"；前端 toast "草稿过大（超 2MB），请简化画布或导出后清空"。auto-save 触发后维持 dirty，下次定时器到点不再尝试（防刷屏）
- **约束 B — JSON parse / 其他失败**：服务端 400 + error="draft_corrupted"；DELETE draft 失败重试 1 次后 silent，下次打开时草稿恢复弹窗仍会出现（保底）

### 决策模式可复用经验

- **判断点 7 用户拍 X 而非 codex 原方案**：codex 看不到具体使用规模时默认建议上限 50 + 拒写，但用户更了解项目实际容量（5 人内网 < 100 用户，单用户 50 个进行中画布属异常情况）。把数字提到 200 后，5 人内网语境下几乎不可能触达，本质上把"配额管理"这个伪命题在工程上消灭掉。**这是"项目语境前置"在 codex 报告之外的另一面应用——用户语境校准 codex 通用建议**
- **判断点 5 fetch keepalive 替代 sendBeacon**：codex 提醒后 Claude 才意识到 `fetch(url, { keepalive: true })` 已经能完成 sendBeacon 的核心功能且支持 PUT method 复用 endpoint。这是 codex 第二视角揪出 Claude 草案盲点的实战正例
