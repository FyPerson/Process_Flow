**关键发现 — 4 个 high**：
1. **draft autosave interval 被 project 变化重置**：每次编辑都会清掉旧 interval，"30s 一次"实际变成"最后一次编辑后 30s 才发"。持续编辑时草稿可能长期没落盘
2. **413/429 后未拦截重试**：注释说"不重试直到 data 变化"，但实现没记录 blocked snapshot，下个 30s tick 仍会对同 snapshot 再 PUT，导致刷屏
3. **草稿 PUT vs 主版本 DELETE 顺序无保护**：旧的草稿 PUT 如果晚于主版本保存的 DELETE 到达服务端，会让已删除草稿"复活"
4. **keepalive 64KB 限制 vs 草稿 2MB 设计冲突**：fetch keepalive 通常仅支持小请求体，2MB 草稿在卸载时会被浏览器丢弃；判断点 5 的可靠性不足

**4 个 medium**：

- `countActiveEditorsByCanvas` 没排除自己 → "X 人在编辑"角标可能包含自己
- 切画布时 `otherEditors` 不清空 → 显示旧画布协作者
- PUT draft 没校验 data 是合法 JSON → `draft_corrupted` 语义没落地

**1 个 low**：canonicalizeProject 按 id 排序可能掩盖"数组顺序有业务语义"的变更（需 Claude 核对）

## 第二层：Codex 原话技术细节

### Summary（codex 原话）

> 整体实现覆盖了阶段 4 的主路径，但草稿自动保存与主版本 5s 保存之间存在几个会导致草稿缺失、过期草稿复活或重复请求的边界问题。心跳逻辑基本可用，但列表计数和前端 stale 状态需要再收紧。

### High（4 条）

#### Issue #1（high / bug）— src/hooks/useDraftAutosave.ts: useEffect 主循环

**Problem**：
> 自动保存 effect 依赖 `project`，每次编辑都会清掉旧 interval 并重新创建新 interval。结果不是"每 30s 保存一次"，而是"最后一次编辑后 30s 才保存"；如果用户持续编辑且主版本 5s 保存失败，草稿可能长期没有落盘。

**Suggestion**：
> 把最新的 `project`、`baseVersion`、`canvasId` 写入 ref，interval 只按 `enabled/canvasId` 等稳定身份启动；tick 从 ref 读取最新快照。不要让 `project` 变化重置 30s 计时器。

#### Issue #2（high / bug）— src/hooks/useDraftAutosave.ts: too_large/quota 处理

**Problem**：
> 注释写明 413/429 后"不重试直到 data 变化"，但实现只设置 `too_large`/`quota`，没有记录失败快照。下个 30s tick 仍会对同一 snapshot 再次 PUT，导致重复 413/429 和 toast/日志刷屏。

**Suggestion**：
> 增加 `blockedSnapshotRef` 和 `blockedReasonRef`。当 413/429 发生时记录当前 snapshot；后续 tick 若 snapshot 未变且 reason 仍是 too_large/quota，直接跳过。snapshot 变化后清除阻塞状态。

#### Issue #3（high / edge-case）— useDraftAutosave + useMultiCanvas.save() 后 DELETE draft

**Problem**：
> 草稿 PUT、卸载 keepalive PUT、主版本保存成功后的 DELETE draft 之间没有顺序保护。旧的草稿 PUT 如果晚于主版本保存和 DELETE 到达服务端，可能把已删除的过期草稿重新插回 drafts 表，下一次打开又提示恢复旧内容。

**Suggestion**：
> 在客户端保存主版本前暂停/取消普通草稿 PUT，并在保存成功后清理本地 autosave 基线；更稳妥的是给 drafts 增加客户端递增 revision 或 clientSavedAt，服务端 upsert 只接受不旧于当前记录的写入，并让 DELETE 带版本/时间护栏避免旧请求复活。

#### Issue #4（high / edge-case）— useDraftAutosave: flushOnUnload

**Problem**：
> `fetch keepalive` 通常有约 64KB 的请求体限制，而草稿设计允许 2MB。把 2MB JSON 草稿依赖 keepalive 兜底会在较大草稿上直接失败或被浏览器丢弃，判断点 #5 的可靠性不足。

**Suggestion**：
> 把 keepalive 明确降级为"小草稿尽力发送"：超过安全阈值时不发 keepalive，只依赖常规定时保存并在 UI 保留未保存状态；或改成更小的增量/压缩 payload。

### Medium（4 条）

#### Issue #5（medium / edge-case）— useDraftAutosave: lastSnapshotRef 初始化

**Problem**：
> `lastSnapshotRef` 只在禁用/null 状态清空；如果组件不卸载而从画布 A 切到画布 B，旧画布 snapshot 会被拿来和新画布 project 比较，可能在用户未编辑 B 的情况下写入一份"初始草稿"。

**Suggestion**：
> 记录 `lastCanvasIdRef`，当 `canvasId` 变化时把 `lastSnapshotRef` 重置为新画布当前 canonical snapshot，并把状态设为 `no_changes`。

#### Issue #6（medium / consistency）— heartbeats.ts: countActiveEditorsByCanvas

**Problem**：
> 任务摘要说画布列表的 activeEditors 要排除自己，但 `countActiveEditorsByCanvas` 没有 `excludeUserId` 参数，SQL 会把当前用户自己的心跳也算进去。除非 `routes/canvases.ts` 另有额外扣减，否则"X 人在编辑"角标会包含自己。

**Suggestion**：
> 给 `countActiveEditorsByCanvas` 增加可选 `excludeUserId`，SQL 中追加 `AND h.user_id != ?`，并让列表路由传入当前用户 id。

**Claude 备注**：本次实施在 `routes/canvases.ts` 层做了"扣减自己"的兜底（额外查 `selfActiveRows` 然后 `Math.max(0, count - 1)`）。codex 不知道这层。可改成 service 层一次性 SQL 排除自己更清晰。

#### Issue #7（medium / bug）— useHeartbeat: 状态更新与错误处理

**Problem**：
> 切换 canvas 时没有先清空 `otherEditors`，心跳失败时也保留上一次成功结果。这样网络失败或切换画布后，前端可能长期显示旧画布或已过期的协作者，违背"客户端只展示服务端 query 结果"的语义。

**Suggestion**：
> 在启用且 canvasId 变化时立即 `setOtherEditors([])`；心跳请求失败时也建议清空或至少记录 `lastSuccessAt` 并在超过 TTL 后清空展示。

#### Issue #8（medium / missing-constraint）— drafts.ts: data JSON 校验缺失

**Problem**：
> PUT draft 没有校验 `data` 是合法 JSON；一旦异常客户端写入非 JSON 字符串，`useDraftRecovery` 只会 console.warn 并跳过弹窗，用户也没有机会通过 UI 删除这份损坏草稿。路由注释里的 `draft_corrupted` 语义实际没有落地。

**Suggestion**：
> 在 `PutDraftRequestSchema` 或 route 层对 `body.data` 做 `JSON.parse` 轻量校验，非法时返回 400 `draft_corrupted`。对历史损坏草稿，恢复检测可以展示"草稿损坏，可放弃"的弹窗并允许 DELETE。

### Low（1 条）

#### Issue #9（low / ambiguity）— useDraftAutosave: canonicalizeProject 排序语义

**Problem**：
> canonical 会按 id 排序 sheets/nodes/connectors。如果这些数组的顺序在渲染层代表层级、tab 顺序或连接线顺序，则纯 reorder 变更会被 canonical 掩盖，草稿 diff 检测不到真实变化。

**Suggestion**：
> 让 Claude Code 核对 `MultiCanvasProject` 的数组顺序是否有业务语义；若有语义，只对确定为集合的字段排序，或把显式 order/zIndex 字段纳入 canonical。

### Recommendations
> - 优先修 `useDraftAutosave`：稳定 interval、失败快照阻塞、画布切换重置基线，这三项会直接影响草稿是否按预期保存。
> - 专门补一组并发/顺序测试或伪测试：普通 draft PUT 在途、主版本 save 成功并 DELETE、旧 PUT 晚到时不应复活草稿。
> - 把 keepalive 明确写成 best-effort，并用 payload 大小阈值控制行为。
> - 核对未提供的 `src/api/canvases.ts`、`routes/canvases.ts`、`useMultiCanvas`、migration 0001/0004，重点看 keepalive 参数、activeEditors 是否排除自己、主保存后 DELETE draft 顺序。

### Risks
> - 最大风险是用户以为有 30s 草稿兜底，但持续编辑或主保存失败时实际没有周期性保存。
> - 主版本保存成功后，旧草稿请求晚到可能造成过期恢复弹窗，用户恢复后会覆盖或混淆已保存内容。
> - 较大草稿在关闭页面时 keepalive 兜底不可依赖，可能形成"测试小数据通过、真实大画布失败"的问题。
