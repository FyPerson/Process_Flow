## 决策模式可复用经验

### 1. 用户语境校准 codex 通用建议

**典型例子**：判断点 7 配额选简化方案 X
- codex 默认建议：配额 50 + 拒写 + 弹窗
- 用户决策：上限提高到 **200**，5 人内网 < 100 用户语境下几乎不会触达，把"配额管理"这个伪命题在工程上消灭掉
- 经验：项目语境前置不只是 codex 报告之前的过滤；codex 报告之外用户用具体规模知识也可以校准其默认建议

### 2. fetch keepalive 替代 sendBeacon

**典型例子**：判断点 5 卸载兜底
- Claude 草案默认 sendBeacon endpoint
- codex 提醒后才意识到 `fetch(url, { keepalive: true })` 已经能完成核心功能且支持 PUT 方法 + 复用 endpoint
- 经验：codex 第二视角揪出 Claude 草案盲点的实战正例 → 不引入第二套写入路径，权限/配额/version 校验只维护一处

### 3. 修法描述 ≠ 修法实现（再次踩坑）

P4 二审 codex 揪出的 4 个 high 都是"我以为修了实际没改对"：
- #1 interval 实际被 project 变化重置（effect deps 含 project）
- #2 too_large/quota 后注释写"不重试"但实现没记 blocked snapshot
- #3 PUT vs DELETE 顺序无保护（fire-and-forget）
- #4 keepalive 64KB 限制和 2MB 草稿设计冲突

**经验**：和 P3D-2 step 3 / P3B 二审一致——交 codex 审前必须 Read 完整文件确认现场，不能只看自己修法描述。

### 4. 4 轮审查链节奏

P4 节奏对齐 P3F/P3G/P3B：
- 01 取舍审 → 7 判断点决策（design-level）
- 02 实施代码审 → 4 high 必修
- 03 修法验证 → 2 high 仍未修干净（"修法描述 ≠ 修法实现"）
- 04 四审 → confidence=high → 通过门禁

收益递减点判断：四审给的剩余意见已经是"业务可接受/可读性优化"级而非"改方向"级 → 直接进 Playwright 端到端，不再走五审。这与 P3F-3 / P3G 收尾节奏一致。
