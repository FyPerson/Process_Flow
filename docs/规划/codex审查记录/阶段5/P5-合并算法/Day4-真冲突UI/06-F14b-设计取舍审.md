## 第二层 — codex 7 项原话技术细节

### High — 2 条

**H1 — ambiguity / F-14b 目标 / 选项 X**
- problem：规格没有明确 F-14b "end-to-end" 边界到底是双用户浏览器全链路还是单浏览器面对服务端并发版本变更。若继续沿用 H4 原文"双 browser context 模拟 A/B 用户"，X 会被认为偏离验收口径
- suggestion：把 F-14b 验收语句改成"用一个真实浏览器客户端触发 BFV 保存路径，另一方并发写入由 API 作为服务端已提交版本模拟；双 ctx 隔离能力由 F-14a 覆盖，不作为 F-14b 的硬约束"

**H2 — edge-case / 选项 X 具体落地**
- problem：X 依赖"浏览器在 admin API 推送前已持有过期 baseline 并进入 dirty 状态"，但没定义如何观测和冻结这个状态。如果 autosave 在 5 秒窗口内先于 API 或保存动作触发，仍可能把 baseline 推进，导致用例退化为直接保存
- suggestion：为每条路径写清楚时序断言——先等待浏览器读到 baseVersion=N，再制造 dirty，但在 debounce 未触发前由 API 推到 N+1，然后立即点击保存或触发明确保存动作；保存前断言客户端仍显示 dirty/保存中且服务端版本已是 N+1

### Medium — 4 条

**M1 — missing-constraint / 三路径前置条件混淆**
- problem：直接保存、自动合并、真冲突三条路径目前共用"过期 baseline + admin API 推新版本"描述，但直接保存目标是 baseVersion === currentVersion
- suggestion：直接保存单独定义为无 admin API 推送，验证 baseVersion=N / currentVersion=N / 保存后 DB version=N+1

**M2 — missing-constraint / 字段最小单位**
- problem：没规定"字段冲突"和"不重叠字段"的最小数据单位。节点名称、位置、样式、画布元数据走不同序列化路径，可能覆盖不到合并算法的核心分支
- suggestion：指定两组固定字段——冲突路径用同一节点 label/name；自动合并路径用 admin 改节点 A label / user01 改节点 B label 或同节点不同可合并字段

**M3 — edge-case / UI 验收**
- problem：UI 验收只写"显示 Toast"和"弹窗含冲突列表"，没要求验证保存状态、版本推进、合并结果回写
- suggestion：每条路径至少增加 3 个断言——(1) SaveStatus 最终回到已保存等价状态；(2) 服务端 version 按预期推进；(3) 重新 GET 或刷新后能看到 mergedData 或冲突待处理状态

**M4 — readability / F-17 审核风险**
- problem：规格解释了 X 的工程理由，但没把覆盖矩阵写出来。后续审查者可能只看到"少了一个 browser context"
- suggestion：在测试说明加入覆盖矩阵——F-14a 双 ctx/token/version 隔离 / 单测 saveErrorDispatcher 分支 / F-15 服务端接口 / F-14b 真实浏览器保存后 UI 响应

### Low — 1 条

**L1 — other / 第三条路**
- problem："双 ctx + 一方禁用 autosave / mock fetch" 没准入条件，可能比 X 更不真实且更脆
- suggestion：只在已有正式测试开关（URL 参数 / localStorage flag）时考虑；不要把 fetch 拦截或 ref hack 放进 Day 4 MVP 主线

### recommendations / risks / notes_for_claude_code

- 主线选 X + 验收口径改成"单真实浏览器 + API 并发写入"
- 降低 flaky 具体手段：唯一 canvasId / 每步读服务端 version / 避免固定 5 秒等待 / **明确按钮保存优先于等待 autosave**
- risks：X 不覆盖 admin 浏览器 saveErrorDispatcher 链路（被单测/F-14a/F-15 承担）；不改验收文字 X 可能被 F-17 审为偏离 H4
- notes_for_claude_code：拍板 X 并在规格/测试注释显式记录范围调整
