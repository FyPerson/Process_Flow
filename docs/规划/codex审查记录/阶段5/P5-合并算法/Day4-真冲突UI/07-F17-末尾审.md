## 第二层 — codex 3 项原话技术细节

### Medium — 1 条

**M1 — edge-case / useMultiCanvas.ts catch / saveErrorDispatcher.ts**
- problem：catch 块把 err 直接 as ApiError 断言后传入 planSaveError。若底层抛出 null/undefined/普通 Error/网络库非 ApiError 形状对象，planSaveError 访问 err.status 可能二次异常，导致原始错误被掩盖，setServerError 也可能拿到不符合 BFV 预期的对象。
- suggestion：planSaveError 入参改为 unknown，先做对象形状守卫；非对象或非 ApiError 形状统一返回 rethrow，并保留原始 error。若 setServerError 类型只能接 ApiError，rethrow 分支先判断形状不符合时不要写 serverError。

### Low — 2 条

**L1 — edge-case / F-14a useDraftAutosave 行为**
- problem："加载画布后未修改仍自动 PUT 推 baseline +1"是预期特性还是遗留？brief 只说测试已主动消化该推进，没说是产品层确认设计。生产上会产生无用户编辑的版本推进，影响审计、冲突概率和用户对版本号的理解。
- suggestion：进入 bump 前在 release note 或代码注释中标记为已知设计语义；如果不是明确设计应挂 Day 5 修复（加载完成后先建立 draft/autosave snapshot 避免无改动 PUT）。

**L2 — readability / F-14b P2.10 Toast 验证**
- problem：Toast 文案未抓到被标为 WARN 可接受为非阻塞，但无法区分"toast 真没出现"和"测试采样时机错过"。降低回归定位价值。
- suggestion：把 e2e 中对 toast 的采样前移到触发保存动作后立即轮询 DOM；或在测试 helper 中监听 toast 容器 mutation。仍可保持 WARN 不作为 bump 阻塞。
