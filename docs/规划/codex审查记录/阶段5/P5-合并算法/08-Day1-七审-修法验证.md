## Issues 按 severity 排列 + 本轮立即修法

### High（1 条 —— 已修）

#### H1 [edge-case] 审计脚本 parentId === null 漏检（已修）

**位置**：`scripts/audit-canvas-integrity.mjs / checkProject parentId 校验`

**问题**：审计脚本把 `parentId === null` 当作"没有 parentId"直接跳过，但 assertProjectIntegrity 只跳过 undefined；**历史 DB 中若有 parentId:null，审计会通过，Day 2 computeDelta 入口仍会抛数据完整性错误**。

**修法**：脚本现在显式记 `node ${id} parentId is null (use undefined or remove the field)` 违规。本地测 db 验证：含 `parentId:null` 的 case 现在被正确捕获。

### Medium（1 条 —— 已修）

#### M1 [edge-case] 审计脚本 sheet.nodes/connectors 缺失静默跳过（已修）

**位置**：`scripts/audit-canvas-integrity.mjs / checkProject sheet 遍历`

**问题**：脚本使用 `sheet.nodes ?? []` 和 `sheet.connectors ?? []`，会让缺失 nodes/connectors 的旧快照**静默通过**；若字段存在但不是数组，则可能直接抛未分组的 TypeError。

**修法**：在每个 sheet 内先用 `Array.isArray` 校验 nodes/connectors；缺失或非数组时记录违规并跳过该 sheet 后续依赖数组的检查。本地测 db 验证：含 `sheet.nodes` 缺失的 case 现在被正确捕获。

### Low（2 条 —— 已修 1 / 挂账 1）

#### L1 [readability] JSON parse error 计数（已修）

**问题**：parse 失败会加入 allIssues，但**没有递增 totalIssues**；如果只有解析失败，报告会显示"共 0 处违规"误导。

**修法**：parse 失败时也递增 totalIssues。本地测 db 验证：corrupt-json case 现在正确显示 "1 处违规"。

#### L2 [readability] schema parentId 测试通过 regex 判失败原因（挂账）

## codex 关键 Recommendations

- **CONNECTOR_DIFF_FIELDS 与 StorageConnectorContentFields 当前 14 个字段完全对齐**，双向 Exclude + AssertNever 模式会在漏字段或多字段时触发 TS2344；**临时删除 `label` 再跑 tsc 是可选的负向验证，不是进入 Day 2 的必要前提**
- schema parentId 6 个 case 覆盖了关键边界；跨 sheet parentId 会按当前 sheet 局部 nodeMap 识别为 not found，链式 group 嵌套会被 group 带 parentId 规则拦下
- **L1 注释不必写进 ConflictType 映射表**；parentId 指向废弃 group 是数据完整性语义，不是冲突类型判定规则
- **NODE 和 CONNECTOR 的 4 个静态断言保持独立更清楚**；现在抽象工厂类型收益不大，反而可能降低维护可读性
