### 4 个 spec gap（codex 额外发现）

| 编号 | 主题 | 最终方向 |
|---|---|---|
| **G1** | 事务边界 | `BEGIN IMMEDIATE` 内顺序：读 current/base → 计算 delta → 权限校验 → 写 nodes_meta → rehydrate JSON → 更新 canvases → 插 canvas_versions → 插 conflict_logs（同事务）；409 conflict_logs 走独立短事务避免主事务回滚后日志丢失；保存成功后删除当前用户 draft 允许失败静默保留（非主数据一致性） |
| **G2** | 响应数据语义 | 无并发保存（情况 1）可不返 data；自动合并成功（情况 2，merged=true）**必须返** `data + version + mergedFrom`；前端用服务端 data 替换本地状态并清 dirty |
| **G3** | sheet 级 delta | **MVP 保守 409**：sheet 元信息（id/name/description/activeSheetId）任意字段双方都改 → 直接 409；不同 sheet 节点/边变化可合并；删除 sheet 与对方修改该 sheet → 409；新增不同 sheet → 可合并。不做 sheet 字段级 diff（频次极低不值得 MVP 投入） |
| **G4** | 权限 vs 合并顺序 | `deltaB`（client vs base）作为权限校验和 nodes_meta 同步的**唯一**输入；`mergedData` 只用于最终 JSON 写库；专项单测：A 保存自己节点变更 → current；B 基于旧 base 改自己节点 → 合并时不得检查或更新 A 节点的 creator_id/updated_by |

## codex 列出的 5 大核心风险（实施时必须挡住）

1. ❌ 不建 canvas_versions 或不回填 → 退化为不可靠 409-only 或误合并 ← **D1 + Day 1 拦截**
2. ❌ connector 只按 id 不按 semanticKey → E2 漏判看到重复连线 ← **D3 + Day 2 拦截**
3. ❌ merged=true 后客户端继续用旧 data → 下次保存连续冲突 / 覆盖 hydrate 字段 ← **GATE-1 + Day 3 拦截**
4. ⚠️ UI 做太重 → 4 天 MVP 测试和事务一致性被挤压 ← **D4/D5 轻量版 + Day 4 卡关**
5. ❌ 409 弹窗"取消"不存草稿 → 改动只在内存丢失风险 ← **D4 完整流程补全 + Day 4 拦截**
