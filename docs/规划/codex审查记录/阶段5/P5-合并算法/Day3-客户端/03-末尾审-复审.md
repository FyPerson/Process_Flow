## codex 戳穿的 bug 链（high #2 A 方案）

1. **A 修法分支**：推进 `serverVersionRef.current = result.version`（line 980 在 if 之外）+ 保留 `projectRef.current = mergedFromVersion 基线 + 后续编辑`（不进 if 内 setProject 块）
2. **核心不变量**（codex 强调）：`serverVersionRef.current` 必须描述当前 `projectRef.current` 的服务端基线
3. A 方案单边推进 ref 到 result.version 但 project 没推进 → **不变量破坏**：ref 说"我基于 v=result.version"但实际 project 是 v=mergedFromVersion + 后续编辑
4. 下一轮 save 时 baseVersion = serverVersionRef = result.version，服务端 currentVersion 也是 result.version
5. → 进**直接保存路径**（baseVersion === currentVersion，不走合并算法）
6. → 直接用 projectRef 旧基线 + 后续编辑 **覆盖** server 上的 mergedData
7. → **A 用户已合入的改动全部丢失**（与"未做合并"等价但更糟——还擦掉了 A）

## Claude 复核确认

```
projectRef = base_v=mergedFromVersion + 后续编辑E
serverVersionRef = result.version  ← 单边推进
server mergedData 持久化在 v=result.version

下次 save 时：
  baseVersion = serverVersionRef = result.version
  data = projectRef = base_v=mergedFromVersion + E
  apiSaveCanvas(canvasId, baseVersion=result.version, data=...)

服务端：
  currentVersion = result.version
  baseVersion === currentVersion → 直接保存路径
  UPDATE canvases SET data = data → 覆盖 mergedData
  → A 改动丢失
```

codex 完全对。我之前把"changeSeq 不等保留 dirty"当成"修法生效"，但忽略了 ref 推进让下次 save 跳过合并的副作用。

## Claude 拍板（独立推荐）

**承认 A 方案错误**。剩下选项：

### 选项 b（codex 偏推荐）— 不推进 baseVersion + 不清 dirty + 不删草稿

```ts
if (result.merged) {
  if (changeSeqRef.current === capturedSeq) {
    // 默认路径（无后续编辑）：整体替换 + 推进 ref + 清 dirty + 删草稿
    serverVersionRef.current = result.version;
    setServerVersion(result.version);
    setProject(result.mergedData);
    projectRef.current = result.mergedData;
    bumpLoadRevision();
    setDirty(false);
    await apiDeleteDraft(...);
    resetDraftAutosaveSnapshot();
  } else {
    // 有后续编辑：所有 server-side state 都不动（核心不变量保持）
    // - serverVersionRef 不推进（保持 mergedFromVersion）
    //   下次 save 用旧 baseVersion → 服务端再走一次合并把 A 改动重新合入
    // - project 不替换（保留本地后续编辑）
    // - dirty 保持 true → 下个防抖 save
    // - 不删草稿（与 base_version_expired 同策略保留兜底）
    console.warn('[merge] keep local edits during merged save; next save will reconcile via merge');
  }
  // setLastSavedAt / setConflict / setAutoSaveDisabled 仅是"标志型"更新，两路径都做，可放外面
  setLastSavedAt(Date.now());
  setConflict(null);
  setAutoSaveDisabled(false);
  return { status: 'merged', version: result.version, mergedFromVersion: result.mergedFromVersion };
}
```

**优点**：
- 核心不变量满足（serverVersionRef ≡ projectRef 服务端基线）
- 下次 save 会再次走合并路径，A 改动重新 attempt 合入（与"未做这次合并"等价）
- 用户后续编辑保留
- 5 人内网下，第二轮合并仍然非常便宜

**代价**：
- 用户**永远看不到 A 用户的改动**（mergedData）—— 直到所有本地后续编辑都保存完为止；如果用户连续编辑，可能多轮合并都不能展示 A 改动
- 合并工作 server 重做 N 次（N = 用户保存期间的"突发编辑回合"数）—— 5 人内网可接受

### 选项 c — 客户端 delta engine 重放本地编辑

需要把 server merge 算法部分移植到客户端（detectNodeConflicts/applyDelta 客户端版）。**严重超出 Day 3 范围**，至少需要 1 整天。

### 拍板：选项 b

5 人内网项目最简单稳妥；用户感知 = "保存期间继续编辑 → 下轮 save 时再次合并 A 改动"；一致性正确。
