## codex 协作模式沉淀（应用到 codex-cli-bridge SKILL）

Day 2 取舍审 v1→v2 翻车 / 翻盘暴露的两个工程纪律：

### 1. Windows 沙箱 1326 的根治写法

**问题**：codex 在 Windows 上启 PowerShell 子进程读文件时一律报 `CreateProcessWithLogonW failed: 1326`（已 ≥4 次踩坑：0428 / Day 1 多次 / Day 2 v1）。

**根治写法**：

```bash
# ❌ 错误：codex 自己 spawn 读文件——sandbox 1326
codex exec --skip-git-repo-check --sandbox read-only \
  -o /tmp/codex-result.md \
  "$(cat /tmp/codex-prompt.txt)"  # 仅 brief + 文件路径

# ✅ 正确：本地 cat 文件内容前置到 prompt + stdin 注入避开 ARG_MAX
codex exec --skip-git-repo-check --sandbox read-only \
  -c MaxContextChars=180000 \
  -o /tmp/codex-result.md - \
  < /tmp/codex-prompt-with-files.txt  # 含全部所需文件内容
```

**关键点**：

- 使用 `-` 作为 PROMPT 参数让 codex 从 stdin 读
- prompt 末尾直接 cat 所有需审查的源码 + 决策文档（Markdown 代码块包起来）
- prompt 头部明确"不要尝试用任何子进程或文件读取工具去读这些文件——你已经有完整内容了"
- ARG_MAX 限制（Windows 32KB）—— 大 prompt 不能用 `$(cat ...)`，必须 stdin

### 2. confidence=high ≠ 实读

v1 报告里 codex confidence=high，但 `notes_for_claude_code` 字段声明"PowerShell 沙箱启动失败 1326，所有源码文件未实读"——只看 confidence 会被骗。**每次 codex 报告必须三步检查**：

1. 看 `notes_for_claude_code` 自我盲区声明
2. grep prompt.txt + stdout.log 验证是否有 `truncated by MaxContextChars` / `sandbox failed` / `1326` 标记
3. 才判断 issue 真假

## 给 Day 2 实施 Claude 的话

- 阶段 A 是 D5 helper 提取的**前置必做**——不做 helper 提取就是浪费工。**不要为了"省时间"跳过阶段 A** 直接写 detector
- applyDelta 字段补丁严格按 NodeDelta.changedFields / ConnectorDelta.changedFields，**禁止** `{ ...incomingNode }` 整对象 spread——服务端归属字段会被覆盖

- applyDelta 输出后**必跑** `assertProjectIntegrity(mergedData, 'mergedData')`——R-Day2-2 兜底
- E7 dangling edge 必须在 assertProjectIntegrity 之前已丢弃，否则 connector endpoints 校验会拒绝合法合并
- saveCanvas 分支 2 调 tryMerge **必须显式捕获 DataIntegrityError**，不混进普通 catch
