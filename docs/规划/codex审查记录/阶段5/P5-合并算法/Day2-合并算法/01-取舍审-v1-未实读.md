### 7 个判断点逐条评审

#### D1 detector 输出粒度
- 推荐：X，或轻量第 3 选项：`detector -> { conflicts, warnings }`，但不产出 `mergedData`
- 理由：测试覆盖更干净，detector 测 N/E/G 分类，`applyDelta` 测字段补丁，`tryMerge` 测编排。
- 反方优势：Y 对调用方更短，且可避免“检测通过但 apply 失败”的心理割裂。
- 实施提醒：N6 这类“有合并语义但当前应阻塞”的场景，用 `Conflict` 表达阻塞，不要提前生成半成品 `mergedData`；用户拍板后的 resolver 可另做。

#### D2 tryMerge 顶层签名
- 推荐：X：`tryMerge(baseData, currentData, incomingData, ctx)`
- 理由：`saveCanvas` 分支 2 接入点最清晰，合并入口负责 delta 计算和检测编排，调用方只处理结果。
- 反方优势：Y 让 `DataIntegrityError` 的抛出点更靠近 `saveCanvas`，捕获路径直观，也方便 Day 3 直接持久化 `deltaA/deltaB`。
- 实施提醒：若选 X，`MergeResult` 或内部返回值最好保留 `deltaA/deltaB` 给 Day 3 conflict logs 使用；否则 Day 3 会被迫重复 computeDelta。

#### D3 N3 同人多窗口策略复核
- 推荐：X，保持同人多窗口也保守冲突。
- 理由：5 人内网场景下同人双窗口同时改同节点同字段的频率很低，保守 409 的扰民成本小于引入例外规则后的解释成本。
- 反方优势：Y 实施确实很便宜，且“不重叠字段自动合并”更符合用户直觉。
- 实施提醒：如果未来收到真实反馈，再按字段交集放宽；Day 2 不建议改 Day 1 已固化语义。

#### D4 sheet 边界外的隐式语义
- 推荐：a collision 不深比；b 双删幂等；c 删除 + 任意内部修改都算 `sheet_removed_modified`
- 理由：sheet 是比 node/edge 更粗的边界，MVP 保守处理符合 Day 1 的 sheet/project 元信息策略。
- 反方优势：a 深比可减少“同内容重复新增”的误判，但代价是引入 sheet 级规范化比较；不值得。
- 实施提醒：c 必须以 `sheetsModified` 整体为准，不只看 `sheetMetaChangedFields`；否则“删 sheet vs 改内部节点/边”会被错误放过。

#### D5 applyDelta 的副作用边界
- 推荐：X，`applyDelta` 纯函数；副作用由 `saveCanvas` 共享 helper 执行。
- 理由：merge 模块保持无 db 依赖，字段补丁可独立单测，权限和 `nodes_meta` 同步仍由服务层统一掌控。
- 反方优势：Y 的事务封装更集中，调用点更短，短期集成代码少。
- 实施提醒：提取分支 3 的副作用 helper 时，输入应是 `deltaB + mergedData/currentData必要索引 + 权限上下文`；不要让分支 2/3 各写一份。字段补丁天然更适合 X，Y 反而容易为了方便拿 incoming 整对象覆盖。

#### D6 detector + applyDelta 代码组织
- 推荐：X：`detector.ts`、`applyDelta.ts`、`tryMerge.ts`
- 理由：D5 选纯函数后，“一概念一文件”最清晰，也和 Day 1 `types.ts / computeDelta.ts` 对齐。
- 反方优势：Y 文件少，早期跳转少。
- 实施提醒：可接受第 3 选项是 `detector.ts` 独立，`applyDelta` 先留在 `tryMerge.ts`，但只适合 apply 很短；一旦字段补丁覆盖 nodes/connectors/sheets，还是拆文件。

#### D7 单测组织
- 推荐：X：`detector.test.ts` + `applyDelta.test.ts` + `tryMerge.test.ts`
- 理由：25+ case 单文件会变成分类表和补丁语义混在一起，失败定位差。
- 反方优势：Y 对端到端阅读更顺，能从一个文件看到完整合并行为。
- 实施提醒：建议分配：detector 12-15 个覆盖 N1-N10/E1-E7/G3/G4；applyDelta 8-10 个覆盖 added/removed/modified/deprecated/alsoDeprecated/字段补丁；tryMerge 5-7 个覆盖无冲突成功、冲突阻塞、DataIntegrityError、sheet 删除边界、warnings。

### 已知风险复核
- 风险 1（changedFields 字段补丁）：成立，且是 Day 2 最大风险。实现上应只对白名单字段打 patch，服务端字段、权限字段、deprecated 反向防御字段不能从 incoming 整对象透传。
- 风险 2（DataIntegrityError 捕获）：成立。`saveCanvas` 合并路径必须单独 `catch DataIntegrityError` 返回 `500 + data_integrity_error`，不要混成 `409` 或普通 `save_failed`。
- 风险 3（N3 保守）：成立。文档和测试名要明确“同人多窗口 MVP 仍冲突”，避免后续 reviewer 误以为漏判。
- 新增风险：权限校验必须仍只基于 `deltaB`，不能基于 `mergedData` 反推。
- 新增风险：`nodes_meta` 同步要以合并成功后的最终 JSON 和 `deltaB` 共同校验，尤其 removed/deprecated/alsoDeprecated 的删除、更新、保留语义。
- 新增风险：connector 引用边界要明确，只允许 sheet 内引用；若存在跨 sheet connector，`sheet_removed_modified` 和完整性校验需要先拒绝或显式定义。
- 新增风险：分支 2 合并成功后也必须执行和快速路径一致的 version、snapshot、annotations delete、事务边界，不能只写主表。
