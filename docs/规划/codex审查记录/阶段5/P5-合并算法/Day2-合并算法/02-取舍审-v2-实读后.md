## Day 2 取舍审

### 总体判断
整体方向是对的：Day 2 应保持 merge 核心纯函数化，`saveCanvas` 负责权限、`nodes_meta`、事务和落库副作用。关键警告是：合并路径不能复刻快速路径的“整份 incoming 保存”思路，必须以 `deltaB` 做权限和副作用输入，以 `currentData + deltaB 字段补丁` 得到最终 JSON。

### 7 个判断点逐条评审

#### D1 detector 输出粒度
- 推荐：X，但做成 `detectConflicts(deltaA, deltaB, currentData?, ctx): Conflict[]` + `tryMerge` 编排 `applyDelta`
- 理由：测试最清晰，detector 测“是否冲突”，applyDelta 测“如何补丁”，tryMerge 测端到端组合；N6 这种冲突场景不需要产出 mergedData，detector 直接阻断更自然。
- 反方优势：Y 能避免 detector 与 applyDelta 对 E2 去重、E7 dangling edge 的判断重复，代码路径更短。
- 实施提醒：把共享判断抽 helper，例如 connector semantic equality / non-semantic field compare，避免 detector 和 applyDelta 各写一份漂移。

#### D2 tryMerge 顶层签名
- 推荐：X：`tryMerge(baseData, currentData, incomingData, ctx): MergeResult`
- 理由：`saveCanvas` 分支 2 调用形态最干净，且 `computeDelta` 的 integrity 校验与 merge 语义绑定在一个入口里，调用方只需显式捕获 `DataIntegrityError`。
- 反方优势：Y 更利于 Day 3 直接记录 `deltaA/deltaB`，也让 detector/apply 层更纯。
- 实施提醒：若 Day 3 conflict_logs 需要 delta，建议第 3 选项：内部仍用 X，但 `tryMerge` 在失败/成功结果里可选返回 `debugDelta?: { deltaA, deltaB }` 给 `saveCanvas` 记录；不要让业务调用者到处手算 delta。

#### D3 N3 同人多窗口策略复核
- 推荐：X，保持同人多窗口同字段也冲突；但注释要统一“N3 不重叠可合并，N4 同字段冲突”。
- 理由：5 人内网下同人多窗口同时改同节点同字段的概率很低，误判成本只是一次 409，不值得为 MVP 引入“后窗口覆盖前窗口”的语义争议。
- 反方优势：Y 的实现成本确实低，对同一用户在两个窗口改不同字段的体验更好。
- 实施提醒：当前 types 注释里 “N3 同人多窗口同节点改不同字段 → auto-merge” 与 brief 里“同字段双方都改 → 冲突”并不矛盾，但实现时要以字段交集为准：不重叠合并，重叠冲突；不要把“同人多窗口”扩大成所有同节点都冲突。

#### D4 sheet 边界外的隐式语义
- 推荐：a collision 拒绝；b 双删幂等；c 删除 vs 任意修改该 sheet 都 `sheet_removed_modified`。
- 理由：a 深比收益低且 ID 生成撞车本身异常；b 与节点/边双删幂等一致；c 只看 sheetMeta 会漏掉最重要的内部节点/边变更，导致 B 改了节点但 A 删 sheet 时被静默吞掉。
- 反方优势：a 若内容完全一致时放过能减少极少数重复导入误冲突；c 若只改内部且可迁移到别的 sheet，理论上能做更智能保留，但 MVP 不值得。
- 实施提醒：`sheetsModified.has(s1) && deltaA.sheetsRemoved.has(s1)` 或反向都应冲突；不要只检查 `sheetMetaChangedFields.length`。

#### D5 applyDelta 的副作用边界
- 推荐：X，applyDelta 必须纯函数。
- 理由：`server/services/merge` 不应依赖 db；权限和副作用已经有 `saveCanvas` 事务上下文，提 helper 后分支 2/3 可以共用 `validateDeltaBPermissions`、`rewriteNodesFromMeta`、`syncNodesMetaFromDeltaB`。
- 反方优势：Y 一次调用封装完整保存流程，短期接线更少，不容易忘记 annotations 删除。
- 实施提醒：分支 3 当前的数组 delta 与 Day 1 `Delta` 形态不同，建议先把快速路径也迁移到 `computeDelta(currentData, incomingData)`，再共用 helper；否则会退化成两套副作用逻辑。

#### D6 detector + applyDelta 代码组织
- 推荐：X：`detector.ts` / `applyDelta.ts` / `tryMerge.ts`
- 理由：D5 选纯函数后，一概念一文件最好测，也与 `types.ts`、`computeDelta.ts` 现有结构一致。
- 反方优势：Y 文件少，早期跳转少。
- 实施提醒：第 3 选项也可接受：`detector.ts` 单独，`applyDelta` 暂放 `tryMerge.ts`；但只适合 applyDelta 很小。按当前 sheet/node/edge 顺序和 E7 warning 复杂度，独立文件更稳。

#### D7 单测组织
- 推荐：X，分模块测试。
- 理由：25+ case 单文件会很快变成“失败了但不知道是哪层语义坏了”；分层能定位是 detection、patch 还是编排问题。
- 反方优势：Y 更容易按 N1-N10 / E1-E7 顺序通读全局规则。
- 实施提醒：建议分配：`detector.test.ts` 12-15 个覆盖 N4/N6/N7/N9/E2/E4/E5/sheet/project；`applyDelta.test.ts` 8-10 个覆盖字段补丁、alsoDeprecated、E2 去重、E7 drop、sheet 增删；`tryMerge.test.ts` 5-7 个覆盖成功合并、冲突短路、DataIntegrityError、report 计数。

### 已知风险复核
- 风险 1（changedFields 字段补丁）：成立，是 Day 2 最大风险。applyDelta 更新已有 node/edge 时只能逐字段写 `change.after`，并且先 strip 服务端归属字段，不能 spread incoming 整对象。
- 风险 2（DataIntegrityError 捕获）：成立。`saveCanvas` 需要扩展结果类型或 route 映射，明确返回 500 + `data_integrity_error`，不要落入普通 throw 后的 `save_failed`。
- 风险 3（N3 保守）：方向成立，但实现描述要精确成“同字段冲突；不同字段可合并”，否则会比已锁定类型注释更保守。
- 新增风险：`activeSheetId` 合并后可能指向被删除 sheet，需在 apply 后或 detector 中兜底 `active_sheet_missing`。
- 新增风险：connector 端点与 `parentId` 在字段补丁后可能变成无效引用，apply 后最好跑一次 `assertProjectIntegrity(mergedData, 'mergedData')`，E7 允许的 dangling edge 必须在 assert 前已丢弃。
- 新增风险：合并路径写入 `mergedData` 前仍要走与快速路径同等的服务端 meta rewrite，否则 merged=true 返回给前端的数据会带错 creator/updated/deprecated 字段。
- 新增风险：新增 sheet 内的节点也需要批量插入 `nodes_meta`，不能只处理既有 sheet 的 node added。
- 新增风险：删除 sheet 时要删除该 sheet 下所有 `nodes_meta` 和 annotations；当前快速路径只有按 removed node 删除，合并路径若支持 sheet removed 必须补 sheet 级清理。

### 项目语境合规性
以上建议按内网、单实例、最多 5 人并发、总用户 < 100 的前提给出。没有建议 Redis、队列、CRDT、SLO/CDN 等超出当前项目范围的方案；取舍重点放在纯函数边界、事务内副作用复用和 MVP 冲突保守性。
