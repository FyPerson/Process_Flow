## ⚠️ 后续修正：路径 b 不可达性发现（2026-05-07 B-4 实施时）

### 发现

写 tryMerge 测试时，原计划锁定"路径 b（deltaB 未改 activeSheetId + currentData 暂态 activeSheetId 指向已删 sheet）"的 case 报错，揭穿了**该路径在 tryMerge 层级实际不可达**——computeDelta 入口的 assertProjectIntegrity 对**两个入参**都校验 activeSheetId 存在性。

### 不可达链路

1. A 用户 saveCanvas 时调 `computeDelta(base, A_incoming)`
2. 若 A 删了 `base.activeSheetId` 指向的 sheet 但没改 activeSheetId → `A_incoming.activeSheetId` 悬空
3. `assertProjectIntegrity(A_incoming, 'incomingData')` 立即抛 `DataIntegrityError`（computeDelta.ts:130）
4. A 的 saveCanvas 被拦下 → **永远写不入 → 不会产生 currentData 暂态 activeSheetId 缺失**
5. B 后续 tryMerge 看不到这个状态

### 影响

| 项 | 修正 |
|---|---|
| **codex B-2.5 M2 论证前提** | 错误。"detector 推到 B-3 是因为路径 b 看不到" 这一论证基于不存在的状态。但 active_sheet_missing 防御仍有意义（见下） |
| **applyDelta 防御仍有意义** | 路径 a（deltaB 主动改 activeSheetId 到被删 sheet）依然可达；路径 c 是脏 delta 兜底。applyDelta 内部映射保留 |
| **B-3 验收清单"必修测试覆盖路径 b"** | 不必要 — tryMerge 层不可达；applyDelta 单测层面路径 b 仍构造保留作为防御覆盖（08 归档已锁定） |
| **路径 b 在 tryMerge.test.ts 的处理** | 改为"锁定不可达性"case：构造该状态，断言 computeDelta 立即抛 DataIntegrityError |

### 复盘

我（Claude）当时直接接受了 codex M2 的论证而没反向验证 — 这是"修法描述 ≠ 修法实现"反复的第 6 次（前 5 次见 0506 / 0507 0428 等）。**反向验证证伪**才是断言/防御逻辑成立的硬证据。codex 论证 ≠ 实际可达，是同一类教训的另一面。
