## Issues 按 severity 排列

### Medium（5 条 —— Day 2 起步前 must 修 4 条 + 1 条挂账并行）

#### M1 [missing-constraint] assertProjectIntegrity 缺 activeSheetId/endpoints 校验

**位置**：`server/services/merge/computeDelta.ts: assertProjectIntegrity`

**问题**：完整性扫描仍只覆盖重复 ID 和 null handle，没有校验：
- `project.activeSheetId` 是否存在于 sheets 中
- `connector.sourceID/targetID` 是否指向**同 sheet 内**节点

客户端 incoming 由 schema superRefine 兜底，但 Day 2 的 baseData/currentData 来自 DB 和 canvas_versions，**历史脏数据可以绕过 schema**。后续 E7/N9/active_sheet_missing 判断可能被错误前提污染（指向已不存在的端点）。

**建议**：在 `assertProjectIntegrity` 中先收集 sheetIds 和每个 sheet 的 nodeIds，再校验 activeSheetId、每条 connector 的 source/target 是否存在；失败统一抛 DataIntegrityError 并带 `sourceLabel + sheetId + edgeId`。`parentId/relatedNodeIds` 若是硬引用，也应在同处明确校验；若不是硬约束，要写明不参与完整性判定。

→ **Day 2 起步前 must 修**

#### M2 [contradiction] schema parentId 校验形同虚设 + brief 描述不准

**位置**：`server/schemas/canvas.ts: MultiCanvasProjectSchema.superRefine` + brief Q1

**问题**：brief 说 schema 已校验 parentId，但实际代码（line 224-228）只是个空注释，没 addIssue；且循环中边收集 nodeIds 边检查 parentId，即使要校验也会**误伤后置父节点**（拓扑顺序保存的画布）。incoming 仍可携带不存在的 parentId，base/current 更没有兜底。

**建议**：先定规则——如果 parentId 必须指向同 sheet 节点，用第二遍循环校验，并同步加入 assertProjectIntegrity；如果允许悬空 parentId，就修正文档/注释，避免 Day 2 把它当成可靠引用。

→ **Day 2 起步前 must 修**（先定规则；本项目 parentId 是分组 group 节点的引用，**应**硬约束）

#### M3 [ambiguity] node_modified_both 用户身份策略未明确

**位置**：`server/services/merge/types.ts: ConflictType 映射表`

**问题**：`node_modified_both` 类型存在，但 N 映射表里 N3 写成"同人多窗口 auto-merge"，N4 写成"不可能"。这与现有权限并不完全一致——
- admin 可改任意人节点 → N6/N7/N8 可发生在 admin vs creator 之间
- 同节点创建者也能改自己的节点（自然路径）
- admin 与 creator、多个 admin、同一用户多窗口都可能落到"同一 node 同字段被双方改"

**Delta 本身不携带 userId**，无法仅靠 delta 判断是否是同人多窗口。

**建议**：Day 2 前明确策略——
- 不同用户同字段改同节点应走 `node_modified_both`
- 同一用户多窗口若要"后者覆盖"，detect 函数必须拿到 userA/userB 或 saved_by/current saver 信息
- 否则建议 MVP 先把同字段修改统一判冲突，避免静默覆盖

→ **Day 2 起步前 must 修**：detect 函数签名应接 userA/userB（A 是 currentVersion 的 saved_by，B 是 SaveCanvasInput.user）

#### M4 [edge-case] edge_semantic_conflict 定义不全

**位置**：`server/services/merge/types.ts: edge_semantic_conflict 注释`

**问题**：映射表只覆盖"semanticKey 相同但 ID 不同且内容差异"，但两个客户端也可能新增同 ID、同 semanticKey、不同 label/style/waypoints 的边。若按 E2 直接保留 A，会**静默丢掉 B 的非语义字段**。

**建议**：把 `edge_semantic_conflict` 定义扩展为"semanticKey 相同但**非语义字段内容不一致**"，不限定 ID 是否相同；或明确同语义新增边永远保留 A 并给 warning/report。Day 2 加 add-add 同 semanticKey、同/不同 ID、同/不同内容的测试矩阵。

→ **Day 2 起步前 must 修**（注释扩展定义，detector 实施时再覆盖）

#### M5 [missing-constraint] M4（白名单防漂移）真没闭环

**位置**：`server/services/merge/types.ts: StorageNode / NODE_DIFF_FIELDS`；`server/schemas/canvas.ts: NodeSchema`

**问题**：v3 M4 修法只是注释闭环——但**不会真正防止未来 NodeSchema 新增第 18 个 storage 字段但漏加 NODE_DIFF_FIELDS**。现有 H1 测试只证明 4 个已知字段进入 changedFields，不能自动发现新字段遗漏；并且 `[key: string]: unknown` 会削弱 `keyof StorageNode` 类静态断言的效果。

**建议**：补自动漂移检测：
- 优先拆出 `StorageNodeKnownFields`，用类型断言确保 `typeof NODE_DIFF_FIELDS[number]` 与可编辑 storage 字段 union 完全一致
- 或导出 NodeSchema 字段名常量，写单测比对 schema 内容字段、服务端 meta 字段和 NODE_DIFF_FIELDS
- connector/sheet/project 白名单也建议做轻量对表测试

→ **Day 2 起步前并行处理**（不阻塞 detector 实施，但应在 Day 2 同步加防漂移）

### Low（2 条）

#### L1 [ambiguity] semanticKey 注释自相矛盾

**位置**：`server/services/merge/types.ts: connectorSemanticKey 注释` + `server/schemas/canvas.ts: EdgeSchema`

**问题**：semanticKey 的 null/undefined 契约本身闭环，但注释里同时写"handle 一律是 undefined 或非空字符串"和"schema 当前允许空字符串"。这两句**冲突**；当前 schema 的 `z.string().max(64).optional()` 确实允许 `""`。

**建议**：二选一——
- 若空字符串是合法显式 handle：注释改成"undefined 或 string，空字符串会与 undefined 区分"
- 若空字符串不合法：EdgeSchema 加 `.min(1)`，并同步测试

→ **Day 2 起步前 5 分钟**改注释（建议保留空字符串合法，因为本项目 React Flow 默认 handle 可能传 ""）

#### L2 [readability] 17 个字段三处重复

**位置**：`server/services/merge/types.ts: 文件头注释 / StorageNode 注释 / NODE_DIFF_FIELDS`

**问题**：17 个节点字段在文件头、StorageNode 显式声明、NODE_DIFF_FIELDS 三处重复。当前可读性可以接受，但未来加字段需要同步三处，维护成本和漂移概率会增加。

**建议**：保留 StorageNode 与常量作为权威来源，文件头注释改为说明原则并引用常量；真正的字段一致性依赖自动测试或类型断言（即 M5 的修法），而不是长注释清单。

→ **挂账 Day 2 顺手或 Day 3**（与 M5 配套）

## Recommendations（codex 四审建议）

- v3 三审的 M1/M2/M3/L1 可以视为按承诺闭环
- M4 如果目标只是"注释说明"，也已完成；但如果目标是"防止未来漏字段"，还没有闭环 → 建议升级到自动检测（M5）
- 可以进入 Day 2 的 detectNodeConflicts/applyDelta 设计，但建议**先处理或并行处理 3 个前置项**：扩展完整性校验、明确 node_modified_both/N3/N4 策略、补 NODE_DIFF_FIELDS 漂移防护
- assertProjectIntegrity 的 O(sheets + nodes + connectors) 扫描在单实例、最多 5 人并发、schema 上限和 2MB body 限制下不是性能阻塞；不建议为性能牺牲完整性校验
- ConflictType 映射表较长但目前有价值，不必急着拆文件；更重要的是让每个类型都有 Day 2 测试或 detector 分支对应

## Risks（codex 列）

1. 如果 Day 2 直接假设 DB 快照的 activeSheetId 和 edge endpoints 永远有效，历史脏数据会把合并冲突误判成自动合并或普通 409
2. 如果同节点同字段冲突不区分 admin/创建者/同人多窗口，可能出现静默覆盖或过度 409，两者都会影响协作体验
3. 如果不补白名单漂移测试，未来新增节点 storage 字段时，合并算法可能静默丢失该字段修改，问题很难从类型系统发现

## Day 2 进入 detectNodeConflicts/applyDelta 前的建议必修清单（codex 四审）

1. **扩展 assertProjectIntegrity** 校验 activeSheetId 与 connector endpoints，并决定 parentId 是否硬校验
2. **明确 node_modified_both 与 N3/N4 的用户身份策略**，必要时让 detector 接收 userA/userB
3. **补 NODE_DIFF_FIELDS 与 NodeSchema/StorageNode 的自动对表保护**
4. 为 add-add **同 semanticKey 边的内容差异**补映射和测试
