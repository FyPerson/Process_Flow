### Medium（4 条 —— Day 2 起步前 must 修 3 条 + 1 条挂账）

#### M1 [edge-case] H3 防御层留有遗漏 sheetsAdded/sheetsRemoved 路径

**位置**：`server/services/merge/computeDelta.ts: computeDelta / computeSheetDelta / indexNodes / indexConnectors`

**问题**：DataIntegrityError 只会检查双方都存在的 sheet 内部节点/连线重复；如果某个 sheet 是 sheetsAdded 或 sheetsRemoved，当前流程不会对该 sheet 的 nodes/connectors 建索引，因此 DB 中新增 sheet 快照若带重复 nodeId/connectorId 会**绕过** H3 防御层。客户端 incoming 正常会被 schema 拦住，但 currentData/baseData 来自 DB，正是 H3 注释声称要兜底的来源。

**建议**：在 computeDelta 入口先做 `assertProjectIntegrity(base, 'baseData')` 和 `assertProjectIntegrity(incoming, 'incomingData')`，**一次性校验所有 sheet** 的 sheetId/nodeId/connectorId；或让 indexSheets 在遍历每个 sheet 时顺手调用 indexNodes/indexConnectors。补充重复 connectorId、added sheet 内重复 nodeId/connectorId 的测试。

→ **Day 2 起步前 must 修**

#### M3 [edge-case] connectorSemanticKey null vs undefined 契约不明

**位置**：`server/services/merge/types.ts: connectorSemanticKey`

**问题**：JSON.stringify 元组能解决分隔符碰撞，也能区分 undefined 与空字符串，但当前 `sourceHandle ?? null` 会把 undefined **和 null 编成同一个 key**。schema 正常路径不允许 null，但历史 DB 或损坏数据可能出现 null；这与 brief 中"undefined vs null vs '' 应不同"的目标不一致。

**建议**：二选一明确契约——
- 如果 null 永不合法：在注释中写明 undefined/null 归一为未指定，并在 integrity 校验中拒绝 null
- 如果必须区分：用存在性哨兵编码，例如 `[sheetId, sourceID, targetID, hasOwn ? value : {__absent: true}, ...]`

→ **Day 2 起步前 must 修**（注释明确即可，不必加哨兵；schema 已禁 null）

#### M4 [ambiguity] StorageNode 仍保留 `[key: string]: unknown`

**位置**：`server/services/merge/types.ts: StorageNode`

**问题**：brief 说 StorageNode 去掉了 `[key: string]: unknown` 的隐式宽容，但代码仍保留索引签名。运行时白名单本身对 H1 已经对齐 NodeSchema，不过类型层无法在未来 schema 新增 storage 字段时给出**编译期提醒**，H1 类问题仍可能复发。

**建议**：如果目标是编译期防漏，删除 StorageNode 的索引签名，或增加类型级断言/测试。若保留宽容期，则更新 brief 和注释，承认白名单不靠 StorageNode 类型兜底。

→ **挂账（Day 2 顺手处理 / 也可挂 Day 3）**：注释明确"白名单是真实的硬边界，类型签名不兜底"即可

## Recommendations（codex 阶段建议）

- Day 2 可以开始，但建议先修或明确 3 个 must：**全量 project integrity 校验、ConflictType 16/17 映射收敛、semanticKey 对 null 的契约**

- **H2 alsoDeprecated 的触发条件正确**：仅内容变且 false→true 时填 true；仅废弃走 deprecated 分类；已废弃后再改内容不应填
- 可挂账项：StorageNode 是否删除索引签名、MergeReport sheet/project 统计、migration 0005 约束复核、注释和测试样例清理

## Risks（codex 列）

1. 如果 Day 2 applyDelta **用整节点覆盖**而不是字段补丁，已废弃节点可能被 incoming 的 false 或伪造元字段污染
2. 如果不先做**全量 integrity scan**，历史 DB 中 added sheet 内部重复 ID 可能在自动合并时被传播
3. 如果 ConflictType **数量和映射不收敛**，Day 2 detector 可能把应自动合并、应 warning、应 409 的场景混淆
4. 如果 semanticKey 对 **null 的策略不明确**，历史数据或损坏数据可能把 null handle 与缺省 handle 误判为同一语义边
