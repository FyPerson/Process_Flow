## Issues 按 severity 排列

### High（5 条）

#### H1 [missing-constraint] 判断点 1 / canvas_versions 迁移与保存路径

**问题**：选择 1A 是正确方向，但草案没有明确现有画布如何回填 canvas_versions。当前假设该表未建，已有 canvases.version 可能已经大于 1；如果只从新保存开始追加快照，用户基于部署前已打开的 baseVersion 保存时，服务端会取不到 baseData，导致混乱的 409 或无法合并。

**建议**：采用"A-lite"——0005 migration 建 canvas_versions，并为每个现有 canvas 插入一条当前版本快照 `(canvas_id, version=canvases.version, data=canvases.data)`；新建画布和每次成功保存必须在同一事务内追加快照。对找不到 baseVersion 的请求统一返回 `base_version_expired`，前端提示刷新并保留草稿。50/30 天清理可先做成同步保存后的轻量 opportunistic cleanup 或推到阶段 6，不应阻塞 P5 MVP。

#### H2 [edge-case] 判断点 1 / 草案 C 两路 diff

**问题**：草案 C 不只是"精度差"，而是会把部分真冲突误合并，尤其是 N6、N7、N9 这类需要知道 base→current 的场景。只看 client vs current 无法区分"对方刚改过"和"客户端自己基于旧状态改动"，会破坏阶段 5 的核心验收目标。

**建议**：不要把 C 作为 MVP 简化方向。若工期压缩，应简化快照保留策略和 UI，而不是取消 baseData。B 也不建议做，签名 baseData 的复杂度和信任边界都比建表更差。

#### H3 [missing-constraint] 判断点 3 / connector diff 与端点校验

**问题**：选择 3A 合理，但草案只说按 id 做 added/removed/modified，没有明确 connector 的语义去重规则和端点校验顺序。E2"同语义边去重"不能只按 id 判断，因为两个客户端可能用不同 id 创建同一 source/target/handle 语义边；E7/N9 又依赖合并后端点是否存在。

**建议**：为 connector 定义 `semanticKey = sheetId + sourceID + targetID + sourceHandle + targetHandle`，E2 按 semanticKey 去重，id 只用于同一边对象的修改冲突。apply 顺序明确为：sheet 增删校验 → 节点增改/废弃/删除权限 → connector 增改删 → 统一端点校验。主动新增边引用被对方删除节点走 N9 409；被动产生的悬空边走 E7 丢弃并进入 mergeReport warnings。

#### H4 [ambiguity] 判断点 4 / 去掉覆盖按钮后的冲突闭环

**问题**：倾向 4B 保守合理，但"只留查看他人版本/取消"会让合法覆盖诉求没有明确路径，尤其是 admin 接管、同人多窗口、对方误改等场景。草案说用户想覆盖必须重载再编辑再保存，但没有说明草稿如何带过去、如何避免用户误以为取消后数据已安全保存。

**建议**：落地 4B 时补一个明确流程：409 弹窗默认先 PUT draft；"查看最新版本"保存当前草稿并重载；"取消"只关闭弹窗但保留 dirty 与草稿状态；后续用户如果要覆盖，只能在最新版本上手动重新应用改动。若要给 admin 覆盖能力，也建议放到后续版本，并要求二次确认和写 conflict_logs。

#### H5 [missing-constraint] spec gap / 事务边界与副作用写入

**问题**：brief 没有单独列出事务边界取舍。阶段 5 会同时写 canvases、canvas_versions、nodes_meta、annotations 清理、conflict_logs、draft 删除或保留提示；如果这些操作不在一个明确事务模型内，可能出现主版本写成功但快照缺失，或 nodes_meta 与 JSON 不一致。

**建议**：把 `saveCanvas` 的事务作为 P5 门禁——`BEGIN IMMEDIATE` 内读取 current/base、计算 delta、权限校验、写 nodes_meta、rehydrate JSON、更新 canvases、插入 canvas_versions、插入 conflict_logs。409 冲突日志也应在独立短事务内写入，避免主事务回滚后日志丢失。保存成功后删除当前用户 draft 可以允许失败并静默保留，因为它不是主数据一致性的一部分。

### Medium（8 条）

#### M1 [ambiguity] 判断点 2 / node diff 字段语义

**问题**：选择 2A 复用现有 strip 思路是对的，但"直接调 strip + deepEqualJsonShape 判断单字段变化"还不够清楚。strip 解决的是 hydrated 元字段剥离，不等价于字段级 delta 规范；如果没有显式列出可 diff 字段，后续新增 UI 字段时可能被错误合并或错误忽略。

**建议**：采用"2A+薄白名单"——复用 `stripHydratedNodeMetaForCompareOrHydrate` 和 `deepEqualJsonShape` 做规范化比较，但为 node 和 connector 各自定义顶层 diff 字段清单。`is_deprecated` 单独进入 deprecatedChanged，不混在普通 modified；服务端 hydrated 字段永远不进入 changedFields。

#### M2 [missing-constraint] 判断点 2 / connector 字段白名单

**问题**：现有 strip 函数是节点元信息相关，不能自然覆盖 connector。connector 的 `style`、`labelStyle`、`data` 等嵌套字段如果没有限制，字段级合并可能出现同一嵌套对象不同 key 的误判。

**建议**：为 connector 定义 MVP 字段级粒度——至少覆盖 `sourceID/targetID/sourceHandle/targetHandle/label/edgeType/notImplemented/waypoints/style`。嵌套对象建议第一版按顶层字段冲突处理，例如双方都改 `style` 就 409，不做 style 内部 key 级合并，以节省实现和测试成本。

#### M3 [performance] 判断点 1 / canvas_versions 保留策略

**问题**：A 的 50 个版本 + 30 天保留在数据量上可接受，但"清理 cron"会拉长阶段 5 关键路径。当前阶段的目标是让合并可用，不是完整运维保留策略。

**建议**：P5 只实现建表、回填、追加、读取；清理策略先写成 TODO 或放阶段 6。若担心磁盘，保存成功后对当前 canvas 做一次简单删除：保留最近 50 条，且不碰最近 30 天。不要在 P5 引入独立 cron、调度器或后台 UI。

#### M4 [readability] 判断点 5 / 自动合并报告 UI

**问题**：倾向 5A 的"toast + 抽屉详情"产品上合理，但抽屉详情可能比合并核心更耗时，尤其还要解释字段级 diff、用户、时间和 E7 warning。4 天 MVP 下，这可能挤占单测和后端正确性。

**建议**：建议降级为"toast + 可复制/可展开的简短详情区域"，不做完整抽屉。必须显示 E7 丢弃边和自动合并数量；普通 added/modified 明细可只显示前 5 条和总数。后续再做正式抽屉。

#### M5 [other] 判断点 6 / conflict_logs 范围

**问题**：选择 6B 合理，但表结构需要提前保证能表达 auto_merged、conflict、base_version_expired、edge_dropped 这些结果，否则后续观察价值有限。只记录 409 会漏掉自动合并是否频繁、E7 是否造成困扰。

**建议**：P5 建表 + 写入路径，不做后台 UI。日志 details 存 JSON，至少包含 `resolution`、`baseVersion`、`currentVersion`、`newVersion`、`conflictTypes`、`warnings`、`actorUserId`。SQL 查询说明可以写在开发记录中，UI 推到阶段 6 后。

#### M6 [missing-constraint] spec gap / 返回数据语义

**问题**：方案响应里成功只返回 `version`、`merged`、`mergeReport`，但自动合并后客户端本地数据未必等于服务端最终 rehydrated data，尤其是 nodes_meta 重写、E7 丢弃边、对方改动并入后。若不返回最终 data，前端可能继续基于过时本地状态编辑。

**建议**：P5 保存成功响应建议返回 `data` 或要求前端在 `merged=true` 时立即 GET 最新画布。更简单的是：无并发保存可不返回 data；自动合并成功必须返回 `data`、`version`、`mergedFrom`，客户端用服务端 data 替换本地状态并清 dirty。

#### M7 [missing-constraint] spec gap / sheet 级 delta

**问题**：brief 聚焦 node/connector，但方案 §5.6 还涉及 sheet 增删和 sheet 元信息冲突。当前 saveCanvas 已按 sheet+node key 算节点 delta，但没有说明 sheet 新增、删除、重命名、activeSheetId 变化怎么合并。

**建议**：MVP 明确 sheet 策略——新增不同 sheet 可合并；删除 sheet 与对方修改该 sheet 冲突；双方改同一 sheet 的 `name` 或 project `activeSheetId/name/description` 同字段冲突；不同 sheet 的节点/边变化可合并。若 UI 实际很少改 sheet，可先保守 409，但必须写清楚。

#### M8 [edge-case] spec gap / 权限校验与合并顺序

**问题**：自动合并会把 deltaB 应用到 currentData 上，但 nodes_meta 权限校验仍必须只针对 deltaB。草案提到了现有路径正确，但没有强调合并后不能把对方 deltaA 的节点误算成当前用户修改，否则会重现 §5.7 曾经修过的问题。

**建议**：实现上把 `deltaB` 作为权限和 nodes_meta 同步的唯一输入；`mergedData` 只用于最终 JSON。单测加一条：A 保存自己的节点变更到 current，B 基于旧版本只改自己的节点，自动合并时不得检查或更新 A 节点的 creator/updated_by。

### Low（3 条）

#### L1 [readability] 判断点 4 / 冲突文案

**问题**：4B 去掉覆盖后，弹窗文案如果仍沿用 §5.8 的"覆盖"语义，会与实际按钮不一致，用户可能不知道下一步怎么处理。

**建议**：文案改为"画布已有新版本，你的改动已保存在草稿。请先查看最新版本，再决定是否重新应用你的改动。"按钮为"保存草稿并查看最新版本""继续编辑草稿"。

#### L2 [other] 判断点 5 / mergeReport 内容裁剪

**问题**：mergeReport 如果逐项返回所有节点/边，在大画布或批量操作时可能让响应变大、UI 噪音高。

**建议**：mergeReport 采用汇总 + 截断——每类最多返回前 20 条明细，额外给 `totalCount`。E7 warning 不应截断到完全不可见，至少保留总数和前几条 edgeId。

#### L3 [other] 门禁判断 / 4 天可完工性

**问题**：按 1A/2A/3A/4B/5A/6B 全量落地，4 天可以尝试，但关键路径很紧。最大风险是 3A 的边 field-level diff + 5A 抽屉 UI + 25+ 单测同时推进，容易牺牲测试质量。

**建议**：建议优先级为——canvas_versions 与迁移、delta/merge 核心、nodes_meta 权限不回退、connector add/remove/modified 基础冲突、自动合并后客户端状态替换、关键单测。UI 只做最小可用 toast/弹窗，conflict_logs 不做后台页。

## Recommendations（codex 建议最终方向表）

1. 判断点 1 选 **A-lite**：建 canvas_versions、回填当前快照、每次保存追加，清理延后或做轻量按画布清理
2. 判断点 2 选 **A+薄白名单**：复用现有 strip/deepEqual，但显式定义 node 和 connector 的 diff 字段，`is_deprecated` 单独处理
3. 判断点 3 选 **A**，但嵌套字段按顶层冲突，不做深层 key 合并；E2 用 semanticKey 去重
4. 判断点 4 选 **B**，并补全"先存草稿再查看最新版本"的流程；覆盖能力不进 MVP
5. 判断点 5 选 **A 的轻量版**：toast 必做，详情可展开但不做复杂抽屉；E7 warning 必须显示
6. 判断点 6 选 **B**：建 conflict_logs 并写入 auto_merged/conflict/base_version_expired 等结果，不做后台 UI
7. **额外门禁**：自动合并成功时前端必须拿到服务端最终 data 或立即重新 GET，否则本地状态会与服务端不一致

## Risks（codex 列出的核心风险）

1. 如果不建 canvas_versions 或不回填当前快照，阶段 5 会退化成不可靠的 409-only 或误合并
2. 如果 connector diff 只按 id，不按语义 key 去重，E2 会漏判，用户可能看到重复连线
3. 如果自动合并后客户端继续使用旧本地 data，下一次保存会基于错误状态，触发连续冲突或覆盖服务端 rehydrate 字段
4. 如果把合并报告 UI 和后台日志 UI 做得过重，4 天 MVP 的测试和事务一致性会被挤压
5. 如果 409 弹窗取消路径不自动保存草稿，用户会误以为改动安全但实际只在内存里
