## Issues 按 severity 排列

### High（4 条 —— Day 2 起步前必修）

#### H1 [missing-constraint] NODE_DIFF_FIELDS 漏 name 和 detailConfig（**真漏，必修**）

**位置**：`server/services/merge/types.ts:NODE_DIFF_FIELDS / StorageNode`；`src/types/flow.ts:FlowDefinition.nodes[]`

**问题**：按提供的 `FlowDefinition.nodes[]` 对表，客户端可持久化字段 `name`、`detailConfig` 没有进入 `NODE_DIFF_FIELDS`，`StorageNode` 类型里也没有显式声明。Day 2 的 `applyDelta` 如果只消费 delta，这两类修改会被 `computeDelta` 判定为未变化并**静默丢失**。

**建议**：Day 2 起步前确认真实 storage 形态——如果节点详情仍是顶层 `name/detailConfig`，把它们加入 `StorageNode` 和 `NODE_DIFF_FIELDS`，并补充对应单测；如果实际已经统一放进 `data`，则同步收敛 `src/types/flow.ts` 与服务端类型，避免白名单按错误模型维护。反向风险是字段过宽会引入非业务运行态字段，所以应只加已确认持久化字段。

#### H2 [missing-constraint] NodeDelta 缺 alsoDeprecated 信号

**位置**：`server/services/merge/types.ts:NodeDelta`；`server/services/merge/computeDelta.ts:computeNodeDelta`

**问题**：`NodeDelta` 没有显式表达"内容变化 + `is_deprecated` false→true"。当前这类输入只进入 `modified.changedFields`，且 changedFields 不含 `is_deprecated`；在 G4 约束下，`applyDelta` 无法仅凭 deltaB 判断是否要写 `nodes_meta.deprecated_by/deprecated_at`。

**建议**：Day 2 起步前给 `NodeDelta` 增加 `alsoDeprecated: boolean` 或更明确的 `deprecationChanged: 'none' | 'deprecated_now'`，在 `computeNodeDelta` 中对 `!baseDep && newDep && changedFields.length > 0` 置位，并补测试覆盖"改字段同时标废弃"。反向风险是继续让 applyDelta 回读 incoming 全量数据，会违反 deltaB 唯一权限源的设计。

#### H3 [edge-case] 重复 ID 索引覆盖

**位置**：`server/services/merge/computeDelta.ts:indexNodes / indexConnectors / computeDelta sheet 索引`；`server/services/canvases.ts:saveCanvas 快速路径索引`

**问题**：sheet、node、connector 重复 ID 现在都会被 Map 后写覆盖先写，`computeDelta` 不暴露输入损坏。Day 2 合并时这会把重复项误判成修改、删除或无变化，最坏会丢节点/边。

**建议**：Day 2 起步前在 schema 层拒绝 incoming 重复 `sheets[].id`、同 sheet 内重复 `nodes[].id`、同 sheet 内重复 `connectors[].id`；服务层/merge 层再加防御性校验，用于发现历史 DB 快照损坏。客户端输入建议返回 400，历史数据损坏建议返回明确 409/500 或 typed data-integrity 结果，不要继续合并。

#### H4 [missing-constraint] ConflictType 最小集合不足

**位置**：`server/services/merge/types.ts:ConflictType`

**问题**：`ConflictType` 目前不足以支撑 Day 2 冲突检测，已知缺 `node_modified_both` 和 `sheet_id_collision`；还缺少边 ID 撞车、同语义边属性冲突等 Day 2 很可能需要表达的类型。

**建议**：Day 2 写 detector 前先补齐最小集合：
- 节点类：`node_modified_both`、`node_deprecated_modified`、`node_removed_modified`、可选 `node_removed_deprecated`、`node_id_collision`、`node_meta_missing`、可选 `node_parent_removed_by_other`
- 边类：`edge_id_collision`、`edge_semantic_conflict`、`edge_modified_both`、`edge_removed_modified`、`edge_endpoint_removed_by_other`
- sheet/project 类：`sheet_id_collision`、`sheet_removed_modified`、`sheet_meta_conflict`、`project_meta_conflict`、可选 `active_sheet_missing`

反向风险是类型过细会增加前端分支，但类型不足会让冲突日志和弹窗无法解释真实原因。

### Medium（5 条 —— Day 2 顺手 / Day 3 顺手）

#### M1 [edge-case] semanticKey 规范化 + computeConnectorDelta 重复实现

**位置**：`server/services/merge/types.ts:connectorSemanticKey`；`server/services/merge/computeDelta.ts:computeConnectorDelta`

**建议**：统一只调用 `connectorSemanticKey`，把实现改为 `JSON.stringify([sheetId, sourceID, targetID, sourceHandle ?? null, targetHandle ?? null])`。同时明确空字符串 handle 是否允许且是否应与 undefined 等价；若不允许，schema 层直接拒绝空 handle。

#### M2 [ambiguity] 数组顺序持久化语义未确认

**位置**：`server/services/merge/types.ts:SHEET_META_DIFF_FIELDS / PROJECT_META_DIFF_FIELDS`；`src/types/flow.ts:MultiCanvasProject / CanvasSheet`

**问题**：当前 delta 用 Map 按 ID 比较 sheet、node、connector，**完全忽略数组顺序**。若 sheet tab 顺序、节点/边渲染顺序或层级顺序是持久化语义，Day 2 applyDelta 会静默丢失 reorder。

**建议**：Day 2 起步前确认这些顺序是否属于业务持久化字段。若是，应引入显式 `order` 字段或单独 order delta；若不是，应在 schema/保存过滤中保证运行态顺序字段不入 storage。

#### M3 [edge-case] is_deprecated true→false 静默忽略

**位置**：`server/services/merge/computeDelta.ts:computeNodeDelta`

**问题**：`is_deprecated` true→false 当前会被 `computeDelta` 静默忽略，因为该字段不在白名单且只识别 false→true。若客户端或旧草稿发来"取消废弃"，合并路径不会报错也不会显式拒绝，行为不透明。

**建议**：按既有单向废弃规则，在 schema 或 delta 计算阶段显式拒绝/记录 true→false；至少补测试证明该输入会被服务端强制保持废弃且不会更新 `updated_by/updated_at`。

#### M4 [missing-constraint] baseVersion zod 校验未确认

**位置**：`server/schemas/canvas.ts`（未提供给 codex）；`server/services/canvases.ts:saveCanvas 版本三分支`

**建议**：确认 `SaveCanvasRequestSchema` 使用有限正整数约束（int + positive）；服务层也建议保留轻量 guard。`base_version_expired` 应只用于"baseVersion 对应 canvas_versions 快照不存在"，不要复用给 JSON 解析失败、delta 计算异常或数据完整性错误。

> **注**：本项目 `server/schemas/canvas.ts:292` 已有 `baseVersion: z.number().int().positive()`，符合要求。Day 4 前再确认一次。

#### M5 [other] mergedData / report 目前 unknown 类型

**位置**：`server/services/canvases.ts:SaveCanvasResult merged=true 分支`；`server/routes/canvases.ts:PUT merged=true 响应`

## Recommendations（codex 阶段建议）

- **结论**：可以进 Day 2，但建议把"白名单补齐、NodeDelta deprecation 信号、重复 ID 校验、ConflictType 最小集合、semanticKey 统一实现"作为 Day 2 写 detector/applyDelta 前的**门禁**
- **Day 2 起步前必修**：补 `name/detailConfig` 或明确 storage 不含它们；给 `NodeDelta` 增加 `alsoDeprecated`；加 schema/service/merge 重复 ID 校验；补 `node_modified_both`、`sheet_id_collision` 等冲突类型；把 `computeConnectorDelta` 改为调用统一 semanticKey helper
- **Day 2 顺手**：补 baseVersion 正整数防御、废弃 true→false 的显式处理、sheet/node/connector 顺序语义确认，以及 React Flow 运行态字段是否入 storage 的审计
- **Day 3 顺手**：把 merged 分支类型从 `unknown` 收窄，确保 `mergedData` 是最终持久化后的权威数据，并补充合并报告中的 sheet/project 摘要
- **Day 4 处理**：复核 0005 migration 是否保留原表全部约束，并补冲突日志 details 的结构化格式

## Risks（codex 列）

1. 白名单漏字段会造成用户修改在自动合并中静默丢失，这是 Day 2 最大风险
2. 缺少 `alsoDeprecated` 会让内容修改同时标废弃的节点无法正确同步 `nodes_meta`，破坏 G4
3. 重复 ID 不先拦截会污染 deltaA/deltaB，后续 detector 的判断都不可信
4. 冲突类型不足会迫使 Day 2 用泛化 conflict，前端弹窗、日志和测试都难以对齐具体场景
5. 如果 mergedData 不是最终服务端重写后的数据，客户端 GATE-1 dirty 清零后会持有过期或伪造元字段
