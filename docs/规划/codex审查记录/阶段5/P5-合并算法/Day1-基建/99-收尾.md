## Day 1 关键经验（可复用）

### 1. codex confidence=high ≠ "它读过所有文件"

**踩坑**：二审 v1 codex 标 confidence=high 但 H2 给了一堆 React Flow 运行态字段（误报）。grep prompt.txt 命中 `[truncated by MaxContextChars]` 才发现 7 文件 103KB 被截 45%。codex 自己在 `notes_for_claude_code` 字段声明"未读三个文件"——是它的**自我盲区声明**。

**经验**：每次 codex 报告必须三步检查：(1) 看 `notes_for_claude_code` 自我盲区声明 → (2) grep prompt.txt 验证 truncated 标记 → (3) 才判断 issue 真假。后续 6 轮全部用 `-MaxContextChars 150000`，0 截断。

### 2. ⚠️ "修法描述 ≠ 修法实现"第 4 次踩

Day 1 nice M5（NODE_DIFF_FIELDS 防漂移）我用元组 `[T extends never ? true : ...]` 模式自审觉得编译通过 = 闭环。codex 五审 H1 揪出"TS 不会拒编译，是假断言"。改 `type AssertNever<T extends never> = T;` 真断言后 T 不是 never 时直接报 TS2344。

**经验**：CLAUDE.md 红线已有"交 codex 前必须 Read 完整文件"，但本次教训更深：**Read 完整文件不够，必须用反向测试证明断言/防漏机制生效**（如临时删 `'name'` 跑 tsc 验证报错）。

前 3 次同主题：0505 P3D-2 step 3 helper 写没用 / 0506 P3B 复制粘贴 idMap 死代码 / 0506 P4 useDraftAutosave interval 被 effect deps 重置。

### 3. schema 注释 placeholder 等于零保护

codex 四审 M2 揪出 schemas/canvas.ts:224-228 是空 if + "暂时只 warn 不阻塞"注释——**校验形同虚设**。我之前 brief 说"schema 已校验 parentId"是错的判断。

**经验**：注释里的"暂时""TODO""真要做..."等模糊语病在代码中等于零保护，**必须登记技术债务台账**。如果当时把"parentId 校验是 placeholder"写进 docs/规划/技术债务登记.md，本次审查不会因 brief 描述错误浪费一轮。
