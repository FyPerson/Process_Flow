## Issues 按 severity 排列

### High（1 条 —— 必修）

#### H1 [missing-constraint] M5 静态断言假闭环 ⚠️

**位置**：`server/services/merge/types.ts: _NodeDiffFieldsCovered`

**问题**：当前双向 Exclude 只是生成了一个**合法类型别名**；即使 NODE_DIFF_FIELDS 漏字段，结果也只是变成 `[true, ['StorageNodeContentFields has keys not in NODE_DIFF_FIELDS:', 'xxx']]`，**TypeScript 不会拒绝编译**。也就是说 M5 现在结构上像断言，实际不是断言。

**建议**：改成带约束的断言类型：

```ts
type AssertNever<T extends never> = T;
type _NodeDiffFieldsForwardAssert = AssertNever<_NodeDiffFieldsForward>;
type _NodeDiffFieldsBackwardAssert = AssertNever<_NodeDiffFieldsBackward>;
```

可再补一个 type-level 测试或临时 ts-expect-error 验证漏字段确实会失败。

→ **必修**：本是 v4 M5 的修法目标，没真闭环就等于没修

### Medium（3 条 —— 必修 1 + 挂账 2）

#### M1 [missing-constraint] parentId 没校验自引用 + 环

**位置**：`server/services/merge/computeDelta.ts: assertProjectIntegrity` + `server/schemas/canvas.ts: superRefine`

**问题**：parentId 只校验存在且父节点 `type=group`，但没有拒绝自引用和环。例如 group 节点 parentId 指向**自身**会通过；两个 group **互相 parentId** 也会通过。这会污染层级结构，后续布局、删除、权限或前端渲染都可能出现不可预期行为。

**建议**：
- 如果业务允许嵌套 group，也应按 sheet 做 DFS/visiting 状态检测，拒绝 parentId 环
- 至少显式拒绝 `node.parentId === node.id`
- 如果不允许 group 嵌套（**本项目实际不允许**——P3D-2 mergeSnapshotByPermission 已有"group 嵌套清理"逻辑），则直接拒绝 type='group' 的节点设置 parentId
- schema 和 assertProjectIntegrity 两处保持同一规则

→ **Day 2 起步前必修**（约束本项目"不允许 group 嵌套"硬规则）

#### M2 [ambiguity] DetectContext 不足以承载 Day 2 apply 职责

**位置**：`server/services/merge/types.ts: DetectContext`

**问题**：DetectContext 对 detectNodeConflicts 判 `node_modified_both` 基本够用，但不足以承载 Day 2 applyDelta/权限校验。删除权限需要当前用户角色和 private owner 信息；node_meta_missing、creator 校验、updated/deprecated 元信息同步需要 nodes_meta 索引或等价查询结果。把这些需求都塞进 DetectContext 会让 detector 与 apply **职责混杂**。

**建议**：保持 DetectContext 聚焦 detect（`userA / userB / visibility` 可保留）；另定义 ApplyContext 或 PermissionContext，包含：
- `userBRole / isAdmin`
- `ownerId`
- `nodesMetaIndex`
- `now / canvasId`

`currentVersionSavedAt` 更适合 report/log/UI 文案上下文，不应成为冲突判定必需字段。

→ **Day 2 detect 实施时拆**：detector 不动，apply 时新增 ApplyContext

#### M3 [edge-case] 启用强校验后历史脏数据可能集中暴露为 500

**位置**：`server/services/merge/computeDelta.ts: assertProjectIntegrity 启用范围`

**问题**：新增 activeSheetId、endpoint、parentId 强校验后，**历史 canvases.data 或 canvas_versions 快照**中只要已有损坏数据，Day 2 合并路径会直接抛 DataIntegrityError，可能把并发保存从可恢复 409 变成 **500**。当前 brief 只说明测试通过，不能证明生产存量数据都满足新约束。

**建议**：Day 2 合并路径上线前做一次**只读完整性审计**，覆盖 canvases 当前 data 和 canvas_versions 历史快照。对 dangling connector、无效 parentId、缺失 activeSheetId 制定修复策略：
- 删除悬空边
- 清空非法 parentId
- 把 activeSheetId 指到首个 sheet

→ **Day 4 部署前必跑**（脚本 + 修复策略）；本项目生产数据 1 个画布（"文博业务流程图"）+ 测试 db 各几个，规模可控
