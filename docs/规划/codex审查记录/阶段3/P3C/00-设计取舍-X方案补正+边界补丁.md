**结论**

可以进入实施，但不要按原 X 方案“只改 save rewrite”直接做。P3C 方向整体合理，X 也比 Y/Z 更合适，但需要补几个实现边界，否则 tooltip 元数据会出现空值、伪造展示或保存链路丢字段。

**必修项**

1. X 方案要改成“GET 返回时以 `nodes_meta` 为准”
   现在 `GET /api/canvases/:id` 是直接 `JSON.parse(row.data)` 返回，单改 [server/services/canvases.ts](</E:/业务全景图/server/services/canvases.ts:478>) 的保存 rewrite 不能保证现有已废弃节点带上 `deprecated_by/at`。更稳的是 GET 时一次查 `nodes_meta`，把节点元信息 hydrate 进响应。

2. 保存 rewrite 不能直接读旧 meta 塞 by/at
   当前流程是先 rewrite `canvases.data`，后更新 `nodes_meta.deprecated_by/at`。如果刚执行标废弃，rewrite 读到的还是旧的 `null`。要么调整顺序，要么在 rewrite 时对本次 `deprecatedChanged/alsoDeprecated` 显式写 `deprecated_by=user.id`、`deprecated_at=now`。

3. `stripMeta` 必须加入 `deprecated_by/deprecated_at`
   不会和 `alsoDeprecated` 冲突，反而是必须的。否则客户端只伪造 by/at 也会被误判为 content modified，普通用户标废弃可能被错误拦到 creator/admin 权限路径。

4. schema 不要写入不被接受的 `null`
   如果节点里可能出现 `deprecated_by: null` / `deprecated_at: null`，schema 必须是 `z.number().int().nullable().optional()`。更推荐：未废弃节点直接 omit；废弃但历史数据缺失时也 omit，让前端 tooltip fallback 到“已废弃”。

5. 补“新建后立刻标废弃”边界
   现在 added 节点的 `nodes_meta.is_deprecated` 插入为 0。如果用户新建节点后还没保存就点标废弃，会造成 `canvases.data` 和 `nodes_meta` 不一致。要么前端禁止未持久化节点标废弃，要么服务端 added+`is_deprecated=true` 时同步写 `deprecated_by/at`。

6. 要显示“X”，只返回用户 ID 不够
   如果 tooltip 文案是“张三在 YYYY-MM-DD 标废弃”，需要 JOIN users 返回 `deprecated_by_username`，或前端已有用户表映射。否则只能显示“用户 #12”。

**取舍意见**

取舍 1 推荐 A。NodeDetailPanel 单入口符合“低频、审计、不可逆”动作，能降低误触。右键和 hover 都更适合高频编辑，不适合单向审计动作。

取舍 2 推荐 A，但实现时不要让红色角标也跟着 `opacity=0.5` 变淡。建议节点主体半透明，角标保持不透明；tooltip 挂在角标或整节点上都可以。

取舍 3 不做取消废弃是成立的，但必须有二次确认。不要做 toast undo，除非服务端支持撤销；否则会制造“看起来能撤、实际不能撤”的错觉。

取舍 4 放前端。边的半透明是派生视觉状态，不要写回服务端，不要改 connector storage。最清爽做法是在构造 React Flow edges 时根据 source/target 节点的 `is_deprecated` 注入 `data.hasDeprecatedEndpoint` 或 `className`，由 `DraggableEdge`/CSS 统一降 opacity、弱化 label、必要时关闭 animated。

**建议项**

明确废弃后节点是否还能编辑、连线、被 NodeSelector 选作关联节点。至少 UI 上应弱化这些入口；否则“已废弃”会变成纯视觉标签，审计语义不够强。

确认 group 节点是否允许标废弃。若允许，不建议默认级联废弃子节点；若不允许，前端按钮直接隐藏或禁用。

二次确认里建议展示影响范围：例如“将标记该节点为废弃，并弱化 3 条相关连接；普通用户无法撤回”。这比单纯警告更能减少误点。
