## 第二层（codex 原话技术细节）

### 🔴 Issue 1（high / edge-case）— 复制 group + children 时 child position 错误偏移

- **Location**：`src/hooks/useFlowClipboard.ts, remapClipboardNodes`
- **Problem**：父节点也在剪贴板内时，子节点的 `position` 仍统一加了 `OFFSET_X/OFFSET_Y`。React Flow 子节点的 `position` 是相对父节点坐标；新 group 已经整体偏移后，child 再偏移会改变组内布局，表现为复制分组后子节点在新分组内额外右下偏移。
- **Suggestion**：当 `parentId` 命中 `idMap` 时保留子节点原始相对坐标，不再加粘贴偏移；只对无父节点的根节点、或脱离旧父节点后转为绝对坐标的节点加偏移。补单测断言：group 原位置 (100,100)、child 相对 (10,10) 粘贴后 group 为 (130,130)，child 仍为 (10,10)。

### 🔴 Issue 2（high / contradiction）— 半组复制分支不可达

- **Location**：`src/hooks/useFlowClipboard.ts, remapClipboardNodes / copyNodes`
- **Problem**：任务说明声称半组选区会"按源父定位转绝对坐标"，但当前实现中 `sourceParent = clipboardData.find(n => n.id === oldParentId)` 在 `idMap.has(oldParentId)` 为 false 后实际不可达，因为 `idMap` 正是由 `clipboardData` 全量构造的。只复制 group 内 child 时没有父节点位置上下文，只会删除 `parentId/extent` 并保留相对坐标，粘贴位置可能错误。
- **Suggestion**：让 `copyNodes` 在复制时拿到全量节点列表或父节点 lookup，并把 child 的绝对坐标或源父 position 写入剪贴板元信息；或者把 `remapClipboardNodes` 签名改为接收 `sourceNodeById`。如果产品决定不支持半组选区绝对定位，则应修改行为约定和测试描述，不能继续声明已实现。

### 🟡 Issue 3（medium / edge-case）— 单测没断 position

- **Location**：`src/hooks/useFlowClipboard.test.ts`
- **Problem**：新增单测没有覆盖上述坐标语义：case 2 只断言 `parentId`，不检查 child 相对坐标是否被错误偏移；case 3 标题写"转绝对坐标"，但只断言删除 `parentId/extent`，没有断言 position。现有验证记录因此无法证明 Issue 1 的坐标修法到位。
- **Suggestion**：补两个位置断言：完整复制 group+child 时 child 相对坐标不变；半组选区在有父节点上下文时输出 `parent.position + child.position + OFFSET`，无父上下文时明确断言采用降级策略。

### ⚪ Issue 4（low / other）— lint 拼接绕过

- **Location**：`scripts/check-id-generation.mjs`
- **Problem**：lint 规则已比二审前更强，但仍只覆盖直接模板字符串前缀。常见绕过还包括 `'n_' + Date.now()`、`${prefix}_${Date.now()}`、多行模板字符串、`new_edge_${Date.now()}` 等。作为回归保护可以接受，但不能视为完整静态证明。
- **Suggestion**：至少补充字符串拼接和 `new_edge/new_group` 模式；如果后续还要扩大门禁价值，可用 AST 检查 `Date.now/performance.now/getTime/toString(36)` 出现在疑似 id 赋值上下文中，并维持 helper 文件 allowlist。

## Recommendations

1. **门禁结论**：暂不建议直接进 commit、bump v1.10.0、三端部署；先修 `remapClipboardNodes` 的坐标逻辑并补单测后，再用现有 lint、tsc、npm test、Playwright 流程复验。
2. `duplicateSheet` 与 `remapClipboardNodes` 分两套逻辑本身可以接受，因为一个操作 storage 全量 sheet，一个操作 React Flow 剪贴板局部节点；不建议为了抽象而强行合并。但可抽出小 helper（生成 `nodeIdMap`、remap `parentId/relatedNodeIds`、过滤 connector 端点）避免未来前缀规则再次分歧。
3. 测试文件 type-check 技术债在当前内网小规模部署语境下可以不作为阻塞项，前提是生产 tsconfig 已过、测试能运行、CI 明确不会把测试 TS 类型错误当部署门禁。建议顺手移除 `@ts-expect-error` 这类未来会被 test type-check 卡住的写法。
4. Playwright case 4d 的 storage 层检查不能证明运行时 `data.id === node.id`，因为 storage 格式没有嵌套 `data.id`；这点主要依赖 `useFlowClipboard.test.ts` 的纯函数单测即可，端到端文案建议改清楚。

## Risks

1. 复制完整分组后组内节点布局可能发生额外偏移，用户会看到粘贴出的分组内部结构错位。
2. 只复制分组内子节点时，因缺少源父节点坐标，粘贴位置可能落在相对坐标附近，和用户预期的画布位置不一致。
3. 当前 lint 规则可防止已知模板字符串回退，但无法阻止所有手写时间戳 ID 生成方式。
4. `duplicateSheet` 依赖后续节点构造逻辑根据缺失 `creator_id` 标记本地新增；本次提供内容未包含该调用方，深审需读取 BusinessFlowVisualization 或对应构造函数。
