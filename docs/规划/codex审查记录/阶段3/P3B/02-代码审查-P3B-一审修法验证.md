## 第二层（codex 原话技术细节）

### 🔴 Issue 1（high / contradiction）— pasteNodes idMap 只写不用

- **Location**：`src/hooks/useFlowClipboard.ts` / Issue 1 修法说明
- **Problem**：任务说明称 parentId 和 relatedNodeIds 已全部 remap，但 pasteNodes 里 idMap 只写入不使用；返回的新节点仍保留 copied.parentId，cleanData 也保留源 relatedNodeIds。复制分组内节点或带 relatedNodeIds 的节点时，新节点会继续指向旧分组或旧节点，若旧对象仍在同一 sheet，当前"引用指向当前 sheet 内节点"的断言也会误通过。
- **Suggestion**：在 pasteNodes 内显式处理引用：parentId 若在 idMap 中则映射到新父 ID；若不在本次复制集合中则删除 parentId/extent 并把坐标转为绝对坐标，或直接拒绝半组选区；relatedNodeIds 应只保留并映射 idMap 命中的引用。补用例覆盖复制 group+children、只复制 group 内 child、复制带 relatedNodeIds 的节点，并断言新副本不指向旧对象。

### 🔴 Issue 2（high / edge-case）— 复制 group 用错 helper

- **Location**：`src/hooks/useFlowClipboard.ts`；`src/hooks/useMultiCanvas.ts duplicateSheet`
- **Problem**：复制路径没有按对象类型选择 ID helper。useFlowClipboard.pasteNodes 对所有节点使用 newNodeId；useMultiCanvas.duplicateSheet 也对 sourceSheet.nodes 中的所有节点使用 newNodeId。只要复制内容包含 group，分组节点会变成 n_<...>，违反 newGroupId / g_<...> 的约定，也削弱了"三种 ID 前缀互不相同"的测试目标。
- **Suggestion**：在两个路径中按 node.type 分派：group 用 newGroupId，普通节点用 newNodeId，边用 newEdgeId。随后补单测或端到端断言：复制画布含分组时，新分组 ID 必须匹配 `/^g_[A-Za-z0-9_-]{10}$/`，子节点 parentId 指向这个新的 g_ ID。

### 🟡 Issue 3（medium / edge-case）— pasteNodes 未同步 data.id

- **Location**：`src/hooks/useFlowClipboard.ts pasteNodes`
- **Problem**：粘贴新节点只更新 React Flow node.id，没有同步 data.id。虽然 convertNodesToStorage 主要使用 node.id，但代码中已有 data.id 字段约定；粘贴后 node.id 与 node.data.id 不一致，后续若有 UI、权限、调试或关联逻辑读取 data.id，会拿到旧节点 ID。
- **Suggestion**：构造 cleanData 后设置 cleanData.id = newId；如果是 group，也同步 data.id，并保持 label/name 等字段不变。建议加断言：粘贴后每个节点 node.id === node.data.id。

### 🟡 Issue 4（medium / other）— lint 脚本模式过窄

- **Location**：`scripts/check-id-generation.mjs`
- **Problem**：lint:ids 是有价值的兜底，但当前 FORBIDDEN 模式偏窄，容易漏掉等价 Date.now 写法，例如 `n_${Date.now()}`、`g_${ts}`、`g_${Date.now()}`、`edge_${Date.now()}`、`edge_${ts}`、`node_${Date.now()}` 或换成 timestamp/now 变量名。SKIP_FILES 整体跳过 utils/ids.ts 也可能掩盖核心 helper 内部被改回时间戳实现。
- **Suggestion**：要么改成 AST/语义扫描；要么至少扩展 regex 覆盖 n/g/e/group/edge/node 与 Date.now/ts/timestamp/now/baseTs 的组合。utils/ids.ts 不建议整文件跳过，可以只忽略注释，或单独断言该文件不得出现 Date.now。

### ⚪ Issue 5（low / other）— 测试文件无类型检查

- **Location**：`tsconfig.client.json`；`package.json`
- **Problem**：把 *.test.ts 排除出 client type-check 对 Vite/client 构建是合理的，但当前 npm test 通过 tsx 执行测试，不等价于 TypeScript 类型检查；测试文件本身的类型错误可能不会在 tsc --noEmit -p tsconfig.client.json 中暴露。
- **Suggestion**：不作为部署阻塞项。更完整的做法是新增 tsconfig.test.json，include src/**/*.test.ts 和 server/**/*.test.ts，并在 CI 或 npm test 前增加一次测试类型检查；若当前项目只追求运行时回归，现状也可以接受。

## Recommendations

1. **门禁判断**：暂缓 commit、版本 bump 和部署，先修复 pasteNodes 的引用 remap/清理，以及复制分组时使用 newGroupId 的问题。
2. Issue 1 不需要强行上 remapCopiedNodeReferences helper；但无论是否抽 helper，所有复制入口都必须有同一套可测试的 parentId、relatedNodeIds、connector sourceID/targetID 处理规则。
3. 端到端 case 4b 的断言应从"引用存在于当前 sheet"升级为"新副本引用新副本，不引用源对象"，否则旧对象仍存在时会漏检。
4. npm test 前置 lint:ids 属于合理的 fail-fast。

## Risks

1. 动态字段风险仍无法从提供片段完全排除（FlowNodeData 有 `[key: string]: unknown`），必须用类型审计和测试白名单共同约束。
2. 复制/粘贴和复制画布是低频但高破坏面的路径，一旦产生旧 ID 引用，保存成功后可能形成语义错误而不是 schema 错误，后续排查成本较高。
3. lint 脚本目前更像已知坏模式扫描，不是完整防线；未来开发者稍换写法就可能绕过。
