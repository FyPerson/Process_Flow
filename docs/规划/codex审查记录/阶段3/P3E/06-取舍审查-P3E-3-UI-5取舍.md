**第二层：技术细节（codex 原话）**

> codex-cli 0.128.0 / spec-critique / advice-only / confidence: high
> 实际耗时：约 2 分钟
> 原文 wrapper：`%TEMP%\codex-bridge-workspace\runs\codex_spec-critique_20260505_143331.json`
> 上下文文件（7 份）：NodeDetailPanel/index.tsx + NodePropertiesPanel + GroupPropertiesPanel + CustomNode + GroupNode + useAnnotations + 01-取舍审归档

**High 1：取舍 1 与现实不匹配**

[src/components/NodeDetailPanel/GroupPropertiesPanel.tsx] 推荐 B 写成"两个 PropertiesPanel 各加 tab"，但提供代码里只有 NodePropertiesPanel 有"属性 / 外观 / 数据"tabs，GroupPropertiesPanel 当前没有 tab 结构。按 B 实施会迫使分组面板新建一套 tab UI，且 pendingPanelTab 还要同时控制两个不同子组件，复杂度高于描述。

**修法（codex 推荐）**：先重新拍板：
- (a) 改为 NodeDetailPanel 顶层二级切换"详情 / 批注"，统一普通节点和分组节点
- (b) 保留 B，但明确 GroupPropertiesPanel 只新增"详情 / 批注"两段式 tab，而不是复制普通节点的四 tab

无论哪种，都把批注正文抽成共享 AnnotationTab/AnnotationPanel 组件。

**High 2：取舍 3 缺 nodeId 绑定**

[取舍 3 / pendingPanelTab 机制] pendingPanelTab 只描述了从徽章点击到打开批注 tab，但没有定义消费时机和目标节点匹配规则。若用户快速点击不同节点，或 selectedElement 更新晚于 pendingPanelTab，可能把错误节点的面板切到批注 tab。

**修法**：pendingPanelTab 不要只有 tab 值，建议建模为 `{ nodeId, tab: 'annotations' }`。NodeDetailPanel 在 `selectedElement.type === 'node' && selectedElement.data.id === pending.nodeId` 时消费，并通过 `onPendingPanelTabConsumed()` 清空。

**Medium 1：徽章位置共存**

[取舍 2 / CustomNode 与 GroupNode 徽章位置] 只说右上角绝对定位，但已废弃角标也在右上。若两个角标同侧堆叠，节点尺寸小、菱形 decision 节点、折叠 group 节点都会更容易遮挡标题或 handle。

**修法**：徽章固定放左上角，废弃角标继续右上角；两者同时存在时左右分离。徽章尺寸固定为最小宽度 24px、高 18px，pointer-events 只作用在徽章本身，z-index 高于节点内容但低于连接 handle，避免影响拖线。

**Medium 2：徽章宽度抖动**

[取舍 4 / 徽章数字展示] 99+ 本身合理，但未规定徽章宽度。数字从 9 到 10、99 到 99+ 时可能导致节点内布局或绝对定位视觉抖动。

**修法**：徽章用固定或最小宽度：1-2 位数保持 22-24px，99+ 保持 30-32px；字体 11-12px，line-height 固定。由于服务端 unresolved 上限是 100，99+ 仍值得做，成本很低。

**Medium 3：批注 tab 内部规则未定**

[遗漏取舍点 / 批注 tab 内部规则] P3E-3 的核心 UI 细节仍未定：新增表单、resolve/reopen 按钮、显示已解决开关、resolved 视觉、错误反馈、长批注折叠和空状态都会影响组件结构与测试断言。

**修法**：进实现前补一段 UI 规则：
- 新增表单放 tab 顶部，按钮文案"发布批注"
- 批注列表在下方按 created_at 正序
- 每条批注右上放 resolve/reopen 小按钮
- 默认只显示 unresolved
- 开关放列表标题右侧
- resolved 使用灰度淡化并显示解决人/时间
- 错误优先 inline 错误条，不用 alert

**Medium 4：loading 与空 Map**

[取舍 5 / loading 与空 Map] loading 期间 hook 返回空 Map 会让徽章后出现，通常可接受，但如果详情面板已经打开批注 tab，空状态可能短暂显示"暂无批注"，随后跳变成列表，语义上像数据变化而不是加载。

**修法**：节点徽章不需要 placeholder；批注 tab 内需要使用 hook 的 loading 状态显示轻量加载行，避免把 loading 当成真正空状态。游客、本地草稿、readReady=false 才显示空状态或不可用文案。

**Low 通过项**：

- ✅ 常显徽章符合 1.C "一眼可见"目标；onClick 阻止冒泡后执行选中节点 + 打开批注 tab
- ✅ BFV 顶层 state + props 驱动优于 ref imperative 和 CustomEvent
- ✅ 游客/本地草稿/loading 由 useAnnotations 返空 Map，节点层只按 count > 0 渲染徽章，无需重复判断

**recommendations**：

- 优先重新确认取舍 1：codex 倾向 NodeDetailPanel 顶层"详情 / 批注"切换
- 如坚持 B，必须把 AnnotationPanel 做成纯组件，输入 sheetId/nodeId/annotations/loading/error/canCreate/onCreate/onResolve/onReopen
- 徽章位置左上角；折叠 group 节点也沿用左上角
- pendingPanelTab 携带 nodeId
- 进 P3E-3 前把 7 条 UI 规则转成最小测试断言：默认隐藏 resolved / 新增 +1 / resolve -1 / reopen +1 / loading 不显示假空状态

**risks**：

- B 方案直接做 → GroupPropertiesPanel 引入临时 tab → 普通节点和分组节点批注行为不一致
- pendingPanelTab 不绑 nodeId → 快速切换时打开错节点的 tab
- 徽章与废弃同右上 → 小节点 / decision / 折叠分组遮挡严重
- tab 内规则不先拍 → P3E-3 实施时反复返工
