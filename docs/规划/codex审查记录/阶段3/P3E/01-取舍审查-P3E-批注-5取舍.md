**High 1：游客可见批注的文档矛盾**

[docs/规划/多人协作-方案.md §4.4](E:/业务全景图/docs/规划/多人协作-方案.md) 游客不可见批注与原 API 表述"GET 可游客（按画布权限）"冲突。如果只在实现里临时拒绝游客，后续 P3G 发布公共画布时容易出现文档、接口和 UI 预期不一致。

**修法**：把 §4.4 改成：GET /api/canvases/:cid/annotations **仅 JWT**；游客请求返回 401/403；前端游客模式不发 annotations 请求，节点徽章也不显示。

**High 2：徽章数字来源的状态复杂度**

[取舍 1.C / P3E-2 数据层] 1.C 能规避浮层定位复杂度，但没有规避"徽章数字来源"的状态复杂度。徽章需要按 canvas + sheet + node 聚合 unresolved 数量，并在新增、resolve、reopen 后同步更新，否则画布徽章和详情 tab 会短暂或长期不一致。

**修法**：P3E-1/P3E-2 明确定义返回和缓存口径——
- GET 返回整张画布批注列表（不按节点拆请求）
- 前端派生 `annotationCountByNodeKey`，key 使用 `${sheetId}:${nodeId}`
- 徽章只统计 unresolved
- POST/resolve/reopen 成功后直接更新本地缓存
- 保存画布不影响批注缓存

**High 3：服务端 resolve/reopen 权限校验路径未拍**

[P3E 权限规则] 批注 resolve/reopen 的权限依赖"批注作者 + 节点 creator + admin"，但目前取舍里没有明确服务端如何校验 node creator，尤其节点被物理删除后 annotations 会被删，节点被标废弃后是否仍允许加批注/关闭批注也未拍板。

**修法**：P3E-1 写死服务端校验——
- POST 必须验证最新 canvases.data 中 sheetId/nodeId 存在
- resolve/reopen 通过 `annotations.sheet_id/node_id` 查 `nodes_meta.creator_id`
- 已物理删除的批注理论不存在（§5.7 同事务清理已落地）
- **标废弃节点建议仍允许新增和关闭批注**（废弃本身可能需要讨论）

**Medium 4：批注展示规则未定**

[遗漏取舍点] 批注列表顺序、resolved 是否默认展示、徽章是否计入 resolved、长内容展示都未定，会直接影响 UI 和测试断言。

**修法**：
- 列表按 created_at **正序**（形成对话上下文）
- 徽章只计 unresolved
- tab 默认显示 unresolved，提供"显示已解决"开关
- 长批注在面板内保留换行并限制单条最大高度，超出滚动或折叠

**Medium 5：content 长度 + 单节点上限未定**

[docs/规划/多人协作-方案.md §3.1] annotations 表可满足 P3E MVP，但 content 缺少数据库层长度约束，方案也未给同节点批注数量上限。只靠前端限制不够，恶意或误操作仍可能写入大量文本。

**修法**：不一定要改表结构，但 P3E-1 的 Zod 和服务端必须限制——
- content trim 后 1-2000 字符
- 单节点 unresolved 批注上限 100 或全部 500
- 超限返回 400/409 + 中文提示

**Medium 6：P3E-2 估时偏乐观**

[取舍 5] P3E-2 单独抽 hook 不算过细，但估时 0.3 天偏乐观。hook 会承担加载时机、游客跳过、sheet/node key 聚合、乐观更新、错误回滚、与选中节点联动，实际是后续 UI 是否顺的关键。

**修法**：保留 P3E-2 独立 step，但把验收写清楚，不建议并入 P3E-3。若要压缩流程，可把 P3E-2+P3E-3 放同一天同一 PR，但仍分两个 commit 和两组验收。

**Low 1（通过）：取舍 1/2/3 整体方向成立**

1.C、详情面板内新建、纯文本且永久不做 mention/markdown 这三个取舍成立，能复用 P3D-2 的权限和 disabled 地基，并避免浮层、右键菜单和 XSS 面扩大。

**recommendations**：

- 进入 P3E-1 前先补一小段"批注产品规则"
- API 保持 4 个端点；GET 只对登录用户开放；所有端点都必须复核画布访问权
- 前端 hook 以整画布批注为缓存单位，节点徽章从缓存派生（不要每个节点单独请求）
- schema 暂不需要为软删批注加字段
- P3E-4 测试至少覆盖：游客无徽章无请求 / 新增后徽章 +1 / resolve 后徽章 -1 / reopen 后徽章 +1 / 跨 sheet 同 node_id 不串数

**risks**：

- §4.4 不改 → P3G 公共发布时可能误把团队批注暴露给游客
- 徽章计数无统一派生源 → 状态漂移
- 无 content 长度限制 → 误粘贴大文本卡顿
- resolve/reopen 只看 annotation id 不复核画布权限 → 跨画布越权风险
