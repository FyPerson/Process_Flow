## 第二层（codex 原话技术细节）

### Issues

#### high - edge-case (BFV 鉴权未完成竞态)
- **location**：`src/pages/BusinessFlowVisualization/index.tsx 初始化 useEffect`
- **problem**：用 `user === null` 判断游客会把"鉴权尚未完成"和"确定未登录"混在一起。如果 useAuth 初始 user 为 null，登录用户首屏可能先加载 demo 数据；之后 user 变成登录态时 project 已存在，逻辑直接返回，导致登录用户仍看到 demo。
- **suggestion**：初始化默认数据前等待鉴权状态稳定，使用 auth 的 loading/status 字段区分"未知""游客""已登录"。只有明确 guest/unauthenticated 时加载 demo，明确 authenticated 时创建空项目。

#### high - edge-case (并发回滚不安全)
- **location**：`src/hooks/useAdminCanvases.ts patchCanvasCb / archiveCanvasCb`
- **problem**：乐观更新的失败回滚不是并发安全的。两个 patch 同时作用于同一画布时，先发的请求失败会用旧 snapshot 覆盖后发且已成功的本地结果；archive 与 patch 交错时也可能把已归档移除的画布插回或覆盖字段。
- **suggestion**：给每行维护 mutation 序号，回滚前确认当前行仍处于本次 mutation 产生的状态；或简化为任一 PATCH/DELETE 失败后直接 refetch 权威列表，避免用过期 snapshot 做本地合并。

#### medium - edge-case (空草稿误标 dirty)
- **location**：`src/pages/BusinessFlowVisualization/index.tsx loadProject(createEmptyProject())`
- **problem**：登录用户首次进入无本地数据时，createEmptyProject 通过 loadProject 写入并 markDirty，会把系统自动创建的空草稿当成用户改动。结果可能触发 beforeunload 提示、保存状态显示未保存。
- **suggestion**：为空草稿初始化提供不标 dirty 的入口，例如 loadProject(data, { dirty: false })；只有用户实际编辑、重命名、加节点后再 markDirty。

#### medium - edge-case (CanvasSwitcher 切换丢改动)
- **location**：`src/components/CanvasSwitcher/index.tsx handlePick`
- **problem**：切换画布时没有检查当前 BFV 的 dirty/saving/conflict。setSearchParams 是 SPA 内部跳转，不触发 beforeunload；用户从下拉切到其他画布时，本地未保存改动可能被静默丢弃。
- **suggestion**：把 dirty/saving/conflict 或 onBeforeSwitch 回调传给 CanvasSwitcher，在选择非当前 canvasId 前弹确认；取消则不改 URL。

#### medium - other (patch 后排序陈旧)
- **location**：`src/hooks/useAdminCanvases.ts patchCanvasCb`
- **problem**：patch 成功后不 refetch，updated_at 使用 Date.now() 且不重新排序，会让管理员列表显示的更新时间和排序与服务端不一致。
- **suggestion**：patch 成功后至少用本地 updated_at 重新排序；更稳妥是成功后 refetch。

#### low - ambiguity (隐藏字段仍提交)
- **location**：`src/pages/Admin/CanvasesTab.tsx handleSaveEdit`
- **problem**：编辑私有画布时 UI 隐藏 is_public_to_guest 复选框，但提交仍发送 editing.is_public_to_guest。语义上会把隐藏字段也写回服务端。
- **suggestion**：构造 patch 时按可见性条件发送字段：public 才包含 is_public_to_guest，private 只发 name/description。

#### low - readability (调试日志残留)
- **location**：`src/api/canvases.ts / src/pages/BusinessFlowVisualization/index.tsx / src/hooks/useMultiCanvas.ts`
- **problem**：多个 DEBUG #15 console.log / console.error 仍在生产路径中。内网风险较低，但会污染控制台。
- **suggestion**：部署前移除调试日志，或统一放到受环境变量控制的 debug logger 下。

### Recommendations

1. 门禁建议：修复两个 high 后再进入 P3F-4；medium 可按发布风险决定是否同批处理。
2. Admin 画布列表优先选择"失败 refetch / 成功轻量 refetch"简单策略，内部低并发场景下比复杂本地合并更可靠。
3. BFV 默认数据逻辑以"鉴权已确定"为前置条件，否则测试全绿也可能漏掉真实登录首屏竞态。

### Risks

1. 如果 auth 初始态确实是 user=null，登录用户仍可能看到并保存 demo 数据，直接违背 #17 的产品目标。
2. 管理员批量编辑或网络抖动时，乐观回滚可能展示错误元信息。
3. 用户通过 CanvasSwitcher 切换画布时可能丢失未保存本地改动。
