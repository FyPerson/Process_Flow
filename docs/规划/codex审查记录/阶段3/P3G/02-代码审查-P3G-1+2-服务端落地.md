## 第二层：codex 原话技术细节

### Medium（3 条）

#### #1 contradiction / schemas/canvas.ts:PublishCanvasRequestSchema + routes/canvases.ts:POST /publish

**problem**：注释和路由语义写的是 published_note "trim 后" 1-500 字，但实际 validateBody 先跑 z.string().min(1).max(500)，route 里才 trim。结果是前后空格导致原始长度超过 500 时会被 400 拒绝，即使 trim 后正文不超过 500；全空格字符串则先通过 schema，再由 route 拒绝。

**suggestion**：把 trim 语义收敛到 schema，例如使用 z.preprocess 或 z.string().transform(s => s.trim()).pipe(z.string().min(1).max(500))，route 直接使用已 trim 的值；同时补测"正文 500 字 + 前后空格"应按产品契约决定通过或拒绝。

#### #2 edge-case / migrations/0003_canvas_publish_fields.sql

**problem**：migration 回填条件只判断 published_at IS NULL。如果存在历史 public 行 published_at 非 NULL 但 published_by 或 published_note 为 NULL，迁移不会补齐，列表仍可能出现部分发布元信息为空。注释目标是"历史 public 回填 published_at/by/note"，当前 SQL 只覆盖 published_at 缺失的子集。

**suggestion**：若业务上要求历史 public 三个 published_* 都可展示，应把条件改成任一字段缺失即回填缺失字段，例如用 COALESCE 分别补 published_at、published_by、published_note，并用 WHERE visibility='public' AND archived=0 AND (published_at IS NULL OR published_by IS NULL OR published_note IS NULL)。如果允许部分为空，应在注释和前端展示契约里明确。

#### #3 missing-constraint / migrations/0003_canvas_publish_fields.sql

**problem**：新增 published_by / unpublished_by 是外键，但没有明确 ON DELETE 行为。项目已有用户软删还好，但如果将来存在物理删除用户的管理动作，删除 users 行可能被历史发布记录阻塞，或者迁移依赖的 created_by 同样受影响。

**suggestion**：按现有用户生命周期确认是否永远不物理删除用户；若不保证，建议外键显式使用 ON DELETE SET NULL，并确保前端展示可 fallback 到"用户 #id"或"已删除用户"。

### Low（4 条）

#### #4 readability / services/canvases.ts:publishCanvas 注释

**problem**：migration 文件顶部注释写到"首次发布时清空 unpublished_at"，但 publishCanvas 实现注释和代码都明确"不动 unpublished_at / unpublished_by，保留上一次撤回"。同一功能的注释互相矛盾，后续维护者容易误改状态机。

**suggestion**：统一注释口径：如果当前决策是保留 unpublished_*，就删除 migration 注释中的"首次发布时清空"描述，或明确"曾考虑清空但已放弃"。

#### #5 missing-constraint / routes/canvases.publish.test.ts:测试 12

**problem**：测试 12 没有验证真实 migration 0003 对既有数据的执行时机。beforeEach 已经跑完 migrations，测试随后插入 legacy public，再手动复制 UPDATE SQL；这只能验证手写 SQL 片段，不验证 migration 文件在"旧库升级"场景中真的生效。

**suggestion**：增加一个专门的迁移测试：先只执行 0001/0002，插入 legacy public，再执行 0003，最后断言 published_* 被回填；这样能覆盖真实升级路径。

#### #6 edge-case / routes/canvases.publish.test.ts:POST visibility 强制 private

**problem**：测试只覆盖 visibility='public'，但 brief 特别要求覆盖"不是 private 的所有可能值"。当前 schema enum 会拒绝缺失、null、非法字符串，route 会拒绝 public；但测试没有锁住这些边界，后续 schema 放宽时可能漏出直接创建 public 的路径。

**suggestion**：补充参数化测试：visibility 缺失、null、'PUBLIC'、'private '、非法字符串均返回 400；visibility='public' 返回 403。这样能区分 schema 校验失败和 route 策略拒绝。

#### #7 other / src/api/canvases.ts:apiFetch

**problem**：文件中仍保留大量 DEBUG#15 console.log / console.error。虽然不影响 P3G 服务端契约，但发布/撤回操作会在前端控制台输出请求路径、状态和响应预览，内网场景下也是不必要噪音。

**suggestion**：如果这些调试日志不是本次故意保留，应在后续清理或用开发环境开关包住，避免管理端操作时控制台长期刷屏。

## Recommendations

1. 优先修正 published_note 的 trim 校验契约，并补上对应边界测试。
2. 把 migration 回填测试改成真实升级路径，避免测试复制实现 SQL 后产生假覆盖。
3. 统一 publish/unpublish 字段生命周期注释，尤其是 unpublished_* 是否在重发布时保留。

## Risks

1. 历史 public 数据如果存在部分 published_* 缺失，admin 列表仍可能显示不完整发布信息。
2. 发布说明长度在前后空格场景下可能与产品文案不一致，造成用户认为有效输入被拒。
3. 当前测试矩阵覆盖主路径较完整，但 migration 时机和 schema 边界仍有回归风险。
