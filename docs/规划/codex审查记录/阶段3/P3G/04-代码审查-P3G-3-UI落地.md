## 第二层：codex 原话技术细节

### Medium（2 条，建议 Playwright 前修）

#### #1 edge-case / 打开关闭 editing+publishing dialog 流程

**problem**：editError 和 publishError 没有在打开新 dialog 或取消关闭时清空。一次保存/发布失败后，取消再打开另一个画布的编辑或发布 dialog，会直接显示上一次的错误，属于 dialog state 泄漏。

**suggestion**：把打开/关闭 dialog 收敛成小 helper：打开 editing 前 setEditError(null)，打开 publishing 前 setPublishError(null)，关闭时也同步清空对应 error。提交成功关闭时也可以一并清空，避免后续复用残留。

#### #2 edge-case / handleUnpublish 和 handleArchive 行级操作

**problem**：撤回和归档没有行级 submitting guard，用户在请求未返回前可以再次点击同一行或其它操作。典型场景是连续撤回：第一次成功把 public 变 private，第二次返回 already_private，最终可能在已经成功的行上显示误导性错误。

**suggestion**：增加 rowSubmitting 状态，例如 { id, action }，在 handleUnpublish/handleArchive 从 confirm 通过后设置，finally 清空；同一行按钮在提交中禁用，或至少同一 action 禁止重复触发。

### Low（4 条）

#### #3 edge-case / rowError 生命周期

**problem**：rowError 只在发起撤回/归档前清空，不会在手动刷新、编辑成功、发布成功或后台 refetch 成功后清空。列表数据已经变新时，旧错误仍可能挂在同一 id 的行上。

**suggestion**：在手动刷新前、任一成功 mutation 后，或 hook refetch 成功回调处清空 rowError。最小改法是在刷新按钮 onClick 和各成功路径 setRowError(null)。

#### #4 edge-case / PublishedCell

**problem**：PublishedCell 以 published_at 作为整条发布记录是否存在的开关；如果 DB 出现 published_by 或 published_note 有值但 published_at 为 null 的脏数据，UI 会直接显示"—"，隐藏可用于排查的部分信息。

**suggestion**：如果希望更强防御脏数据，可改成逐字段 fallback：时间缺失显示"时间缺失"，发布人缺失显示"#?"，说明缺失显示"（无说明）"；只有三者都缺失时再显示"—"。

#### #5 edge-case / textarea maxLength

**problem**：maxLength=550 比服务端 trim 后 max=500 更严格：如果用户粘贴大量前后空白加 500 字有效内容，浏览器会先截断原始输入，可能丢掉有效内容，虽然 trim 后本可通过 schema。当前限制对正常内网使用问题不大，但这是契约上的小偏差。

**suggestion**：更一致的做法是移除 textarea 的 maxLength，只用 trim 后计数和提交校验；或者把 raw maxLength 提高到明显的防误粘贴上限，并继续以 trim 后长度作为唯一业务校验。

#### #6 readability / mapApiError

**problem**：invalid_input 被统一映射成"发布说明 / 画布名"错误，忽略了 ApiError.issues。若服务端返回 description、is_public_to_guest 或其它字段校验问题，用户看到的错误会不够准确。

**suggestion**：优先展示 issues[0].message 或按 path 做简单映射；没有 issues 时再回退到当前通用文案。

## Recommendations

1. 建议进 Playwright 前优先处理两个 medium：dialog error 清理、行级操作提交中保护。
2. Playwright 可重点覆盖：发布失败后取消再打开不残留错误、连续点击撤回不会出现误导性 already_private、刷新后旧 rowError 消失。
3. PublishedCell 当前 fallback 不阻断上线；若 migration 已保证回填完整，partial-null 只需作为低优先级防御。

## Risks

1. 如果不清理 dialog error，admin 会在不同画布之间看到过期错误，容易误判当前操作失败原因。
2. 如果不加行级 submitting，重复点击会制造自相矛盾的 UI 状态，尤其是撤回成功后又显示 already_private。
3. 如果保留 550 raw maxLength，极端粘贴场景可能提前截断有效发布说明。
