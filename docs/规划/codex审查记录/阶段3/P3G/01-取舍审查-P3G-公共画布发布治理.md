## 第一层：非技术总结

整体方向是对的：把 public 产生收敛到独立 publish/unpublish 端点，适合当前内网小团队场景。主要盲点集中在 4 处：
1. **owner_id 语义**——草案规则 4 改 owner=admin 会"悄悄夺权"原 owner，建议 publish/unpublish 都不动 owner_id，仅历史 NULL 兜底
2. **状态机前置条件缺失**——重复 publish / 对 archived 画布操作 / 对 private 画布 unpublish 等行为未定义
3. **P3G-2 的 admin-only 路由切换边界**——PATCH 是混合权限路由（普通 owner 改 name/description + admin 改 is_public_to_guest/owner_id），整条切 requireFreshAdmin 会误伤普通用户
4. **历史数据迁移**——已有 public 画布无发布元信息，列表展示会遇到 NULL；要么回填要么前端兜底"历史数据无发布记录"

## 第二层：codex 原话技术细节

### High（3 条）

#### #1 数据完整性 / 取舍点 / 规则 4

**problem**：unpublish 后固定把 owner_id 设为撤回 admin，会把原私有 owner 的所有权悄悄转移给 admin。若画布原本由普通用户创建、admin 只是发布/撤回，撤回后原 owner 失去私有访问权，这比"NULL 只 admin 可见"更隐蔽。

**suggestion**：优先采用"publish 不清空 owner_id，unpublish 不改 owner_id"。public 画布保留 owner 属性只作为归属元信息，不影响 canRead/canWrite 的 public 规则。仅对历史 owner_id=NULL 的 public 画布，在 unpublish 时设置 owner_id=撤回 admin 或要求 admin 选择 owner。

#### #2 产品规则盲点 / 规则 3 / 规则 4

**problem**：没有定义重复 publish、重复 unpublish、对 archived 画布 publish/unpublish、对 private 画布 unpublish 的行为。若实现时默认 UPDATE，可能覆盖 published_at/by/note，或把 private 画布的 unpublished_at 写成无意义状态。

**suggestion**：明确服务端状态前置条件：publish 仅允许 visibility='private' 且 archived=0；unpublish 仅允许 visibility='public' 且 archived=0。状态不匹配返回 409 already_public/already_private，archived 返回 409 archived_canvas 或 404/403 按现有风格统一。

#### #3 权限边界 / P3G-2 step

**problem**："admin-only 路由全切 requireFreshAdmin"容易漏掉 canvas 路由中混合权限的 PATCH：当前 PATCH 同时支持普通 owner 改 name/description，也支持 admin 改 is_public_to_guest/archived/owner_id。若整条 PATCH 直接切 requireFreshAdmin，会破坏普通用户改自己 private 画布元信息的能力。

**suggestion**：不要把 PATCH /api/canvases/:id 整体变成 requireFreshAdmin。保留 requireAuth，按字段分支：普通 name/description 走 canWrite；admin-only 字段在检测到后再要求 fresh admin，或拆出 admin 专用端点。publish/unpublish 可以直接 requireFreshAdmin。

### Medium（6 条）

#### #4 数据完整性 / migration / 规则 4

**problem**：新增 published_at/by/note 和 unpublished_at 后，历史 public 画布没有发布元信息。CanvasesTab 若只在 public 行展示发布人/时间/说明，会遇到 NULL 值；如果字段加 NOT NULL 又会迁移失败。

**suggestion**：迁移时先允许 nullable，并为现有 visibility='public' 画布回填：published_at 可用 created_at 或 updated_at，published_by 可用 created_by 或 updated_by，published_note 用类似"历史 public 画布迁移回填"。如果不回填，前端必须显示"历史数据，无发布记录"。

#### #5 数据完整性 / 规则 4

**problem**：只新增 unpublished_at 不记录 unpublished_by / unpublished_note，会让撤回动作缺少"谁撤回、为什么撤回"。既然发布强调 who/when/why，撤回至少也应保留 who/when，否则排查误撤回时信息不完整。

**suggestion**：至少新增 unpublished_by；是否新增 unpublished_note 可按成本决定。若不想加撤回说明，unpublish confirm 可以只要求二次确认，但 DB 中应记录 unpublished_by/unpublished_at。

#### #6 产品规则盲点 / 规则 3 / 规则 5

**problem**：published_note 的生命周期不清楚：再次发布同一画布时是覆盖上一条 note，还是保留历史？当前单字段模型只能保留最后一次发布说明，不能形成完整审计链。对 5 人内网来说完整事件表可能过重，但需要明确这是"最近一次发布记录"，不是完整审计。

**suggestion**：字段命名/文案上按"最近一次发布"处理，例如列表展示"最近发布说明"。如果希望保留撤回历史，最小方案是加 last_published_* / last_unpublished_*，不要承诺完整审计链。

#### #7 权限边界 / 规则 1 / POST /api/canvases

**problem**："移除 visibility=public 路径"需要明确 CreateCanvasRequestSchema 是否仍接受 visibility 字段。如果 schema 仍要求 visibility，调用方仍可能传 public，只靠 route 层拒绝；如果 schema 改为不接受 visibility，则现有前端创建 private 的调用也要同步改。

**suggestion**：更简单的兼容做法：schema 暂时保留 visibility，但 route 强制 body.visibility 必须为 private，否则 403/400，并新增单测覆盖 admin 传 public 也失败。若要彻底移除字段，应同步 API 类型、前端创建、测试夹具。

#### #8 数据完整性 / 规则 3

**problem**：publish 时没有提到 is_public_to_guest 的处理。现有 public 对游客是否可见由 is_public_to_guest 控制，如果 private 画布历史上该字段异常为 1，发布后可能直接对游客可见；如果默认 0，则 admin 还要再 PATCH 一次。

**suggestion**：在 publish 端点明确策略：默认 publish 后 is_public_to_guest=0，由 admin 另行勾选游客可见；或 publish body 显式允许 is_public_to_guest boolean。考虑避免误公开，建议默认置 0。

#### #9 测试覆盖 / P3G-1 / P3G-4

**problem**：单测只提到 schema 拒绝 visibility，缺少关键权限矩阵：普通用户 publish/unpublish、旧 admin token 被降权后访问 publish/unpublish、archived 画布、already public/private、owner 保留。

**suggestion**：测试重点放在服务端：admin 传 public 创建失败；import/save-as 仍 private；publish/unpublish 仅 fresh admin；publish 保留 owner_id；unpublish 保留 owner_id 或处理 NULL legacy；PATCH visibility strict reject；published fields 写入且列表返回。

### Low（2 条）

#### #10 实现复杂度 / 规则 5

**problem**：published_note 用 span title 展开虽然实现便宜，但长说明在表格里不可复制/不可触屏查看，且 title 在移动端体验弱。内网 admin 台账也可能需要快速扫读。

**suggestion**：表格中截断显示前 30-50 字，旁边加"查看"或复用编辑 dialog 展示完整说明；title 可作为补充，不作为唯一展开方式。

#### #11 readability / P3G 拆分草案

**problem**：P3G-1 同时包含 migration、端点、封堵、单测，P3G-2 又改权限中间件，边界略重；如果中途出问题，难以判断是发布治理还是 fresh admin 切换导致。

**suggestion**：实施顺序建议先做 schema/migration/端点和服务端测试，再做 requireFreshAdmin 覆盖，最后接 UI。若时间紧，fresh admin 切换只覆盖 publish/unpublish 和 admin-only PATCH 字段，不要扩散。

## Recommendations（codex 倾向的整体方案）

1. 整体建议采用 Y+，但把 owner_id 策略改成：publish 保留 owner_id，unpublish 默认也保留 owner_id；只有历史 owner_id=NULL 的 public 画布才兜底设置为撤回 admin 或要求选择 owner。
2. 服务端规则要补齐状态前置条件，避免重复 publish/unpublish 覆盖审计字段。
3. 字段模型保持轻量即可，但文案应称为"最近一次发布/撤回记录"，不要暗示完整审计链。
4. P3G-2 谨慎处理混合权限 PATCH，不要因为 fresh admin 债务偿还误伤普通 owner 的元信息编辑能力。

## Risks

1. 若 unpublish 改 owner 为 admin，普通用户会在撤回后丢失原本属于自己的画布访问权。
2. 若 historical public 没有迁移回填，列表可观察性会出现空发布信息，削弱 P3G 的治理目标。
3. 若重复 publish 允许覆盖 published_at/note，会把第一次发布事实抹掉。
4. 若 publish 默认沿用异常 is_public_to_guest=1，可能造成游客可见范围扩大。
