**结论**

P3D-1 拆分是合理的，不建议和 P3D-2 一锅端。先把 `nodes_meta` 历史补齐、GET hydrate 权威化、前端 creator 数据通路、`canEditNodeData()` 纯函数做成一个可审闭环，能显著降低 P3D-2 全入口改造返工风险。

但当前 P3D-1 定稿还需要补几处，否则“地基”会有错位。

**必修**

1. `stripServerOwnedFields` 不能无脑在 `saveCanvas/createCanvas` 里 strip `is_deprecated`。
   当前保存链路依赖客户端传回 `is_deprecated` 判断 false -> true 标废，代码里也明确说明了这一点（[server/services/canvases.ts](<E:/业务全景图/server/services/canvases.ts:648>)、[src/hooks/useMultiCanvas.ts](<E:/业务全景图/src/hooks/useMultiCanvas.ts:217>)）。
   建议拆两个 helper：
   - `stripHydratedNodeMetaForCompareOrHydrate`：strip `creator_id/creator_username/created_at/updated_by/updated_at/is_deprecated/deprecated_*`
   - `stripServerAttributionForSaveInput`：strip `creator_id/creator_username/created_at/updated_by/updated_at/deprecated_*`，但保留 `is_deprecated`

2. migration 不能只写 `creator_id/created_at/is_deprecated=0`。
   `nodes_meta.updated_by/updated_at` 是 NOT NULL，必须填。建议：
   - `creator_id = canvases.created_by`
   - `created_at = canvases.created_at`
   - `updated_by = canvases.updated_by`
   - `updated_at = canvases.updated_at`
   - `is_deprecated` 尽量从 storage node 的 `$.is_deprecated` 继承，不要硬置 0，避免已标废历史节点被 hydrate 覆盖回未废弃
   - `deprecated_by/deprecated_at` 如果 storage 里有合法值就继承，否则 NULL

3. 前端数据通路不只 `BusinessFlowVisualization -> React Flow data`。
   还要补 `useMultiCanvas.convertNodesToStorage()` 的反向转换。否则节点一旦触发本地同步，`creator_id/creator_username` 会从 project 里掉，后续 remount 或切 sheet 后 `canEditNodeData()` 又失去依据。
   相关现状在 [src/hooks/useMultiCanvas.ts](<E:/业务全景图/src/hooks/useMultiCanvas.ts:187>)。

4. `FlowNodeData` 必须显式加 `creator_id?: number`，不只是 `creator_username`。
   现在 `FlowDefinition.nodes[]` 已有 `creator_id`，但 `FlowNodeData` 没有；`canEditNodeData()` 要比较 `user.id === nodeData.creator_id`，类型上必须落地。

5. `__localNew` 覆盖范围要补全“新节点来源”。
   拖出/工具栏新增、粘贴、创建分组都要打。粘贴时还要清掉原节点的 `creator_* / updated_* / deprecated_by_username`，否则复制别人节点会短暂继承别人 creator。
   另外当前还有 `duplicateSheet()` 会复制节点并生成新 node id（[src/hooks/useMultiCanvas.ts](<E:/业务全景图/src/hooks/useMultiCanvas.ts:610>)）。它也会制造“本地新节点”，要么纳入 P3D-1，要么明确列为 P3D-2 第一项风险。

**migration 校验 SQL**

生产行数我无法从当前 workspace 确认；应在生产库只读执行：

```sql
SELECT COUNT(*) AS canvases FROM canvases;
SELECT COUNT(*) AS nodes_meta FROM nodes_meta;

WITH storage_nodes AS (
  SELECT
    c.id AS canvas_id,
    json_extract(s.value, '$.id') AS sheet_id,
    json_extract(n.value, '$.id') AS node_id
  FROM canvases c
  JOIN json_each(c.data, '$.sheets') s
  JOIN json_each(s.value, '$.nodes') n
)
SELECT COUNT(*) AS storage_nodes FROM storage_nodes;

WITH storage_nodes AS (
  SELECT DISTINCT
    c.id AS canvas_id,
    json_extract(s.value, '$.id') AS sheet_id,
    json_extract(n.value, '$.id') AS node_id
  FROM canvases c
  JOIN json_each(c.data, '$.sheets') s
  JOIN json_each(s.value, '$.nodes') n
)
SELECT COUNT(*) AS missing_meta
FROM storage_nodes sn
LEFT JOIN nodes_meta m
  ON m.canvas_id = sn.canvas_id
 AND m.sheet_id = sn.sheet_id
 AND m.node_id = sn.node_id
WHERE m.node_id IS NULL;
```

迁移后验收口径：`missing_meta = 0`。如果没有 dangling meta，`nodes_meta` 应等于 storage 里的 distinct `(canvas_id, sheet_id, node_id)` 数。

**取舍意见**

- 用 `canvases.created_by` 做历史 creator 是可接受的唯一低风险默认值，但要先查是否存在转移：`owner_id IS NOT NULL AND owner_id <> created_by`。代码里已经支持 admin 改 `owner_id`，所以“transfer 不存在”不是代码层面的绝对事实，只能靠生产数据确认。
- migration 失败会阻止 server 启动，这是对的。当前 migration runner 每个 SQL 文件都包在 `db.transaction()` 里，并且 `schema_migrations` 记录也在同事务内，失败会回滚并下次重试（[server/db/index.ts](<E:/业务全景图/server/db/index.ts:115>)）。
- `getCanvasFull` 用 `nodes_meta` 覆盖 stale storage 字段是想要的行为。`creator_username` 必须只作为 GET 派生字段，不应成为可信存储来源。
- `schema .strict()` 拒绝 `__localNew` 是正确兜底；不要让 schema 接受它。服务端 strip 主要兜 `creator_username` 这类允许透传字段。

**闭环判断**

P3D-1 可以拆出来单独做，且应该单独审。但定稿前建议补上以上必修，尤其是 `is_deprecated` helper 拆分、migration 字段完整性、`useMultiCanvas` 反向数据通路、以及 duplicated sheet/local-new 来源。补完后，P3D-1 的闭环标准就是：历史 meta 不缺、GET hydrate 权威、前端任意本地状态转换后 creator 仍不丢、`canEditNodeData()` 矩阵测试全绿。
