推荐 **方案 Y，但做成 Y+：发布有摩擦 + 服务端事实审计 + 列表可观察**。

方案 X 不推荐。`is_derivative` 解决的是“复制路径不能变 public”，但它不是可靠事实，只是路径标记。admin 仍然可以通过导出 JSON、导入、手工重建、模板化等方式绕过。更重要的是，X 把一个正常产品动作也禁掉了：基于现有公共画布迭代新版本。这会逼 admin 直接改线上 public，反而增加生产风险。

**结论**
选 **Y**。产品上更符合真实工作流，工程上边界更清晰，安全/合规上有 who/when/why 的事实记录。
但不建议只做“确认弹窗”。真正有价值的是：**所有 public 产生都必须经过发布动作，且发布行为可审计、可观察、可撤回**。

**推荐核心规则集**
1. `duplicateCanvas` 和 `importCanvas` 产生的新画布一律为 `private`，`owner_id = 操作者`，不继承源画布 visibility。

2. 普通用户永远不能创建、发布、撤回 public；只有 admin 可以执行发布相关动作。

3. 禁止通过普通 `PATCH /api/canvases/:id` 修改 `visibility`；public/private 切换只能走独立端点：
   `POST /api/canvases/:id/publish`
   `POST /api/canvases/:id/unpublish`

4. 发布必须满足服务端校验：当前操作者是 admin、画布归 admin 或具备管理权限、必须提交 `published_note`，服务端写入 `published_at / published_by / published_note`。

5. 公共画布列表展示发布人、发布时间、发布说明；当 public 数量过多或近期发布频繁时给 admin 明确警告，但先不要做硬性 cooldown。

**不建议上来做的规则**
发布说明 50 字下限、7 天 cooldown、同名禁止发布，这些容易变成产品摩擦而不是安全能力。可以先做软提醒，例如：

- public 超过 10 个时提示清理或归档
- 30 天内发布超过 N 个时提示确认
- 发布说明少于 10 字时前端提醒，但服务端只做非空或基础长度校验

**工作量判断**
建议作为 **独立小阶段**，不要硬塞进 P3F，除非 P3F 本身就是权限/发布治理阶段。

原因是 Y 涉及的不只是 schema 字段，还包括权限边界重划：

- migration：`published_at / published_by / published_note`
- 后端：新增 publish/unpublish 端点
- 后端：禁止普通 PATCH 修改 visibility
- 前端：发布/撤回确认 dialog
- 前端：公共列表展示发布元信息
- 测试：user/admin/guest 权限矩阵、duplicate/import 默认 private、publish/unpublish 行为

如果时间很紧，可以拆成两步：
第一步先做服务端端点和权限封口；第二步再补列表可观察性。

**潜在反对意见**
“Y 还是挡不住 admin 填 `asdf` 然后发布垃圾 public。”

反驳：
是的，Y 不把 admin 当成敌人防御，它防的是误操作和缺少治理。当前系统只有一个 admin，本质上是所有者行为，技术上无法也不应该彻底阻止他发布。真正要解决的是：发布必须是一个明确动作、不能被复制路径顺手带出、发布后能看见是谁何时为什么发布、出问题能撤回。
这比 `is_derivative` 更接近组织级工具需要的真实约束。
