**规则闭环核对**

服务端核心判断在 [server/services/canvases.ts](<E:/业务全景图/server/services/canvases.ts:682>)：

```ts
const isPrivateOwner = row.visibility === 'private' && row.owner_id === user.id;
if (!isAdmin && !isPrivateOwner) return 403;
```

矩阵结论：

- public + 普通用户：`isPrivateOwner=false`，`!isAdmin=true`，403 `forbidden_remove_node`
- public + admin：`isAdmin=true`，放行
- private + owner 普通用户：`isPrivateOwner=true`，放行
- private + 非 owner 普通用户：route 层 [canWrite](<E:/业务全景图/server/routes/canvases.ts:223>) 先拦；即使直调 service 也会 403
- private + admin 非 owner：route 层放行，service 因 `isAdmin=true` 放行

`owner_id=null` 边界也正确：public 的 `row.owner_id === user.id` 为 false，不会把普通用户误判成 owner。

**错误名重命名**

旧的 `forbidden_remove_node_non_admin` 在 `server/src/docs` 搜不到，只有新 `forbidden_remove_node`。route 只是透传 [result.error](<E:/业务全景图/server/routes/canvases.ts:240>)；前端 API 层把任意错误名归一成 `ApiError.error: string`，见 [src/api/canvases.ts](<E:/业务全景图/src/api/canvases.ts:64>)。`SaveStatus` 未硬编码旧错误名，default 会显示服务端 message。因此重命名风险很低。

**visibility 类型边界**

`SELECT` 已补 `visibility, owner_id`，见 [server/services/canvases.ts](<E:/业务全景图/server/services/canvases.ts:546>)。运行时没做 enum guard，但 DB schema 有 `CHECK(visibility IN ('public','private'))`，所以正常路径 OK。若未来想做防御式兜底，可以在读取后加一行 runtime guard；不是本 commit 必修。
