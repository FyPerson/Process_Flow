## 三审结论
- 总体：未通过。核心 `save()` 主路径基本修住了，但还有 **3 条竞态遗漏 + 1 条 deepEqual 新回归**。
- 结论：ref-state 双轨方案方向对，但实现还不是“完全 ref-first”。

## 二审必修项的修复验证
- fetch effect 短路 + GET 身份校验：**部分**
  - A→B 不会被短路误吞，create 后 URL 同 id 重入短路也成立。
  - 但 GET 成功后只 `setServerVersion(row.version)`，没有同步 `serverVersionRef.current`，ref 在 passive effect 前仍可能是旧值/null。
- `save()` 稳定引用 + `saveInFlightRef`：**部分/基本 OK**
  - 并发拦截 OK；sync throw 前未置 `saving`，不会卡住。
  - 身份校验 OK。
  - 但保存成功后没同步 `serverVersionRef.current = result.version`，极短窗口内二次 save 可能拿旧 baseVersion。
- `createOnServer/loadFromServer/discardAndReload` 身份校验：**部分**
  - `discardAndReload` 的 project/version 写入有身份校验。
  - `createOnServer` 实际没有 captured identity 校验，用户 POST 期间切到别的 canvas，回包仍会覆盖当前 hook state。
  - `loadFromServer` 入口直接改 ref，但没同步 state，随后 `setIsLoading(true)` 触发 render 后可能被 `useEffect(() => canvasIdRef.current = canvasId)` 用旧 state 改回去。

## 二审建议项的修复验证
- `deepEqualStorage`：**引入新问题**
  - key order 不敏感做到了。
  - 但它不再等价于 JSON 存储语义：`{ a: undefined }` 和 `{}` 现在被判不同。`convertNodesToStorage` 会生成不少 undefined 可选字段，加载 JSON 后这些 key 往往不存在，会造成假 dirty。
- `formatServerError` 中文文案：**OK**

## 新发现的必修项
1. `src/hooks/useMultiCanvas.ts:130`
   - 问题：`deepEqualStorage` 把 undefined 字段和缺失字段判为不同；`src/hooks/useMultiCanvas.ts:199`、`:215-219` 会生成 undefined 可选字段。
   - 修改：比较对象 key 时过滤 `value === undefined` 的 key，或先做 JSON-equivalent canonicalize 后再 deepEqual。

2. `src/hooks/useMultiCanvas.ts:816` / `:821` / `:832`
   - 问题：`createOnServer` 没有 captured canvas identity。POST 期间用户切到 B，成功回包仍会把 ref/state 改成新建 id。
   - 修改：请求前捕获 `const capturedCanvasId = canvasIdRef.current`；`await apiCreateCanvas` 后若 `canvasIdRef.current !== capturedCanvasId`，不要写 ref/state，也不要让 caller 写 URL。需要返回 `discarded` 或抛出可识别取消错误，并让 `handleSaveAsNew` 跳过 `setSearchParams`。

3. `src/hooks/useMultiCanvas.ts:685` / `:687` + `:358`
   - 问题：`loadFromServer(id)` 先写 `canvasIdRef.current = id`，但没有同步 `setCanvasId(id)`；前置 `setIsLoading(true)` 的 render 会让 sync effect 用旧 `canvasId` 覆盖 ref，导致自己的 GET 回包被误丢弃。
   - 修改：入口处同步 `setCanvasId(id); setServerVersion(null);`，或封装一个同时写 ref+state 的 `setCanvasIdentity(id, version)`。

4. `src/hooks/useMultiCanvas.ts:708` / `:797`
   - 问题：identity mismatch 后 `return`，但 `finally` 仍无条件 `setIsLoading(false)`。旧请求可能提前关闭新 canvas 的 loading，短暂暴露旧 project。
   - 修改：`finally` 里也加身份保护，例如 `if (canvasIdRef.current === capturedCanvasId) setIsLoading(false)`；fetch effect 的 `finally` 也建议加同类 id 判断。

## 新发现的建议项
- `src/hooks/useMultiCanvas.ts:406`、`:741`：所有 `setServerVersion(...)` 成功路径旁边都同步 `serverVersionRef.current = ...`，保持真正 ref-first。
- `save()` 返回类型建议补上 `skipped/discarded`，避免后续调用方误以为只会有 `{ ok, conflict }`。
- 补单测/钩子测试：create 期间切 URL、discard 期间切 URL、loadFromServer 自身切换、deepEqual missing-vs-undefined。

## 不必修但要知道

- `loadFromServer` 当前没有实际 call site，问题暂时偏潜伏，但它是 public return API，建议现在修掉。
