## 整体评级

production-ready: **78%**

能否进 P2I：**能**，但把“只读模式入口收口”和“旧侧栏保存按钮语义收口”放进 P2I 第一批。  
能否进入阶段 3：**还差 4 件事**：浏览器实测、只读真正禁写、旧保存入口统一、脏数据切换/离开保护。

我没有跑通本地 typecheck，`tsc` 被当前执行策略拦截了；下面判断基于代码阅读。

## 设计合理性

整体是“阶段 2 可接受的复杂 hook”，不是工程艺术品，但核心方向是健康的。`useMultiCanvas` 现在胖，问题不在 900 行本身，而在它同时管 **数据转换、项目 mutation、服务端会话、保存状态机、自动保存调度、兼容迁移**。P2H 这种竞态密集修复阶段先放在一个 hook 里是合理的，因为状态闭环集中，修竞态成本低。

建议不要立刻大拆；P2I/P2J 前做“薄拆”就够：

- 抽 `canvasStorageAdapter.ts`：`convertNodesToStorage` / `convertEdgesToStorage` / `deepEqualStorage` / `safeDeepCopy`
- 抽 `useCanvasPersistence`：`canvasId/serverVersion/dirty/saving/conflict/autoSaveDisabled/save/create/discard/autosave`
- 保留 `useMultiCanvas` 做 facade，对 caller 暴露稳定 API

## 状态机

这套状态/ref **不算过多**，大部分是被真实 async 竞态逼出来的。

必要保留：`isLoading`、`saving`、`saveInFlightRef`、`dirty`、`changeSeqRef`、`serverError`、`conflict`、`autoSaveDisabled`、`lastSavedAt`、`loadRevision`、`canvasId + canvasIdRef`、`serverVersion + serverVersionRef`、`projectRef`。

可优化但不急：

- `conflict` 可从 `serverError.status === 409` 派生，但显式保留更利于 UI。
- `autoSaveDisabled` 是“错误后停止重试”的 latch，不等同于 `!!serverError`，保留。
- `saving + saveInFlightRef` 当前重复但合理：一个给 UI，一个做同步互斥。
- `canvasId/serverVersion` 的 state/ref 双轨以后可以合成一个 `sessionRef`，减少心智负担。
- `loadRevision` 是对 React Flow 非完全受控状态的补丁，阶段 2 可接受，长期应由 `FlowCanvas` 自己正确响应 props 变化。

## caller 负担

当前 `useMultiCanvas` 返回 19 个字段，对 P2I/P2J 来说偏重，但还没失控。真正危险的是 `save()` 返回值：

`{ok:true}` / `{ok:true, discarded}` / `{ok:false, conflict}` / `{ok:false, skipped}` / `{ok:false, discarded}`

这对未来 caller 容易漏分支。现在 `BusinessFlowVisualization` 基本没出大错：冲突分支处理了，`discarded/skipped` 静默也基本可接受。但 API 形状建议改成 discriminated union：

```ts
type SaveResult =
  | { status: 'saved' }
  | { status: 'conflict'; currentVersion: number }
  | { status: 'skipped' }
  | { status: 'discarded'; ok: boolean };
```

并把 caller 常用逻辑包成 `saveWithDefaultConflictHandling()` 或 hook 内部 action，别让 P2I 每个页面都自己理解 5 种返回。

## 可维护性

还没到意大利面，但已经到了“下一轮再叠功能会明显变难”的边缘。

必要重构只有三类：

- 抽纯 helper：存储转换、deep equal、错误文案。
- 抽保存会话：把 ref-first、capturedCanvasId、capturedSeq、discarded 这些竞态协议集中注释。
- 给 `save/create/discard/autosave` 上方加一段“核心不变量”注释：ref 先于 state、seq 防吞改动、旧请求回包必须丢弃、失败后停 autosave。

## 用户流体验

主流程感知基本合理：黄点/蓝点/绿点、冲突红字、beforeunload、中文错误文案，这些对内部用户足够。

会困惑的点：

- 左侧旧“保存”按钮仍是本地版本/文件夹保存，不是服务端保存，和顶栏保存状态会冲突。
- 只读模式只是顶栏 badge，画布仍可拖拽/连线，底部标签仍可新增/删除/重命名；用户会以为能改，之后又保存不了。
- 冲突时只有“丢弃本地并重载”，没有“保留我的改动/另存为副本/导出备份”的逃生口，会吓人。
- 自动保存失败后停自动保存是对的，但 UI 需要明确“之后的改动也不会自动保存，点重试”。

## 遗留风险清单

建议在 P2I 前修：

- 只读模式真正禁写：禁用 ReactFlow 编辑、节点面板、底部 tab mutation、快捷键 mutation。
- 统一保存入口：左侧保存按钮要么隐藏，要么改成调用服务端 `save`，本地保存改名为“导出/本地备份”。
- 给 `save()` 返回值加 discriminant，降低未来 caller 漏分支概率。

阶段 2 前修：

- 浏览器实测完整流：登录、首次另存、自动保存、保存中继续编辑、冲突、失败重试、切 sheet、刷新、关闭页。
- 脏数据切换保护：URL canvasId 或未来画布列表切换前，如果 dirty/saving，要确认或先保存。
- 冲突逃生口：至少提供“导出当前本地 JSON”或“另存为新画布”。

阶段 3 前修：

- 把 `loadRevision` 补丁替换成更可靠的 FlowCanvas 受控同步。
- 给保存状态机加 focused tests，尤其是 save 期间编辑、切 canvas、409、网络失败。

长期：

- localStorage 兼容路径设退役时间。
- 把 hook public API 分层成 project 操作、server session、UI status 三组。

## 给阶段 2 收尾最重要 3 件事

1. **浏览器实测矩阵跑完并记录结果**，curl 只能证明契约，不证明 React Flow 状态和 UI 入口一致。
2. **收口所有写入口**：只读禁写、旧保存按钮统一、底部 tab 操作纳入保存/权限体系。
3. **把保存状态机 API 稳住**：`save()` 改 discriminated union，加核心不变量注释，P2I/P2J 才不容易再把竞态修回去。
