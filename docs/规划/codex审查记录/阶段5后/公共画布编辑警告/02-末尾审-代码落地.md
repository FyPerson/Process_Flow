## 第一层：非技术总结

**结论**：方向正确，**0 critical + 0 high**，2 medium + 3 low。主要风险集中在两点：

1. **M1 user 对象引用问题** —— useEffect 依赖整个 user 而不是 user.id。AuthContext token refresh 时若用户选了 view-only，弹窗会被错误重置为 unknown 重弹一次。**修法**：把依赖改成 user?.id 标量。

3. **L2 unknown→view-only remount 浪费** —— 两态都是 readOnly，但 key 变了。可把 unknown/view-only 都映射为 'locked'。

## 第二层：codex 原话技术细节

### Medium（2 条）

#### M1：useEffect 依赖整个 user 对象
- **Category**：edge-case
- **Location**：src/hooks/usePublicEditAck.ts / useEffect deps
- **Problem**：effect 依赖使用了整个 user 对象而不是 user.id。若 AuthContext 因刷新用户资料、token refresh 或父组件重渲染替换了等价 user 对象，且用户当前选择的是 view-only（不写 sessionStorage），effect 会重新读不到 ack 并把 decision 从 view-only 重置为 unknown，导致弹窗再次出现。
- **Suggestion**：把依赖收敛到稳定标量，例如派生 userId = user?.id ?? null，并在 effect 与 ackedKey 中使用 userId。若 role 也影响 ready，可让 ready 继续负责 role 判定，但 effect 只依赖 ready、userId、canvasId。

#### M2：publicEditAutosaveLocked 反向回灌一帧滞后
- **Category**：edge-case
- **Location**：src/pages/BusinessFlowVisualization/index.tsx / publicEditAutosaveLocked 反向回灌
- **Problem**：autosaveLocked 通过 BFV state 反向回灌 useMultiCanvas，存在一帧滞后。通常 5 秒 autosave debounce 会被下一次 render 清理，不太可能真的 PUT；但同一帧内 SaveStatus 可能拿到旧 autosaveLocked=false，手动保存按钮短暂可点。
- **Suggestion**：更稳的做法是让手动保存禁用直接使用 isPublicAndUnacked，而不是 useMultiCanvas 返回的 autosaveLocked 回显；autosaveLocked 仍可通过回灌给 useMultiCanvas。这样 UI gate 不受回灌延迟影响。

### Low（3 条）

#### L1：archived public 画布是否允许复制
- canSaveAsPrivateCopy 对 archived public 画布仍为 true。按当前语义这不一定是 bug：归档公共画布不可写，但复制为私人副本作为逃生口仍然合理。不过如果产品语义是"归档内容不能被派生复制"，这里会漏掉 archived 分支。
- 建议：明确产品口径。允许则保留并注释；不允许则补 `!canvasMetaState.meta.archived`。

#### L2：unknown→view-only remount 实际无视觉变化但丢视口/选择态
- decision 从 unknown 到 view-only 会改变 publicEditAckedKey 并 remount FlowCanvas，但两者都是 readOnly，实际权限和节点视觉不变。用户点击"只查看"会丢失视口、选择态或面板状态，体验上可能显得重。
- 建议：如果只需要在 acked 后解除 React Flow 内部 readOnly 缓存，可把 key 中的 decision 压缩为 writable 维度，例如 unknown/view-only 都映射为 `locked`，acked 映射为 `acked`。

#### L3：ESC 监听 stopPropagation 吞掉页面其他 ESC
- ESC 监听挂在 document capture 阶段并 stopPropagation，会吞掉页面上其他 ESC 监听。对当前强制弹窗语义是合理的，但若同时存在 ConflictResolutionDialog 或其他全局 ESC 行为，公共编辑警告显示期间那些快捷键不会收到事件。
- 建议：保留当前行为也可以，但建议在注释中明确这是 modal 独占键盘事件的有意设计；若未来允许叠弹窗，应引入弹窗层级而不是多个 document capture 监听互相抢事件。

## Recommendations（4 条确认 OK 的链路）

1. handleSaveAsCopy 在 isPublicAndUnacked=true 时能走通：弹窗按钮直接调用 createOnServer，createOnServer 不受 autosaveLocked 限制，并用 saveInFlightRef 做并发保护，这条链路符合预期。
2. 复制成功后的旧公共画布 dirty 被 createOnServer 在 capturedSeq 未变化时清掉，并且 hook ref 会先切到新 private canvasId，未看到旧公共画布 dirty 在解锁瞬间被 autosave PUT 的实质窗口。
3. sessionStorage 写失败后本次 hook state 设为 acked、刷新后重弹，与注释里的 fail-soft 设计一致；在内网小规模场景可接受。
4. 建议补一条用例或人工验收：选择"只查看"后 AuthContext user 对象刷新时不应重新弹窗；这比新增 RTL 更贴近当前风险点，可用现有 e2e 或手工脚本覆盖。

## Risks

- 一帧级 UI gate 滞后一般不会触发 autosave，但可能被非常快的手动点击命中。
- view-only 不持久化是设计选择，但和 user 对象依赖结合后，可能出现非刷新场景下重复提醒。
- FlowCanvas remount 会重置 React Flow 内部状态；如果用户刚调整视口后选择只查看，体验可能受影响。
