## 第一层：非技术总结

**用户需求**：普通用户载入公共画布想"先在私人副本上做草稿再誊抄"。当前问题是 public 画布对普通用户可写但**不可删节点**（v1.16.2 强制），autosave 5s 直接 PUT 到服务端，误改后无法撤销。

**草拟方案 B'**：
1. 加载即弹拦截 Modal（普通用户 + 公共画布 + 未 ack 时）
2. SaveStatus 顶栏一直挂「复制为我的私人画布」主动按钮
3. Modal 三选项：复制副本 / 继续编辑 / 切回本地草稿

**codex 拦掉的关键判断**（草稿会被推翻的部分）：
- "取消"=切回本地草稿语义过重 → 改成"只查看公共画布"
- 主动复制按钮"绑定可写状态"会在 readOnly 分支消失 → 必须独立 prop
- "继续编辑"单击即 ack 不够强 → 加复选框确认
- effectiveCanvasWritable 仅 props 传不行 → 需显式 setNodes/setEdges 同步 readOnly
- handleSaveAsCopy 复制前需先 guard autosave，否则 dirty 公共画布会被竞态保存

## 第二层：codex 原话技术细节

### High（2 条）

#### H1：dirty 公共画布上点复制时 autosave 竞态
- **Category**：edge-case
- **Location**：方案 B' / createOnServer + dirty 公共画布
- **Problem**：公共画布已 dirty 时点击"复制为我的私人画布"，createOnServer 会用 projectRef.current 创建副本，这是合理的；但旧公共画布的 dirty 状态在跳转前仍然存在，理论上自动保存 timer 可能已经排队或即将触发，存在把用户未保存试探改动 PUT 回公共画布的竞态。
- **Suggestion**：复制前应先进入"公共画布拦截态"并禁止主版本 autosave，或在 handleSaveAsCopy 开始时设置本地 guard：复制流程期间 public canvas 的 onSave/autosave 不允许执行；复制成功跳转后由 useMultiCanvas 的 canvasIdProp 切换清 dirty/serverVersion/conflict。还应明确手动保存按钮在未 ack 公共画布时不可用。

#### H2：Modal 三选项 / 取消"切回本地草稿"
- **Category**：ambiguity
- **Location**：Modal 三选项 / 取消"切回本地草稿"
- **Problem**："取消"直接清 canvasId 切回本地草稿语义过重，用户只是想继续查看公共画布时会被强制带离；但如果取消只是关弹窗，又会进入"可看不可改但无明确恢复入口"的中间态。
- **Suggestion**：把第三项改成明确动作而非"取消"，例如"只查看公共画布"；执行后关闭弹窗但保持 effectiveCanvasWritable=false，并在顶栏显示"公共画布只读预览"加"复制为私人副本/继续编辑公共画布"入口。真正的关闭按钮也应等价于"只查看"，不要自动切走。

### Medium（5 条）

#### M1：usePublicEditAck / 触发条件
- **Category**：missing-constraint
- **Problem**：触发条件只描述"普通用户 + 公共画布 + 未 ack"，但没有约束 project/loadRevision 是否已经完成加载。若 canvasMetaState 已是 server 但 project 仍为空或 loadRevision 刚切换，可能出现弹窗时机抖动或旧画布 ack 状态短暂套用到新画布。
- **Suggestion**：弹窗条件应同时要求 canvasMetaState.kind === 'server'、canvasId 非空、project 非空、!isProjectLoading，并在 canvasId/userId 变化时重算 dialogOpen。避免把前一个公共画布的弹窗状态带到下一个画布。

#### M2：SaveStatus 主动入口
- **Category**：contradiction
- **Problem**：草案说 SaveStatus 顶栏"一直挂"复制按钮，但现有 SaveStatus 在 readOnly 分支只显示"只读"和导出按钮；如果 effectiveCanvasWritable=false 导致 readOnly=true，主动复制入口可能反而消失。
- **Suggestion**：SaveStatus 需要新增独立 prop，例如 onSaveAsPrivateCopy/canSaveAsPrivateCopy，并在 readOnly、dirty、saved、conflict 之外都能显示。不要把复制入口绑定到可写状态，否则被动拦截时最需要的按钮会不可见。

#### M3："继续编辑公共画布"选项
- **Category**：edge-case
- **Problem**：保留"继续编辑公共画布"有合理协作场景，但单击即 ack 可能不足以阻止误操作；sessionStorage ack 后 5 秒自动保存会继续生效，用户可能没有意识到影响是立即共享的。
- **Suggestion**：保留该选项，但建议加一次明确确认，例如复选框"我了解修改会自动保存并对所有人可见"。确认后再写 sessionStorage ack。无需二次弹窗，避免流程过重。

#### M4：dirty 状态下复制语义
- **Category**：ambiguity
- **Problem**：未保存改动会被复制到私人副本是正确默认，但如果 prompt 文案仍只是"给这个画布起个名字"，用户可能不知道当前屏幕上的未保存试探改动也会进入副本。
- **Suggestion**：在复制 prompt 或 Modal 主按钮旁明确一句："当前未保存改动会一起复制到私人副本，公共画布不会因此保存。"这比静默复制更符合用户心智。

#### M5：React Flow useNodesState / readOnly 实时响应
- **Category**：edge-case
- **Problem**：仅把 effectiveCanvasWritable 作为 props 传下去，未必能让已存在的 nodes/edges data.readOnly 实时更新。现有代码注释已说明 React Flow 内部 state 对外部 props 重算不敏感，且 useMultiCanvas 的 loadRevision 只在 fetch/loadProject 等整体替换时递增，不会因 acked 改变而 remount。
- **Suggestion**：需要检查 FlowCanvas 如何把 readOnly 注入 node.data/edge.data。如果是初始化时注入，应在 effectiveCanvasWritable 改变时显式 setNodes/setEdges 更新 readOnly 字段，或把 effectiveCanvasWritable 拼进 FlowCanvas key 触发 remount。优先选显式同步，避免 remount 丢选择态。

### Low（2 条）

#### L1：sessionStorage key
- key 包含 userId + canvasId 是合理的，但缺少版本或命名空间演进位。
- 建议：`bfv:public-edit-ack:v1:${userId}:${canvasId}`

#### L2：顶栏按钮可见性候选
- 候选 A 过窄，候选 C 可能扩大任务范围到"任意画布复制/备份"
- 建议选 A'：普通用户 + 公共画布 + 已挂接服务端时显示；admin 不默认显示。

## Recommendations

1. 选"加载即弹"基本对。首次 mutation 拦理论上可以通过 setNodes/setEdges 回滚，但会碰到 React Flow 内部状态、选择态、拖拽中间态和视觉闪烁，当前项目已经踩过类似坑。为降低打扰，把默认动作改成"只查看公共画布"而不是强迫 ack。
2. dirty 状态复制应保留"复制当前内存版本"的语义，并明确提示未保存改动会进入副本、公共画布不会保存。
3. createOnServer 看起来支持已挂接画布克隆，但要补 autosave/手动保存 guard，避免旧公共 dirty 在跳转前被保存。
4. "继续编辑公共画布"建议保留，但加确认勾选。
5. 不建议把取消定义为切回本地草稿；建议定义为"只查看"。
6. sessionStorage + userId + canvasId 合理，建议加 v1 命名空间。
7. 顶栏按钮建议选 A'，只服务本需求。
8. 必须专门验证 readOnly 视觉和 mutation gate 是否随 effectiveCanvasWritable 实时更新。

## Risks

- 如果只在 UI 层 readOnly 覆盖，但自动保存仍看 dirty/canvasId/autoSaveDisabled，试探编辑仍可能保存到公共画布。
- 如果 SaveStatus 的复制按钮在 readOnly 分支不可见，被动弹窗关闭后用户会缺少主要逃生口。
- 如果 acked 从 false 变 true 后 React Flow 内部 nodes/edges 没有同步，用户可能已经获准编辑但节点仍显示只读，或相反。
- 如果"继续编辑公共画布"没有强确认，方案可能无法真正降低误改概率。
