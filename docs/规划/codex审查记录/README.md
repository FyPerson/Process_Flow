# codex 审查记录索引

记录各阶段经历的 codex 审查、commit、评级、关键经验沉淀。

按 **两级层次** 组织：第一级 = 阶段（功能开发） + 横切问题（基础设施），第二级 = 子阶段 / 子主题。

> **关于"自审"记录**：本目录主要存放 codex 审查记录，但也包含 Claude **自审**记录（文件名形如 `NN-自审-...`）。当某个 step 的攻击面已经收敛到 codex 不会有新发现的程度（例如纯展示组件 + 字符串 fallback），Claude 会执行自审（穷举边界场景 + tsc + npm test）而不跑 codex，自审记录走相同归档格式与 codex 审查记录并排存放。目录名虽是"codex审查记录"语义略有冲突但不影响检索——本目录的本质是"step 完成时的审查闭环归档"，自审是其中一种形式。判定何时跑 codex 何时自审的依据，参考 `feedback_ai_collaboration.md` 的"边界：codex 是查漏补缺不是 QA"段。

## 两种归档风格的边界（2026-05-13 起）

本项目同时存在两种归档风格，**新功能用嵌套式 / 历史阶段保留原结构**：

### 风格 1：原结构（本目录内，历史阶段沿用）

`docs/规划/codex审查记录/<阶段或主题>/...` —— 方案文档放 `docs/规划/<主题>-方案.md`，codex 审查记录单独走本目录树。

**适用范围**：阶段 2-5、阶段5后已有子主题（公共画布编辑警告 / 昵称功能 v1.17.0）、横切问题。不重组。

### 风格 2：嵌套式（新功能 2026-05-13 起采用）

`docs/规划/<功能名>/` 单容器内同时存放方案 + codex 审查：

```
docs/规划/<功能名>/
├── 方案.md
├── codex审查/
│   ├── 01-...
│   └── 99-收尾.md
└── README.md（可选）
```

**适用范围**：2026-05-13 之后新立项的独立功能。首例：[游客草稿能力](../游客草稿能力/)（**2026-05-14 完整闭环 v1.19.0**：D-1~D-8 全 8 切片 + codex 取舍审 13 条 + 复审 7 条 + 99-收尾.md 三件套发版）。

**为什么不重组历史阶段**：阶段 2-5 已经在过去 1 个月反复迭代收敛，CLAUDE.md / auto memory / 多份内部相对路径引用都指向现路径，重组的代价 / 收益不划算。等未来重组触发理由出现再分批迁移。

## 阶段 vs 横切问题怎么区分

**阶段**：开发新功能（画布、权限、批注、admin UI）。每个阶段一个一级目录，子阶段（P2H / P3A 等）放在二级。

**横切问题**：不属于具体阶段功能，但**跨阶段反复出现**的环境/工具问题（hook 部署、dev 卡死等）。在某阶段被发现，但根治修复独立于阶段功能，修复后所有阶段受益。

详见 [横切问题/README.md](./横切问题/README.md) 的判定标准。

## 维护方式

新阶段开始时：
1. 在第一级建 `阶段N/` 目录 + `阶段N/README.md`（叙事 + 子阶段表格）
2. 每个子阶段建 `阶段N/PNX/` 目录
3. 子阶段内审查记录用编号命名：`<NN>-<轮次>-<commit>-<主题>.md`
4. 子阶段结束写 `阶段N/PNX/README.md` 做该子阶段叙事
5. 阶段全部结束后更新 `阶段N/README.md` 累计数据 + 在本 README 把状态从 🚧 改 ✅

横切问题发现时：
1. 在 `横切问题/<主题>/` 建目录（如 `dev-hang/` `R1-hook修复/`）
2. 写诊断 + 修复审查记录
3. 更新 `横切问题/README.md` 表格（"何时发现 / 根治时机 / 详情"）

详细操作流程见 [`.claude/skills/deploy/SKILL.md`](../../../.claude/skills/deploy/SKILL.md) "codex 审查归档流程"章节。
