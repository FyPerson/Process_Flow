## 推荐方向
B 改造版。

## 理由
把部署从“不可信 hook 派生链”移到已验证的主 PowerShell，成本最低，能立刻消除“绿字但未重启”的假成功。

## 次选方向（如果有）
A。若必须恢复“push 即部署”，用 server 端 watcher 接管自动化。D 只在 WinRM 已启用且测试稳定后考虑。

## B 改造版 vs A 全自动
值得。阶段 3 前更重要的是部署结果可信；少一步手动命令，换来的是避免假部署。

## D 用 WinRM
PowerShell Remoting/WSMan 本身可靠，通常没有 OpenSSH 的 stdin/PTY/MSYS 环境问题，但会引入认证、TrustedHosts、权限策略新风险。

## 根因最可能猜测
更像 Git Bash 派生进程的非交互 stdin/stdout/stderr 或 MSYS 环境影响 ssh 行为；PATH 概率较低，`2>&1` 最低。
